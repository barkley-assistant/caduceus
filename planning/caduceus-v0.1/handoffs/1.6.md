## Public signatures/contracts used

- **`preflight(cfg, path_env) -> CaduceusResult<CheckOutcome>`** is
  the canonical pre-flight check. It is intentionally a
  side-effect-free, deterministic function that the daemon main
  loop calls *before* acquiring the cron tick lock so configuration
  errors surface immediately.
- **`CheckOutcome`** is a public enum with `Ok`,
  `WorkerCommandEmpty`, `WorkerCommandUnresolved { looked_for }`,
  `BridgeUnreadable { path }`, `GitMissing`, `DirNotWritable { path }`,
  and `ProcessGroupsUnsupported` variants. The structured logger
  serialises each variant as discrete fields.
- **`is_failure()`** and **`describe()`** give callers a one-line
  summary suitable for log lines and operator notifications.
- **`resolve_executable(program, path_env)`** returns an absolute
  path to an executable file or a `CaduceusError::Config` explaining
  the failure.

## Behaviour

- **Worker command check.** The first element of
  `cfg.worker_command` must resolve. Absolute paths are validated
  by `is_executable_file` (regular file with at least one
  executable bit set). Names without a `/` are looked up against
  the supplied `path_env` via `which_in`.
- **Bundled bridge check.** When the second command argument ends
  with `worker-bridge.py`, the file is probed with a one-byte read
  so a permissions error surfaces here rather than at the
  supervisor's first spawn attempt.
- **`git` check.** `which_in("git", path_env)` must succeed.
- **State / workdir writability.** The first existing ancestor of
  `cfg.state_dir` and `cfg.workdir_base` is probed with a
  `create-write-unlink` test on a probe file. Caduceus creates
  the state dir on first run, so a missing leaf is not a failure;
  a known existing unwritable parent is.
- **Process-group check.** True on Unix, false elsewhere. The
  daemon refuses to start on platforms that cannot support
  SIGKILL/SIGTERM broadcast.

## State/schema effects

None. The pre-flight check is read-only with respect to daemon
state; the only filesystem touch is the temporary probe file used
to verify writability, which is removed before the check returns.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- `preflight` is read-only with respect to the operator's
  state. The only filesystem side-effect is the temporary
  probe file used to verify directory writability, which is
  removed before `preflight` returns.
- `which_in` and `resolve_executable` only stat and read
  permissions; they never spawn a process.
- `check_bridge_readable` opens the file read-only and reads a
  single byte, then closes the file handle. It cannot mutate
  the bridge.

## Residual risks

- The mode-0500 unwritability test is conditional on the test
  not running as root. CI environments that drop privileges
  before the test will exercise the strict path; root-level
  CI will exercise the lenient path. The lenient path is still
  meaningful â€” the daemon's own writability probe runs against
  the same permission bits.
