## Public signatures/contracts used

- **`WorkerResult`** is the canonical schema. Fields: `status`,
  `summary`, `commit_message`, `pull_request_title`, `artifacts`
  (BTreeMap), `investigation` (default `false`).
- **`WorkerStatus`** is `{ Success, Failure }` with
  `snake_case` serde renaming.
- **`parse_result_file(path, issue) -> CaduceusResult<WorkerResult>`**
  is the canonical reader. The function performs the read-side
  invariants the contract requires and wraps every failure as a
  contextual `CaduceusError::Worker` with the operation label
  `"read"`, `"parse"`, or `"validate"`.
- **`validate_worker_result(result, issue) -> CaduceusResult<()>`**
  is the pure validator. Tests can drive it without touching the
  filesystem.
- **Constants** — `MAX_RESULT_FILE_BYTES = 1 MiB`,
  `MAX_SUMMARY_BYTES = 64 KiB`, `MAX_TITLE_BYTES = 256`,
  `MAX_ARTIFACT_KEY_LEN = 128`, `MAX_ARTIFACTS = 100`.

## Behaviour

- **Read-side invariants.** `O_NOFOLLOW` open with
  `std::os::unix::fs::OpenOptionsExt::custom_flags(libc::O_NOFOLLOW)`.
  After open, the file's metadata is checked for `is_file()`.
  The cap is enforced *before* allocation: `meta.len() > cap` is
  a hard `CaduceusError::Config`, and the read uses `File::take`
  to bound the buffer to the same cap.
- **Schema stability.** `#[serde(deny_unknown_fields)]` on
  `WorkerResult` means a stray top-level key (e.g. `"rogue": true`)
  produces a JSON parse error wrapped as
  `CaduceusError::Worker { context: "parse", ... }`. The
  structural test confirms the field name leaks into the error.
- **String field rules.** Every required string is
  NUL-checked, trimmed-empty-checked, and bounded by its
  documented limit. `summary` is 64 KiB, `commit_message` and
  `pull_request_title` are 256 chars.
- **Title rules.** `pull_request_title` is one line — any
  newline or carriage return is rejected — and has no control
  characters. `commit_message` may contain newlines but no
  other control characters; `\r` is treated as a control
  character.
- **Artifact rules.** Keys are non-empty, ≤ 128 chars, and
  control-free. The map holds at most 100 entries. The map is a
  `BTreeMap<String, serde_json::Value>` for stable iteration.
- **Error wrapping.** Every I/O, parse, and validation failure
  is wrapped as `CaduceusError::Worker { context, stderr }` with
  the path included in `stderr`. Tests assert the context label
  is exactly `"read"`, `"parse"`, or `"validate"`.

## State/schema effects

None. The reader is read-only; it does not write to disk. The
canonical schema is:

```json
{
  "status": "success" | "failure",
  "summary": "<= 64 KiB NUL-free trimmed non-empty>",
  "commit_message": "<= 256 chars, NUL-free, trimmed non-empty, multi-line OK, no other control>",
  "pull_request_title": "<= 256 chars, NUL-free, trimmed non-empty, single line, no control>",
  "artifacts": { "<= 100 keys, each non-empty, <= 128 chars, control-free>": <any JSON> },
  "investigation": <bool, default false>
}
```

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- `parse_result_file` is read-only. It opens the file with
  `O_NOFOLLOW`, reads the bytes, and returns the validated
  `WorkerResult`. It never writes to disk.
- The validator is pure: same input → same output, no I/O.
- Symlinks are *rejected* (not followed), per the `O_NOFOLLOW`
  contract. The test confirms the read surfaces a clean
  `Worker { context: "read", stderr: ... }` error.

## Residual risks

- The file-size cap is enforced before allocation. A future
  attacker who could write 1+ MiB into the worker's
  result file would still see a clean `Worker { context: "read" }`
  error rather than a memory spike.
- The `Worker` variant's `stderr` field is *not* scrubbed. The
  bridge may emit arbitrary stderr text; the daemon's structured
  logger scrubs it through `error::scrub` before writing to the
  log file. The unit tests do not exercise that downstream path;
  it's covered by the integration tests in Phase 5.
