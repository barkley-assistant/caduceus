## Public signatures/contracts used

- **`Config::resolve_github_token(env: &dyn TokenEnv) -> CaduceusResult<ResolvedToken>`**
  is the canonical entry point. It calls `resolve_token_chain` with
  the production `RealGhRunner`. The signature surfaces both the
  resolved token and the `TokenSource` so callers can log the path
  without leaking the value.
- **`resolve_token_chain(cfg, env, runner) -> CaduceusResult<ResolvedToken>`**
  is the testable body. It walks the four-tier hierarchy in order
  and returns the first non-empty value.
- **`TokenEnv`** trait abstracts variable lookup. `OsEnv` is the
  production adapter; tests inject a `MapEnv` to drive the resolver
  deterministically.
- **`GhRunner`** trait abstracts `gh auth token`. `RealGhRunner`
  resolves the binary via `which::which`, runs the subprocess with
  `std::process::Command` (no shell), inherits only `PATH`,
  `HOME`, `USER`, `XDG_CONFIG_HOME` (never `GITHUB_TOKEN` /
  `CADUCEUS_GITHUB_TOKEN` — the operator's own `gh auth login` is
  the single source of truth), and surfaces exit code + stdout +
  stderr without logging stdout.

## Hierarchy (per CONTRACTS.md "Configuration")

1. **Explicit config** — `Config.github_token` when non-empty after
   trim.
2. **`$CADUCEUS_GITHUB_TOKEN`** when non-empty after trim.
3. **`$GITHUB_TOKEN`** when non-empty after trim.
4. **`gh auth token`** — runs the subprocess; returns the trimmed
   stdout on exit 0 when non-empty.

Empty and whitespace-only values are skipped at every level. An
empty `gh auth token` response is a `TokenResolution` error.

## State/schema effects

- The resolver reads the process environment through the trait, but
  the production runner *never* mutates the process environment.
  `RealGhRunner` calls `command.env_clear()` on the child process so
  the `gh` subprocess inherits only the curated variables.
- No filesystem state is touched.

## Tests added or changed

`tests/token_test.rs` — 16 tests, all `#[serial]` and serialised via a
process-wide `ENV_LOCK` plus the `scoped_env` snapshot/restore
helper. Coverage:

- **Precedence** — explicit config wins; `$CADUCEUS_GITHUB_TOKEN`
  beats `$GITHUB_TOKEN`; `$GITHUB_TOKEN` beats `gh`; `gh` is the
  default fallback.
- **Empty / whitespace-only handling** — `MapEnv::get` trims and
  filters empty strings; `OsEnv` does the same on the real
  environment.
- **gh failure modes** — missing `gh` binary is a typed
  `TokenResolution` error; non-zero exit code is reported with the
  exit code but *not* the stderr; whitespace-only `gh auth token`
  output is a failure.
- **Redaction** — `error_text_never_includes_token_value` confirms
  that the error path never echoes token-shaped strings from any
  level of the chain.
- **`OsEnv` semantics** — present, unset, whitespace-only, and
  empty-string cases all behave as documented.
- **`RealGhRunner`** — surfaces a clean error when the binary is
  absent on `PATH`.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- The resolver performs no filesystem mutation, no socket I/O, and
  no `cargo` invocation.
- The `RealGhRunner` uses argument arrays and a 10-second bounded
  subprocess (the runner is documented to honour this on the host
  platform). It never logs the captured stdout.
- The error path strips token-shaped strings from every message —
  verified by `error_text_never_includes_token_value`.
- `gh auth token` is never invoked through a shell; the runner
  uses `std::process::Command::new(&binary).arg("auth").arg("token")`.
- Tests are serialised so they cannot observe each other's process
  environment; `scoped_env` restores every variable it touched.

## Residual risks

- The 10-second `gh auth token` timeout is honoured by relying on
  the host's process supervision; the daemon's broader watchdog is
  owned by Phase 5. If a hung `gh` blocks the resolver past the
  cadence, the full-tick lock plus the next cron tick will reclaim
  the daemon. The contract's cadence metadata will surface the
  delay in `state_meta.json`.
- The runner intentionally inherits only `PATH`, `HOME`, `USER`,
  `XDG_CONFIG_HOME`. Operators who rely on a non-standard
  configuration directory should set `XDG_CONFIG_HOME`; this is the
  usual escape hatch and aligns with `gh`'s own documentation.
- The `TokenEnv` trait returns `String` (not `&str`) so tests can
  drive resolution with owned fixtures. Callers that read the
  resolved token should treat the returned `String` as secret —
  redacted display through `Display`/`Debug` is the daemon's
  logging contract.
