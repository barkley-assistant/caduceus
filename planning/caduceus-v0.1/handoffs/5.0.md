## Stub policy

Per the contract, the default build contains **no** reachable `unimplemented!()`, `todo!()`, or `panic!()`:

```
$ grep -rn "unimplemented!\|todo!()\|panic!" src/
(no matches outside cfg(test))
```

`finalize` returns `Ok(FinalizeOutcome { commit_oid: None, pr_number: None, pr_url: None })` — a successful no-op, not a stub that panics. The cron tick's `run` subcommand never calls `finalize` until Phase 6 gates orchestration behind a `compile_orchestrator` feature flag (per the packet's instruction).

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No production `unimplemented!`/`todo!`/`panic!`.** Verified above.
- **No public-voice violations.** The `validate_public_text` family is already in place from earlier tasks; Task 5.0 does not relax the rule.
- **No daemon orchestration enabled by default.** `finalize` is only called when the cron tick is wired up in Phase 6; until then the cron `run` is a no-op success (per Task 0.x).
- **No GitHub API calls.** The `FinalizeContext::client` is `()` for Task 5.0; the Phase 6 implementation will swap the concrete client type in.

## Residual risks

- `FinalizeContext::client` is `()` for Task 5.0. Phase 6 will replace this with the concrete client type. The struct's `pub` surface stays stable (the field name `client` and the order of fields never change), so any Phase 6 implementation can substitute the type without breaking the public API.
- `FinalizeAction::Closed` is added for completeness even though no Phase 6 task currently uses it — Task 6.4 may use it for the post-merge close path. Removing it would shrink the public surface but adding it later is a breaking change for any caller that exhaustively matches on the enum.
