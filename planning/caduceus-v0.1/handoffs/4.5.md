## Behaviour (per the contract)

1. **Enumerate registered worktrees** with `git worktree list --porcelain` for every `(owner, repo)` in `config.watched_repos`. The main clone's own entry is skipped (it shows up first in the porcelain list but is not a candidate).
2. **Path safety**: every candidate must canonicalise to a path under `<repo>/.worktrees/`. Any entry outside is reported and skipped.
3. **In-use exclusion**: the in-use set is the union of
   * every `worktree_path` recorded in an unparseable, in-flight claim file under `<state_dir>/claims/`;
   * every fresh heartbeat (`mtime` within the last hour) under `<state_dir>/runs/`. The run_id is taken from the heartbeat's basename; the worktree path is probed at `<workdir_base>/<owner>/<repo>/.worktrees/<run_id>` for every watched repo and added if it exists.
4. **Age cut-off**: a candidate is removed only if its mtime is older than `older_than_days`.
5. **Removal**:
   * Registered entries go through `worktree::remove` (Task 4.3). The branch-retention, path-safety, and idempotency rules all live there.
   * Unregistered orphan directories under `<main>/.worktrees/` are removed with `fs::remove_dir_all` (the documented `git worktree remove --force` refuses on a path git does not know about). Symlinks are detected with `symlink_metadata` and reported, never followed.
6. **`--dry-run`**: prints `would remove …` for each candidate and returns `0`. No state mutation.

## CLI

`caduceus worktree-gc [--older-than-days N] [--dry-run]`

Acquires `DaemonLock` first so a concurrent tick wins the race; prints the number of removed worktrees (or the dry-run report).

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No GitHub calls.**
- **No credential leakage.** The runner (Task 4.1) scrubs the credential variables before invoking git; the GC inherits the runner's allowlist and timeout.
- **No path-escape removal.** A candidate that canonicalises outside `<repo>/.worktrees/` is reported and skipped. The contract's "Refuse any handle whose canonical path leaves `<main>/.worktrees/`" rule from Task 4.3 carries over.
- **No untracked claims, heartbeats, or queue tombstones touched.** The GC reads claim files for the in-use set but never mutates `state.json` or the queue; it does not touch the heartbeats it reads.
- **No recursion into untracked directories.** Orphan removal is a single `remove_dir_all` on the orphan itself.

## Residual risks

- The `watched_repo_paths` helper used by the heartbeat→worktree mapping calls `Config::load()` and the `CADUCEUS_CONFIG` env var. If neither succeeds, the function returns an empty list and the GC falls back to the claim-based in-use set. The CLI path always sets `CADUCEUS_CONFIG`; the bare-bones `caduceus worktree-gc` invocation will work even without a config file as long as the operator passes `--older-than-days` and the operator has not set up claim/heartbeat bookkeeping.
- The `git worktree list --porcelain` parser is tolerant of the documented `worktree <path>` and `branch refs/heads/<name>` keys. An exotic porcelain format (e.g. a detached HEAD) yields a `WorktreeListEntry` with an empty branch, which the GC still removes correctly.
- A GC run that races with a concurrent `create_worktree` may briefly see a worktree that has just been created. The `git worktree list` snapshot is taken per-repo, so an entry that appears between two repo iterations is not seen. This is intentional — the GC is opportunistic and the next tick will catch the entry.
