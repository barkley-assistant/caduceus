## Public signatures/contracts used

- **`github::RateLimitInfo { limit, remaining, reset_at_unix, observed_at }`**
  is the typed result of parsing the `X-RateLimit-*` headers. A
  429 response is normalised to `remaining: 0` even when the
  header is missing or non-zero. The struct carries the raw unix
  timestamp so the meta layer can compute the absolute
  reset time without re-reading the headers.
- **`github::RateLimitInfo::reset_at(now)`** translates the unix
  timestamp to an absolute `DateTime<Utc>` based on the supplied
  *now*. The translation uses `max(0, reset - now)` so a stale
  header never produces a reset time in the past.
- **`github::rate_limit_from_headers(headers, status)`** is the
  pure parser. `None` when no `X-RateLimit-Remaining` header is
  present. A malformed `X-RateLimit-Reset` falls back to "now".
- **`github::poll_interval_from_headers(headers)`** parses the
  server-suggested `X-Poll-Interval` value (in seconds). A
  missing or malformed header returns `None` so the caller can
  fall back to the configured `poll_interval_seconds`.
- **`meta::CadenceDecision`** is the read-only outcome of the
  precheck. The variants are `Proceed`, `Cadence { next_allowed_at }`,
  and `RateLimited { next_allowed_at }`. `tick_outcome()` maps
  the decision to the canonical `TickOutcome` so the daemon's
  outer loop can route to the documented exit code.
- **`meta::CadenceGate::open(state_dir)`** opens (or creates) the
  metadata file rooted at `state_dir/state_meta.json` and
  reuses the existing `MetaStore` surface. Two `CadenceGate`
  instances over the same `state_dir` observe the same state
  тАФ the meta layer's atomic-rename persistence is process-shared.
- **`meta::CadenceGate::precheck(now, poll_interval_seconds) -> CadenceDecision`**
  is pure read-only. It first checks the persisted rate-limit
  reset; if that is in the future and `remaining == 0`, the
  decision is `RateLimited`. Otherwise it checks
  `last_tick_finished + poll_interval_seconds > now` and
  returns `Cadence` when true. Otherwise `Proceed`.
- **`meta::CadenceGate::record_tick_started(now)`** and
  **`record_tick_finished(now, outcome, http_status, poll_interval_seconds, rate_limit, last_error)`**
  are the read-modify-write entry points the daemon calls at
  tick boundaries. `record_tick_finished` persists the new
  observation via the existing `RateLimitObserver` when *rate_limit*
  is `Some`, and updates `next_allowed_poll_at` from the
  observation's `reset_at` (for `RateLimited`) or from
  `now + poll_interval_seconds` (otherwise).
- **`meta::CadenceGate::record_rate_limit(info)`** and
  **`record_poll_interval(now, seconds)`** are the per-response
  hooks the daemon calls when it observes a rate-limit header
  or `X-Poll-Interval`. The poll-interval recorder always takes
  the longer of the existing and the new value, so the
  server-suggested floor never *shortens* the configured
  cadence.

## Behaviour

- **Header parsing.** Every response the daemon receives is
  scanned for `X-RateLimit-*` and `X-Poll-Interval`. Malformed
  values are tolerated: a missing or non-numeric `Remaining`
  drops the entire observation (the response is not necessarily
  rate-limited); a malformed `Reset` falls back to "now"; a
  malformed `X-Poll-Interval` is ignored.
- **Exhaustion detection.** A response is treated as exhausted
  when `X-RateLimit-Remaining == 0` *or* the status is 429. The
  poll loop short-circuits with a typed `CaduceusError::RateLimited`;
  the meta layer can then persist the observation via the gate.
- **429 mapping.** The `get_url` loop now treats 429 as a typed
  `CaduceusError::RateLimited` rather than a generic
  `GitHubApi { status: 429 }`. The reset is taken from
  `X-RateLimit-Reset` (falling back to `now + 60s` when the
  header is absent) so the persisted observation always has a
  defensible reset time.
- **Cadence gate.** A precheck that finds
  `last_tick_finished + poll_interval_seconds > now` returns
  `Cadence { next_allowed_at }` without doing any HTTP work. A
  precheck that finds a persisted rate-limit observation with a
  future reset returns `RateLimited { next_allowed_at }`. The
  decision is read-only тАФ the daemon's tick wrapper translates
  the decision into a `TickOutcome` and exits 0.
- **Tick bookkeeping.** The gate records `last_tick_started`
  and `last_tick_finished` so the precheck can compute its
  next-allowed time without ambiguity. Recording the start
  alone is not enough (an unfinished tick should not block
  future ticks); the cadence gate uses `last_tick_finished`.
- **Stale-observation protection.** A `record_rate_limit` call
  that produces a `reset_at` older than the persisted
  observation is a no-op. The existing `RateLimitObserver` from
  Task 7.2 enforces the rule; `record_rate_limit` reuses it.
- **Process-shared state.** The on-disk metadata file is the
  source of truth. Two `CadenceGate` instances over the same
  `state_dir` see the same `last_tick_finished` and
  `rate_limit`, so a hand-rolled "two process-equivalent
  clients" test exercises the documented semantics.

## State/schema effects

- The on-disk shape of `<state_dir>/state_meta.json` is

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 2 task packet:

- The precheck is read-only with respect to the filesystem тАФ the
  state_meta.json file is only read, not written, on the
  decision path. The gate persists the result of *real* ticks,
  not the precheck itself.
- The 429 тЖТ RateLimited mapping is contained inside `get_url`;
  poll loops short-circuit cleanly. No state mutation is
  triggered by the 429 alone тАФ the daemon's tick wrapper is
  responsible for calling `record_tick_finished` with the
  matching `RateLimitInfo`.
- A cross-host redirect still refuses to forward the bearer
  token (Task 2.1 invariant, unchanged by this task).
- The rate-limit `Reset` is always translated to an absolute
  `DateTime<Utc>` relative to the observation's `observed_at`,
  so two clients that observe the same header at slightly
  different times record observations with a consistent
  `reset_at`.
- No subprocess execution, no env mutation, no mutation
  outside `<state_dir>/state_meta.json`.

## Residual risks

- **Configurable reset computation.** The current `reset_at`
  uses `max(0, reset_unix - now)` to clamp negative values
  arising from clock skew. A heavily skewed wall clock could
  collapse the clamp and report an immediate reset; operators
  that observe a stuck rate-limited state should check the
  host clock.
- **Header normalisation.** `rate_limit_from_headers` treats
  a 429 as exhaustion even when the `X-RateLimit-Remaining`
  header is missing or non-zero. A malicious or buggy proxy
  that always returns 429 with non-zero `Remaining` would be
  treated as exhausted, blocking the daemon until the reset.
  This matches the documented GitHub semantics.
- **Stale-observation protection relies on absolute `reset_at`.**
  Two clients that observe the same header at slightly
  different times record observations with consistent
  `reset_at` (the unix timestamp wins). The implementation
  refuses the older one and keeps the newer.
