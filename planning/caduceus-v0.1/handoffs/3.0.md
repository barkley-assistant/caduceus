## Public signatures/contracts used

- **`IssueKey::parse(input) -> CaduceusResult<Self>`** is the
  canonical parser for `owner/repo#number` references. The function
  never panics and accepts any casing; the returned struct preserves
  GitHub's canonical case for display/API paths, while the
  `display_key()` accessors normalise to lowercase.
- **`IssueKey::display_key()` returns `lower(owner)/lower(repo)#number`.
- **`QueueState::empty()`** builds a version-1 envelope.
- **`QueueState::entry(&IssueKey) -> Option<&QueueEntry>`** borrows
  an entry by its lowercase display key, skipping keys that fail
  validation.
- **`parse_queue_state(text) -> CaduceusResult<QueueState>`** is the
  canonical deserialiser. JSON errors and version-mismatch errors
  both surface as `CaduceusError::StateCorrupt`. A future version
  produces a clear `"unsupported queue state version: got N,
  expected 1"` error rather than best-effort parsing.
- **`serialize_queue_state(&QueueState) -> CaduceusResult<String>`**
  emits canonical one-line JSON, suitable for stable hashing and
  diff-friendly storage.

## Behaviour

- **Schema stability.** Every struct/enum rejects unknown fields
  via `#[serde(deny_unknown_fields)]`. The daemon refuses a schema
  it does not know — operators must upgrade the daemon before an
  upgrade to the queue file format can land.
- **Timestamps.** `DateTime<Utc>` serialises as `YYYY-MM-DDTHH:MM:SS[.fff]Z`,
  the documented RFC-3339 UTC form. The timestamp tests assert no
  `+00:00` suffix appears.
- **Enum casing.** `Phase`, `TicketType`, and `FinalizationStage`
  all use `rename_all = "snake_case"`. The tests assert the exact
  lowercase wire form (e.g. `"in_progress"`, `"previewed"`,
  `"investigation"`).
- **Map key invariant.** Every BTreeMap key must equal its entry's
  `display_key()`. Mismatches are `StateCorrupt`. The invariant
  catches the "matched casing" case where someone re-keys the
  file by hand and forgets to update the entry's owner/repo case.
- **Version mismatch.** A version other than `QUEUE_FILE_VERSION`
  produces a `StateCorrupt` error with the observed and expected
  versions in the message. The daemon never silently accepts a
  newer or older format.

## State/schema effects

The queue file format is now canonically:

```json
{
  "version": 1,
  "entries": {
    "owner/repo#number": {
      "key":         { "owner": ..., "repo": ..., "number": ... },
      "phase":       "queued" | "in_progress" | "previewed" | "done" | "failed" | "skipped",
      "ticket_type": "code" | "investigation",
      "attempts":    <u32>,
      "last_error":  <string | null>,
      "last_run_id": <string | null>,
      "next_attempt_at": <RFC3339 UTC | null>,
      "finalization": { ... } | null,
      "queued_at":   <RFC3339 UTC>,
      "updated_at":  <RFC3339 UTC>
    }
  }
}
```

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- `parse_queue_state` reads the supplied string only. It never
  touches the filesystem; the daemon's load-on-startup code is the
  sole caller that reads from disk.
- `serialize_queue_state` returns the JSON string in memory; it
  never writes to disk.
- `IssueKey::parse` is a pure function. It does not validate
  against GitHub, does not allocate beyond the returned struct,
  and is safe to call in tight loops.

## Residual risks

- The `entry()` lookup walks the `BTreeMap` O(log n) — for queues
  larger than ~10⁴ entries, a `HashMap<String, QueueEntry>` would
  be faster, but the contract pins `BTreeMap` for stable
  iteration order.
- The lowercase `display_key` cache is computed on every call; a
  future optimisation could memoize it on the struct.
