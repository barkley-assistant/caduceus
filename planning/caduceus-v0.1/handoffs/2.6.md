## Public signatures/contracts used

- **`issue::IssueDetail { key, title, body, labels, comments, trusted_comments, events, fetched_at }`**
  is the canonical detail struct. The fields are documented
  by the worker prompt and the public-voice validator; the
  shape is the wire form the worker reads.
- **`issue::IssueComment { author, body, created_at }`** carries
  one comment. `author` is the GitHub login; `body` is the
  comment text; `created_at` is the post timestamp in UTC.
- **`issue::IssueEvent { kind, actor, created_at, label_name }`**
  carries one timeline event. `kind` is the GitHub event
  string (`labeled`, `unlabeled`, `closed`, `reopened`, …);
  `actor` is the login of the user who triggered the event
  (empty for system events); `label_name` is the label name for
  `labeled` / `unlabeled` events and `None` otherwise.
- **`issue::fetch_issue_detail(client, key, trusted_authors) -> CaduceusResult<IssueDetail>`**
  is the canonical entry point. The function dispatches the
  three HTTP requests concurrently and short-circuits on the
  first error.
- **`issue::MAX_RETAINED_COMMENTS = 100`** and
  **`issue::MAX_RETAINED_EVENTS = 100`** are the documented
  per-detail caps. **`issue::MAX_PAGES = 20`** matches the
  contract's hard limit on paginated endpoints.

## Behaviour

- **Concurrent fan-out.** `fetch_issue_detail` issues the
  three requests via `futures_util::try_join3`. The first
  error short-circuits the join; the other two futures are
  dropped and their in-flight HTTP requests finish (or fail)
  in the background. The persistent HTTP cache is consulted
  per request, so repeat calls are bandwidth-friendly.
- **Pagination.** Comments and events use `per_page = 100`,
  the GitHub-side maximum. The `Link: <…>; rel="next"` header
  is followed until either the header is absent or
  `MAX_PAGES = 20` is reached. The 21st page is a typed
  error, never silent truncation.
- **Chronological normalisation.** Comments are sorted
  ascending by `created_at` and the most recent 100 are
  retained. Events are sorted the same way. The retained
  prefix is the *tail* of the sorted list, so the first
  retained comment is the 51st most recent and the last is
  the most recent.
- **Trusted-comment filter.** The `trusted_authors` list is
  applied after pagination: every comment whose `author`
  matches an entry in the list is also surfaced in
  `trusted_comments`. The filter is empty by default, in
  which case `trusted_comments` is empty; the policy lives
  in `Config::feedback_author_allowlist` (set in Phase 1).
- **Null-field tolerance.** Every wire-format field is
  `Option<T>`; a missing or null value is replaced with a
  documented default (`""` for strings, `Utc::now()` for
  timestamps, `None` for `label_name`). A 200 response with
  `null` for any of the documented keys is not an error.
- **Malformed body.** A 200 response with a body that does
  not parse as the documented JSON shape is a typed
  `CaduceusError::Other { .. }` carrying the JSON parse
  reason. The caller should treat this as a transient
  failure and retry on the next tick.
- **404 on the issue path.** A 404 propagates as a typed
  `CaduceusError::GitHubApi { status: 404, .. }`. The daemon's
  outer loop decides whether to retry or surface the failure.
- **Rate-limit on any branch.** A 429 response is translated
  by the Task 2.1 client into a typed
  `CaduceusError::RateLimited { .. }`; the join surfaces the
  first rate-limited error.
- **Concurrent fan-out test.** The "rate limit on one branch
  of the join" test exercises a 200/200/429 response set;
  the join returns the 429. The other two requests finish
  in the background but their results are dropped.

## State/schema effects

- No on-disk mutation. The fetcher is read-only with respect
  to the queue and the metadata file.
- The cache is read; a 200 response on any of the three
  endpoints is stored with its ETag so the next call within
  the window is served from the cache (Task 2.1).

## Tests added or changed

`tests/issue_detail_test.rs` — 9 tests, all green:

- **Complete parse** — the issue, two comments, and one
  labeled event are decoded into the typed `IssueDetail`.
- **Null body / user tolerance** — every documented field
  is `null`; the result is an empty / defaulted detail that
  parses successfully.
- **Empty data** — an issue with no comments and no events
  returns an `IssueDetail` with empty `comments` and `events`
  vectors.
- **Chronological normalisation** — 150 comments are
  submitted in a single page; only the most recent 100
  survive, in ascending `created_at` order.
- **Multi-page comments** — a 2-page paginated response is
  merged into a single ascending list of 4 comments.
- **Malformed comments** — a 200 with a non-array body
  surfaces a typed `Err` with the JSON parse reason.
- **404 on the issue path** — a 404 surfaces a typed
  `CaduceusError::GitHubApi { status: 404, .. }`.
- **Rate-limit on one branch** — a 429 on the events branch
  surfaces a typed `CaduceusError::RateLimited { .. }` even
  when the issue and comments branches succeed.
- **`Serialize` round-trip** — `IssueDetail::to_context_json`
  produces a JSON document that round-trips back to the
  typed struct with the same field values.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 2 task packet:

- The fetcher is read-only with respect to the queue and the
  metadata file. No mutation anywhere outside the Task 2.1
  cache.
- All HTTP traffic goes through the Task 2.1 client, so token
  redaction, redirect policy, 10 MiB body cap, rate-limit
  detection, and the persistent cache all apply unchanged.
- The 100-comment / 100-event cap is enforced after the
  pagination loop, so an issue with 1000 comments still
  returns a bounded prompt. The chronological normalisation
  is ascending, so the first retained element is the
  51st most recent and the last is the most recent — this
  matches the contract's "most recent 100 in chronological
  order" requirement.
- A 429 response on any of the three branches is a typed
  `CaduceusError::RateLimited { .. }`, never a swallowed
  error. The meta layer (Task 2.4) can persist the
  observation from the daemon's tick wrapper.
- No subprocess execution, no env mutation, no filesystem
  mutation outside the Task 2.1 cache.

## Residual risks

- **In-flight requests after a join error.** `try_join3` drops
  the *futures* when one errors, but the underlying
  `reqwest::Request` may still be in flight on the connection
  pool. The Task 2.1 client does not currently cancel
  in-flight requests; for a daemon process that polls on a
  minute cadence this is negligible, but operators that
  dial the request rate-limit below the upstream timeout
  should observe a few in-flight "ghost" requests on a
  failed tick. Phase 5's run-loop can subscribe to the
  `Request` lifecycle if a hard cancellation is required.
- **Background polling not refreshed.** A new trigger label
  on the issue after the worker launch is not picked up
  until the next tick. The `verify` flow (Task 2.5) is the
  authoritative label re-check before the worker runs, so
  this is intentional.
- **Trusted-comment filter is a *filter*, not a *partition*.**
  `trusted_comments` is a subset of `comments`; the worker
  prompt builder is responsible for de-duplicating when
  rendering. The contract does not pin whether trusted
  comments appear in both vectors, so the safe default is
  to keep them in `comments` and surface a *filter* in
  `trusted_comments`. A future refactor could swap to a
  partition if the worker prompt builder prefers that
  shape.
