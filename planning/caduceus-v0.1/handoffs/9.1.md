## Public signatures/contracts used

- **`migrate::run(from: &Path, state_dir: &Path, dry_run: bool) -> CaduceusResult<MigrationReport>`** —
  the import entry point. Accepts both legacy v0 envelopes
  (`{ "entries": [...] }`) and the current v1 schema
  (`{ "version": 1, "entries": { ... } }`). Differentiates the
  two via `looks_like_v0`, a cheap structural sniff of the
  `entries` field. Refuses to overwrite the live state on
  malformed input; atomic install via `install_state`.
- **`migrate::recover_state(repaired: &Path, state_dir: &Path, clear_marker: bool, hold_daemon_lock: bool) -> CaduceusResult<RecoveryReport>`** —
  corruption-marker recovery. Validates `repaired` as the current
  schema, archives the corrupt original as
  `state.json.corrupt-<unix-ts>`, atomically installs the repaired
  content, and only then clears the corruption marker.
- **`MigrationReport { entries_migrated, entries_skipped, outcome: MigrationOutcome }`**
  — surfaced by the CLI; tests assert on it directly.
- **`MigrationOutcome::{Imported, DryRun, AlreadyCurrent}`** —
  tri-state classifier for the import path.
- **`MigrationReport.outcome == MigrationOutcome::AlreadyCurrent`** —
  an empty delta against an existing live state is a no-op; the
  CLI prints `already current; no changes`.
- **`caduceus::migrate`** is now re-exported through `src/lib.rs`
  implicitly (the module is `pub mod migrate;`), so the CLI and
  tests reach the helpers through the canonical path.

## State/schema effects

- No changes to the `QueueState` / `QueueEntry` / `FinalizationCheckpoint`
  schemas. `parse_queue_state` / `serialize_queue_state` are
  unchanged; the migration path uses them as the canonical
  reader/writer.
- The recovery path moves the corrupt original to
  `<state_dir>/state.json.corrupt-<unix-ts>`. This is the same
  archival pattern the metadata corruption recovery already uses
  (see `src/meta.rs::quarantine_corrupt`).
- The import path emits `<state_dir>/state.json.bak-<unix-ts>`
  on every install, including a first-time install where there
  was no prior content. Operators always have a rollback target
  without needing to dig through filesystem snapshots.
- The daemon lock is acquired by both paths so a concurrent
  tick cannot start while a migration or recovery is in flight.

## Tests added or changed

`tests/migration_test.rs` — 10 tests, all green:

- **`empty_legacy_state_imports_empty_current_state_and_is_idempotent`** —
  empty v0 envelope imports an empty v1 state, leaves a backup,
  and is a no-op on a second invocation (returns
  `AlreadyCurrent`).
- **`legacy_state_with_queued_and_failed_entries_imports_to_current_schema`** —
  v0 records with `queued` and `failed` statuses import into the
  current schema with the right `Phase`, `attempts`, and
  `last_error` per entry.
- **`dry_run_does_not_touch_state_file`** —
  `--dry-run` returns a `DryRun` outcome and does not write
  `state.json`.
- **`malformed_legacy_input_does_not_overwrite_existing_state`** —
  a non-JSON `--from` returns `StateCorrupt` and leaves the
  pre-seeded live state byte-identical.
- **`duplicate_legacy_entries_are_reported_and_skipped`** —
  two legacy records for the same `owner/repo#number` produce
  `migrated=1, skipped=1`.
- **`already_current_state_is_idempotent_no_op`** —
  a v1 input identical to the live state returns `AlreadyCurrent`
  with `migrated=0, skipped=0`.
- **`migration_is_atomic_and_leaves_a_backup`** —
  after import, `state.json` and the newest `state.json.bak-*`
  parse to equal v1 states; the backup file exists.
- **`recovery_validates_supplied_file_under_daemon_lock_and_clears_marker`** —
  the recovery path replaces a corrupt `state.json` + marker,
  archives the original corrupt bytes, and clears the marker.
- **`recovery_refuses_to_install_malformed_repaired_file`** —
  a repaired file that fails v1 parsing leaves the active state
  file unchanged and the marker present.
- **`recovery_refuses_when_corrupt_original_cannot_be_archived`** —
  a read-only `state_dir` causes the recovery to refuse (or, on
  platforms where root can still write, to succeed without
  losing the original).

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 9 gate:

- **#1 (whole-tick lock)** — both `migrate::run` and
  `migrate::recover_state` acquire `DaemonLock::try_acquire` before
  installing. A concurrent tick wins the race and the migration
  returns a `Queue` error (`another tick holds daemon.lock`); a
  cron tick that hits the lock returns `TickOutcome::SkippedConcurrent`
  (exit 0). No state mutation happens while the lock is held by
  another tick.
- **#2 (atomic write with fsync)** — `install_state` writes the
  new content to `<state_dir>/state.json.tmp`, `sync_all()`, then
  `rename` into place. A crash mid-install leaves the active file
  untouched. The corrupt-archive step in `recover_state` is also
  a `rename`, not a write-then-rename sequence.
- **#3 (queue helpers never construct claim paths from raw strings)** —
  unchanged. Migration only reads/writes `state.json`; no claim
  paths are touched.
- **#4 (terminal transitions remove claim)** — unchanged. The
  migration path does not transition any `Phase` value; it
  imports legacy `status` strings into the closest current
  `Phase` variant and leaves `attempts`, `last_error`, and
  `last_run_id` to their pre-existing values.
- **#5 (daemon owns branch name)** — unchanged. Migration only
  imports queue entries; no `FinalizationCheckpoint` branch is
  reconstructed from legacy data.
- **#6 (worker session + supervisor)** — unchanged. Migration
  does not invoke a worker.
- **#7 (Rust owns heartbeats / process lifecycle)** — unchanged.
- **#8 (no credential leakage)** — the migration path never reads
  any GitHub credential; tests use only on-disk fixtures.
- **#11 (rate-limit persisted before exit)** — unchanged.
- **MIGRATION.md does not instruct operators to edit `state.json`,
  `state_meta.json`, or claim files in place.** Every recovery
  path goes through `migrate::recover_state` or `caduceus queue
  reset`. The doc explicitly warns against in-place editing.
- **No `todo!`, `unimplemented!`, placeholder ellipses, ignored
  tests, or network-dependent tests** in `src/migrate.rs`,
  `src/cli.rs`, `tests/migration_test.rs`, or `MIGRATION.md`.

## Residual risks

- **Legacy v0 parser is permissive by design.** It accepts unknown
  fields because the historical monolith emitted v0 records
  without `deny_unknown_fields`. The migration path catches
  malformed envelopes as `StateCorrupt`; a malformed legacy
  `status` string is normalised to `Phase::Queued` so the import
  never drops an issue silently. A future tightening could pin
  the v0 schema with `deny_unknown_fields` and bump
  `LEGACY_FILE_VERSION` accordingly.
- **Backup retention is unbounded.** Every install writes a new
  `state.json.bak-<ts>`; over time the directory accumulates
  backups. `caduceus worktree-gc` does not cover the state
  directory, and `caduceus status` does not currently report the
  count. A follow-up could add a retention sweep or report the
  count so operators notice.
- **`recovery_refuses_when_corrupt_original_cannot_be_archived`**
  is best-effort on platforms where the read-only flag does not
  actually block the test runner. The test asserts the active
  file is unchanged both on the strict path (recovery refused)
  and on the lenient path (recovery succeeded and the installed
  content is valid v1). Both branches land the same invariant.
- **The migration CLI does not currently flag conflicts.** When
  the live state has the same `owner/repo#number` as a v1 input
  but with different content, the migration reports the conflict
  as `skipped` and does not overwrite. Operators must use
  `caduceus queue reset` to reconcile. MIGRATION.md documents
  this in §4.
