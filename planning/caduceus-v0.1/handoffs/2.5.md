## Public signatures/contracts used

- **`verify::VerifyOutcome { Proceed, Skip { reason: SkipReason } }`**
  is the structured result of one verification attempt. `Proceed`
  means the worker may launch; `Skip` means the caller should
  release the claim and exit 0; `Err(_)` means the verification
  hit a transient failure and the retry budget must not be
  consumed.
- **`verify::SkipReason { Closed, PullRequest, LabelRemoved,
  Transferred, NotFound, Malformed }`** is the canonical skip
  reason. `as_str()` returns the stable wire form used by
  operator-facing diagnostics.
- **`verify::verify_trigger(client, key, ticket_type, config) -> CaduceusResult<VerifyOutcome>`**
  is the canonical entry point. The function selects the
  expected label from `config.ticket_label_code` for
  `TicketType::Code` and `config.ticket_label_investigation` for
  `TicketType::Investigation`.

## Behaviour

- **Endpoint.** The verifier issues
  `GET /repos/{owner}/{repo}/issues/{number}` with the canonical
  GitHub headers. The Task 2.1 client handles ETag reuse, the
  10 MiB body cap, the same-host redirect policy, and the
  typed status mapping.
- **404 → `Skip { reason: NotFound }`.** A 404 response means
  the issue was deleted, made private, or never existed. The
  caller should release the claim and exit 0. This is the only
  status that maps to a skip; everything else is either
  `Proceed` or `Err`.
- **403 → `Err`.** A 403 is an auth failure; surfacing the
  error preserves the retry budget so a credential rotation
  can recover.
- **429 → `Err(CaduceusError::RateLimited { .. })`.** A 429 is
  rate-limit exhaustion; the daemon's tick wrapper is
  responsible for persisting the observation via the
  Task 2.4 `CadenceGate` and exiting 0.
- **5xx and other non-2xx/304 → `Err`.** A server failure
  preserves the retry budget; the caller's outer loop decides
  whether to retry.
- **Closed state → `Skip { reason: Closed }`.** A response
  with `state: "closed"` is a closed issue, regardless of
  whether the trigger label is still on the object.
- **Pull-request object → `Skip { reason: PullRequest }`.**
  The unified `/issues` endpoint returns PR objects; the
  verifier rejects any object with a `pull_request` field.
- **Removed label → `Skip { reason: LabelRemoved }`.** The
  trigger label is no longer in the issue's `labels` array
  (case-insensitive). A label-set that contains *both* the
  code and investigation trigger labels is also a `Skip`
  because the object is ambiguous per the polling contract.
- **Transfer → `Skip { reason: Transferred }`.** GitHub's
  redirect chain (the Task 2.1 client follows up to
  `MAX_REDIRECTS = 3` same-origin hops) is recorded on the
  response's `final_url`. The verifier compares
  `final_url` to the requested `owner/repo`; a same-host
  redirect to a different path (e.g. `/repositories/12345`)
  yields `Transferred`. A cross-origin redirect is an error
  raised earlier in the client.
- **Malformed body → `Err`.** A 200 response with a body
  that fails to parse is a transient failure: the caller can
  retry; if the malformed state is permanent, the retry
  budget will be exhausted by the daemon's outer loop.

## State/schema effects

- No on-disk mutation. The verifier is a read-only operation.
- The queue is not touched; the caller is responsible for
  admission and claim release.
- The cache is read; a 200 response is stored with its ETag so
  the next call within the window is served from the cache
  (Task 2.1).

## Tests added or changed

`tests/verify_test.rs` — 14 tests, all green:

- **Both ticket types proceed** — a code ticket on an object
  with the code label returns `Proceed`; an investigation
  ticket on an object with the investigation label returns
  `Proceed`.
- **Mismatched ticket type** — a code ticket on an object
  that only has the investigation label returns
  `Skip { LabelRemoved }`.
- **Label removed** — the trigger label is gone; the result
  is `Skip { LabelRemoved }`.
- **Closed issue** — `state: "closed"` yields
  `Skip { Closed }`.
- **404** — `404 Not Found` yields `Skip { NotFound }` (not an
  error), so the retry budget is not consumed.
- **403** — `403 Forbidden` is an `Err(CaduceusError::GitHubApi)`
  that surfaces to the caller without retry-budget loss.
- **429** — `429 Too Many Requests` is a typed
  `CaduceusError::RateLimited { remaining: 0, .. }` so the
  meta layer (Task 2.4) can persist the observation.
- **Transfer** — a 301 to `/repositories/12345` followed by a
  200 yields `Skip { Transferred }` because the response's
  `final_url` no longer contains the requested
  `octocat/hello-world`.
- **Both-label ambiguity** — an object with both trigger
  labels yields `Skip { LabelRemoved }` per the polling
  contract.
- **Pull-request object** — an object with a `pull_request`
  field yields `Skip { PullRequest }`.
- **Malformed body** — a 200 with `not json` is a typed
  `Err(CaduceusError::Other { .. })` carrying the JSON parse
  reason.
- **`SkipReason::as_str`** is stable and round-trippable.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 2 task packet:

- The verifier is read-only with respect to the queue. A
  `Skip` outcome is the documented signal to release the
  claim, but the release itself is the caller's responsibility
  (Phase 3).
- 5xx / 403 / 429 / parse errors propagate as `Err` so the
  retry budget is preserved — the contract explicitly says
  "authentication/rate-limit/server failures return errors
  and do not consume a retry".
- 404 returns `Ok(VerifyOutcome::Skip { reason: NotFound })`,
  not an error, because the contract says "Closed, transferred,
  deleted, or unlabeled issues return `false`". The deletion
  case is the 404 path.
- All HTTP traffic goes through the Task 2.1 client, so
  token redaction, redirect policy, 10 MiB body cap, and the
  persistent cache all apply unchanged.
- The transfer detection reuses the Task 2.1
  `Response::final_url` field, so no separate HTTP-aware
  parsing lives in the verifier.

## Residual risks

- **Same-owner / different-repo transfers.** A
  `/repos/octocat/hello-world/issues/7` redirect to
  `/repos/octocat/heaven/issues/7` would still match the
  `/{owner}/{repo}/` substring check and return `Proceed`.
  The 301-style "issue moved to a new repository" redirect
  GitHub issues is a `/repositories/{id}` path, which the
  substring check rejects. An issue that is moved *within the
  same owner* via the standard `Repository.transfer` API is
  rare and the polling pass picks it up on the next tick.
- **Label-revived edits.** The verifier runs once per tick.
  An operator who removes and re-adds the label within the
  tick window will see a transient `Skip` followed by a
  `Proceed` on the next tick; the polling layer (Task 2.3)
  re-discovers the issue so the queue is consistent.
- **No retry inside the verifier.** A transient 5xx propagates
  to the caller; the daemon's outer loop is responsible for
  the retry policy. The contract says "do not consume a
  retry" which the implementation honours; the next-tick
  re-attempt lives in the daemon's run-loop wiring (Phase 5).
