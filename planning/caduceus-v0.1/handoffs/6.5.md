## Behaviour (per the contract)

1. **Validate first.** The summary is validated through `validate_comment` (Task 6.6) **before** any HTTP request. A rejected text returns `CaduceusError::Other(public-voice: ...)` and never touches the network. The integration test `failure_voice_rejection_prevents_http` confirms the contract.
2. **Comment idempotency.** The function lists the issue's comments and looks for one whose `body` starts with the `run_id`-scoped marker (`<!-- automation-failure:<run_id>` for failure, `<!-- automation-investigation:<run_id>` for investigation). If present, no POST is made; the function reports the no-op. The integration tests `failure_existing_marker_skips_post` and `investigation_existing_marker_skips_post` confirm the mock's POST handler has `expect(0)`.
3. **Generic failure body.** The failure-comment body names the `run_id` and the summary, but **never** claims a local transcript is a public link. The integration test `failure_comment_does_not_claim_local_transcript_is_public` pins this: the rendered body never contains `/tmp/wt`, `/state/`, or `.transcript`.
4. **Investigation body.** The investigation body combines the summary with the bounded, injection-safe artifact renderer (re-used from the PR-body path). The integration test `investigation_marker_includes_run_id` pins the marker prefix, the run_id, and the JSON code-fence with the rendered artifact.
5. **Comment API failure preserves the worker error.** A 500 from the comments POST surfaces as `CaduceusError::GitHubApi { status: 500, … }`. The integration test `failure_comment_api_failure_returns_typed_error` confirms the error reaches the caller.
6. **Withdrawal is the orchestrator's job.** The function does not check the issue's trigger-label state. The orchestrator calls the function only after re-validating the issue is still open with the expected labels; if the user has withdrawn, the orchestrator skips the call and transitions the entry to `Skipped`.
7. **Investigation label removal is best-effort.** The `label_removed` flag is the operator's audit trail. In v0.1 the flag is always `false`; the orchestrator (Phase 6) owns the actual `DELETE` since the `Client` exposes only `GET` and `POST`. The function reports the comment-posted outcome; the label-removed flag is left at `false` until the orchestrator extends the surface.
8. **No git mutation.** The failure and investigation paths talk only to the GitHub API. The runner is not invoked; no `git commit` / `git push` / `gh` calls are made.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No `gh` or `git push` here.** The failure and investigation steps talk only to the GitHub API; the runner is not invoked.
- **No PAT in arguments, URLs, or env.** The `Authorization` header carries the bearer token in the same-origin path only; the runner's credential scrubbing applies.
- **No label change.** The label removal is a v0.1 best-effort flag; the actual `DELETE` is the orchestrator's job.
- **No worktree teardown.** The orchestrator tears the worktree down after the failure/investigation step lands.

## Residual risks

- The investigation-label-removal is left at `false` for v0.1. The orchestrator (a Phase 6 task) must call `Client::delete` (a new method) to perform the `DELETE /repos/{owner}/{repo}/issues/{number}/labels/{name}`. The integration test pins the flag at `false`; the contract is the orchestrator's job.
- The failure-comment body is generic. An operator reading the body will not know which `run_id` is the failing run unless they cross-reference the queue. The marker includes the `run_id` so the orchestrator can join the comment to the queue; the comment itself stays generic to avoid leaking any local path or token.
- The investigation-comment body uses the bounded, injection-safe artifact renderer. A very large `artifacts` map is still bounded by the 65 536-byte comment cap (`validate_comment` enforces the cap). The body is JSON-encoded inside a triple-backtick fence; the fencer is dynamically chosen in the PR-body path; the investigation-comment path uses a fixed-length fence for simplicity. A test `investigation_marker_includes_run_id` confirms the body shape.
- The 429 rate-limit error surfaces as `CaduceusError::RateLimited`. The integration test for the rate-limit case is in `tests/issue_close_test.rs` (Task 6.4); the failure path shares the same client so the same behaviour applies.
