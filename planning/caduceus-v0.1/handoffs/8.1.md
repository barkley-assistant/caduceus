## Public signatures / contracts used

- **`invoke_harness(worktree, prompt_file, run_id, labels, branch_name, extra_argv=())`** is the only function operators are expected to edit. Its signature is now stable: positional `Path` / `Path` / `str` / `Sequence[str]` / `str` arguments plus an optional `extra_argv: Sequence[str]` so tests and operators can pass-through arguments without modifying the bridge's CLI shape. The default body invokes `opencode run --agent gentle-orchestrator -f <prompt>`.
- **`read_required_env(env)`** raises `SystemExit(EXIT_MISSING_ENV)` (==2) when any required `CADUCEUS_*` value is missing or empty. The diagnostic lists missing keys without echoing values.
- **`parse_labels(raw)`** rejects malformed JSON, top-level non-arrays, non-string elements, and the legacy CSV shape; raises `SystemExit(EXIT_MALFORMED_LABELS)` (==2).
- **`verify_prompt(path)`** raises `SystemExit(EXIT_MISSING_PROMPT)` (==2) when the prompt file does not exist or is not a regular file.
- **`main(env, argv)`** uses `Mapping[str, str]` and `Sequence[str]` to match `os.environ`. Returns 0 on harness success, the harness exit code on failure, and the documented 2/127/126 codes on validation errors. It does NOT touch state directories, heartbeats, or `worker-result.json`.

The bridge does **not** call `signal.signal` itself — the daemon's worker supervisor owns SIGINT/SIGTERM/timeout forwarding to the whole process group. `tests/bridge_test.py::test_bridge_does_not_install_python_signal_handlers` pins this; `test_signal_forwarded_to_harness_via_subprocess` exercises the process-group signal plan through a real subprocess.

## State / schema effects

- **No new on-disk formats.** The bridge has no IO footprint of its own; the daemon continues to own queue state, metadata, heartbeats, transcripts, and finalization.
- The canonical `REQUIRED_ENV_VARS` (9 entries) are now structurally aligned with `crate::fixtures::CANONICAL_WORKER_ENV_VARS`, and both are mirrored in `tests/hermes_plugin_test.py::TEST_RUNTIME` so future drift surfaces in three test suites at once.
- `plugin.yaml` adds `requires_env: []` (still inside the v0.18.2 supported field set) so the manifest's intent is explicit.

## Residual risks

- The bridge test fixture `tests/fixtures/bridge_harness.py` is invoked through a `bin/opencode` shell wrapper that exits when the test fixture path imports fail. A future tightening could replace the wrapper with a thin Rust harness so the test no longer relies on Python being on PATH in the test process's environment.
- `tests/docs_contract_test.rs` parses the bridge with `trimmed.starts_with("REQUIRED_ENV_VARS:")` and the manifest set via brace counting. Both are heuristic; future refactors that change the literal text could require test updates. The contract-side guarantees (the comments and the fixture) make the surface explicit so a breaking change would be a conscious decision rather than a silent break.
- The `requires_env: []` field in `plugin.yaml` is part of the v0.18.2 supported field list but Caduceus does not actually require any environment variables at plugin load time. The list is kept explicit so the contract is unambiguous in the manifest.
- `test_signal_forwarded_to_harness_via_subprocess` uses `os.killpg(os.getpgid(proc.pid), SIGINT)` to send the signal to the whole group the daemon would manage. The bridge subprocess is run with `start_new_session=True`, so it is its own session leader. In a real daemon tick the worker supervisor places the harness in its own process group; we test the same shape.
