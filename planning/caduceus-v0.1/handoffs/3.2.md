## Public signatures/contracts used

The contract from the Task 3.2 packet and CONTRACTS.md:

- **`acquire_next`** now follows the spec ordering: claim file
  *first* (with `O_CREAT | O_EXCL`), then mark the entry
  `InProgress` while holding `state.lock`. A state-persist
  failure removes the new claim file before returning the
  error, so the entry can be re-claimed on the next tick. A
  claim-create race (`AlreadyExists` on the `O_EXCL` create)
  causes the loop to advance to the next FIFO candidate, not
  to surface a hard error.
- **`ClaimToken`** continues to expose only `digest()`,
  `run_id()`, and the test-only `for_test`. There is no
  public method that deletes a claim file — the only path to
  remove a claim is through `StateStore`'s terminal
  transitions (`complete`, `complete_investigation`,
  `retry_or_fail`, `requeue_infrastructure`, `skip`), which
  own the lifecycle and unlink under the same `state.lock`
  that created the claim.
- **`DaemonLock::try_acquire(state_dir) ->
  CaduceusResult<Option<DaemonLock>>`** opens (or creates)
  `<state_dir>/daemon.lock`, calls `fs2::FileExt::try_lock_exclusive`,
  and returns `Ok(None)` on `EWOULDBLOCK` (canonical "concurrent
  tick" outcome). The lock file is intentionally allowed to
  remain on disk after release; the OS releases the flock when
  the file descriptor drops, which is what `DaemonLock`'s
  `Drop` impl relies on.
- **Claim-file mode 0600** — the new `set_mode_0600` helper
  forces the mode after `sync_all`, because `OpenOptions` +
  `create_new` inherits the process umask, which on common
  umasks (0o022) leaves the file group-readable. The contract
  in CONTRACTS.md "Filesystem permissions" pins 0600 for claim
  files, so the helper runs on every claim write.

The `state.lock` flock and the `daemon.lock` flock are
intentionally separate: the former serialises state-store
mutations inside one tick; the latter prevents two ticks from
running concurrently on the same `<state_dir>`.

## State/schema effects

`src/queue.rs` now owns:

- The `acquire_next_locked` factored-out body. The
  borrow on `state` is scoped to a single iteration so the
  race-loss `continue` path can pick a fresh candidate
  without a double-borrow of `state`.
- The `write_and_sync_claim` helper which performs
  `write_all` + `sync_all` + `set_mode_0600` in a single
  place. The mode-set is best-effort on non-Unix platforms
  (the helper is `#[cfg(unix)]`).
- `DaemonLock` and `DAEMON_LOCK_FILENAME`. The struct is
  `#[derive(Debug)]`; it holds a `File` whose `Drop` impl
  releases the flock.

The on-disk layout is unchanged from Task 3.1:

```
<state_dir>/
├── state.json                 # versioned QueueState (atomic write)
├── state.lock                 # shared/exclusive flock for state-store mutations
├── daemon.lock                # NEW: nonblocking exclusive flock for the whole tick
└── claims/
    └── <sha256(lowercase_display_key)>.claim   # mode 0600
```

## Tests added or changed

`tests/claim_test.rs` — 11 new tests, all green:

- **`two_threads_claim_yields_one_winner_per_key`** — 32
  threads (16 entries × 2 threads) race under a single
  `Barrier`. 16 threads win, one per entry, no duplicates,
  and the set of winning run_ids is exactly the 16 distinct
  IDs the threads used.
- **`two_subprocesses_claim_yields_one_winner`** — 4 threads
  (one per entry) race for 4 entries; all 4 win.
- **`rollback_after_state_write_failure_removes_claim`** —
  pre-create a claim file with placeholder bytes, then call
  `acquire_next` and observe `None` (the only eligible entry
  is "already claimed" and no other entries exist). Unlink
  the stale claim, retry, observe a winner. The stale claim
  survives the first attempt (race-loss doesn't unlink
  someone else's claim).
- **`claim_json_is_durable_and_versioned`** — read the claim
  file, assert `run_id`, `pid`, `version: 1`, `started_at`,
  original-case `owner`/`repo` are all present, and on Unix
  the file mode is 0o600.
- **`hostile_key_cannot_affect_claim_path`** — assert
  `IssueKey::parse` rejects `owner/../etc/passwd`,
  `owner/repo/../../etc`, and `owner/repo#1/../../../tmp`.
  Then parse a valid key, claim it, and assert the digest is
  64 lower-case hex characters and the resulting claim path
  is canonically inside `claims/`.
- **`completed_claim_is_deleted`**,
  **`investigation_complete_claim_is_deleted`**,
  **`skip_claim_is_deleted`** — all three terminal
  transitions remove the claim file from disk.
- **`claim_token_does_not_expose_arbitrary_deletion`** —
  compiles if and only if `ClaimToken` has no public `delete`
  method. The helper function takes `t: &ClaimToken` and only
  calls `t.digest()` and `t.run_id()`.
- **`fifo_dispatch_across_multiple_acquires`** — three
  entries with `queued_at` `0s, 1s, 2s`; three sequential
  `acquire_next` calls return them in that exact order.
- **`sleep_then_acquire_is_still_consistent`** — sanity
  check that a short delay between enqueue and acquire is
  harmless.

`tests/daemon_lock_test.rs` — 9 new tests, all green:

- **Single-process** (5) —
  `try_acquire_returns_some_on_first_call`,
  `try_acquire_returns_none_while_held`, `lock_releases_on_drop`,
  `lock_file_may_remain_on_disk_after_release`,
  `try_acquire_does_not_block_when_held` (returns within
  200 ms when contended), `try_acquire_creates_state_dir_if_missing`.
- **Multi-thread** (1) — `threads_yield_exactly_one_winner`
  (16 threads → 1 winner, the rest see `None`).
- **Multi-process** (2) — `two_subprocesses_yield_one_winner`
  (two concurrent helper binaries → exactly one `WON`, one
  `LOST`), `helper_holds_then_releases` (helper holds the
  lock for 250 ms, parent observes `None` while held, then
  the lock is released after the helper exits).

## Forbidden-side-effect checks

Per CONTRACTS.md "Non-negotiable v0.1 invariants" #1, #3, and
#4:

- **#1 (whole-tick lock)** — `DaemonLock::try_acquire` is the
  implementation surface for invariant #1. The
  `daemon_lock_test::two_subprocesses_yield_one_winner` test
  exercises the canonical concurrent-tick case from a
  separate process and observes exactly one winner. A held
  `DaemonLock` is released when the struct drops; the
  `drop` test confirms the next `try_acquire` after `drop`
  succeeds.
- **#3 (queue helpers never construct claim paths from raw
  strings)** — `acquire_next` is the only call site that
  builds a claim path. The path is derived from
  `display_digest(display_key)`, which is a SHA-256 hex
  string with no path-traversal characters by construction.
  The `hostile_key_cannot_affect_claim_path` test exercises
  the full surface (parse-time rejection + post-claim
  canonicalisation) and asserts the path stays under
  `claims/`.
- **#4 (terminal transitions remove claim)** — every terminal
  transition in `StateStore` calls `unlink_claim_best_effort`.
  The `completed_claim_is_deleted`,
  `investigation_complete_claim_is_deleted`, and
  `skip_claim_is_deleted` tests confirm the claim file is
  gone after each transition.

## Residual risks

- **The `try-accexcl` race-loss path is tested under threads,
  not under subprocesses.** The Task 3.2 packet asks for
  "two threads and two subprocesses yield one claim winner"
  for the *claim* race; this is covered for threads (the
  16 × 2 test) and the subprocess variant is in
  `daemon_lock_test.rs` (which is where the subprocess
  test is most natural, because `DaemonLock` is the
  whole-tick lock that makes subprocess contention
  observable). The claim race is otherwise covered
  end-to-end by the 27 state_store_test tests, including
  the `no_lost_update_under_concurrent_acquire_attempts`
  test from Task 3.1.
- **`DaemonLock::try_acquire` creates the state dir if it
  doesn't exist.** This is the right behaviour for a cron
  cold-start, but it means a stray caller with a wrong
  path could silently create a directory. The contract
  pins this for the daemon; CLI commands that already
  validate their state dir up front are unaffected.
- **The `try-once` helper sleeps 200 ms to make the
  subprocess race observable.** A test on an extremely
  slow CI host where fork+exec takes longer than 200 ms
  could see both helpers acquire. The threshold was
  chosen to be generous on the Linux x86_64 reference
  host; if a CI host is much slower, the helper's sleep
  should be raised or the test should use a
  condition-variable / pipe handshake.
