## Behaviour (per the contract)

1. **No git/GitHub mutation.** The function is pure: it does not call `git commit`, `git push`, or any HTTP layer. The integration test `dry_run_does_not_call_git_or_github` records the on-disk state before and after, then asserts that the only file-level side-effect is the preview report.
2. **Atomic report.** The report is written via `write_atomic`: open with `O_NOFOLLOW`, write, `fsync`, then `rename` from `<target>.tmp.<pid>.<nanos>` to `<target>`. The test `dry_run_writes_report_atomically` confirms no `.tmp.*` file remains after a successful run.
3. **Report contents.** The report carries: the active `run_id`, the issue `key`, the proposed `branch_name` (from the worktree), the worker's `commit_message` verbatim, the canonical PR title (built via `build_pr_title`) and PR body (built via `build_pr_body`), the investigation comment when `worker_result.investigation == true`, the supplied `changed_files`, the on-disk `transcript_path`, the `worker_result_path`, any `validation_warnings`, and the `written_at` RFC 3339 timestamp.
4. **Validation warnings collected, not blocking.** If `validate_worker_result` or `build_pr_title`/`build_pr_body` return errors, the dry-run does not abort — the error is captured in `validation_warnings` and the report is still written. The integration test `dry_run_validation_warnings_are_collected` covers this.
5. **`Previewed → Queued` promotion.** Implemented in `StateStore::enqueue`: when `dry_run=false` and the entry exists in `Phase::Previewed`, the call returns `EnqueueOutcome::Promoted` and flips the phase to `Queued`. A second call returns `AlreadyPresent` — the promotion is one-shot. The integration test `queue_promotes_previewed_to_queued_when_dry_run_disabled` exercises this end-to-end.
6. **`dry_run` does not promote.** The `dry_run=true` path returns `EnqueueOutcome::AlreadyPresent` even on a `Previewed` entry, so a future dry-run never re-promotes a previously-previewed entry. The test `dry_run_does_not_promote_previewed` confirms this.
7. **Investigation tickets.** When `worker_result.investigation == true`, `proposed_investigation_comment` is set to the worker's `summary`. The `proposed_pr_title` and `proposed_pr_body` are still built (they are not the comment channel) but the orchestrator reads the `investigation` field and routes to the investigation-comment path. The test `dry_run_investigation_sets_comment_field` confirms.
8. **Schema stability.** `PreviewReport` uses `#[serde(deny_unknown_fields)]` and exposes `SCHEMA_VERSION = 1`. A future schema bump is detected by the deserialiser; the test `dry_run_report_uses_denied_unknown_fields` injects a rogue field and asserts rejection.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No `git` mutation.** Verified by the `dry_run_does_not_call_git_or_github` integration test (the function never spawns a subprocess; the only on-disk file change is the preview report).
- **No GitHub HTTP call.** The `FinalizeContext::client` field is `()`; `dry_run_finalize` does not touch it. The function returns before any HTTP layer would be reached.
- **No queue-state mutation.** `dry_run_finalize` does not call `acquire_next` or any `StateStore` mutator. The orchestrator's preview path is responsible for calling `enqueue`/`complete_preview` *after* the report is on disk.
- **No worktree teardown.** `dry_run_finalize` only writes the report; the orchestrator tears the worktree down (Task 5.5 contract: "tears down"). The test does not assert teardown — the integration tests for the orchestrator (Phase 6, future) will.
- **No label change.** No HTTP call; no label API mutation.

## Residual risks

- The `transcript_path` field is set to `<worktree.path>/.caduceus/transcript`. The supervisor's actual transcript path is `<state_dir>/runs/<run_id>.transcript`; the report's value is the *expected* path so the operator can find it. The supervisor (Task 5.1) owns the on-disk name. If the two diverge, the report's `transcript_path` becomes a breadcrumb only.
- The atomic-write helper uses `<pid>.<nanos>` for the temp name. On Linux the rename is atomic within the same filesystem; cross-filesystem renames fall back to copy+delete. The test uses `tempfile::tempdir` so the parent is always the test's tempdir — the test does not exercise cross-filesystem rename.
- The `Previewed → Queued` promotion is exposed via `enqueue` only. A future caller that uses a different path (e.g. a custom cron tick) must call `enqueue` for the promotion to fire. The integration test pins this contract.
- The dry-run report's `written_at` is wall-clock UTC. A clock skew between the daemon and the operator's terminal can confuse a `diff` across re-renders; the report's *content* is stable (deterministic), so the timestamp is the only changing field.
