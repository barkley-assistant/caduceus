## Public signatures/contracts used

- **`Config::from_raw(raw: RawConfig, ctx: &LoadContext) -> CaduceusResult<Self>`**
  is the single, fully validated entry point. The deserialisation layer
  (`RawConfig`) is public so YAML callers and tests can build one;
  `LoadContext` carries `hermes_home`, `plugin_root`, and the captured
  `RawEnv`. Task 1.3 plugs its env-aware resolver into this seam.
- **`Config::test_defaults(root: &Path) -> Self`** produces deterministic
  root-anchored defaults. No host-dependent `Config::defaults()`
  constructor exists, per the contract.
- **`expand_leading_tilde(path: PathBuf) -> PathBuf`** handles the
  single non-shell path-expansion rule from the contract.
- **`is_valid_repo_slug(repo: &str) -> bool`** is the public
  owner/repo validator reused by Phase 2 watching and Phase 4 worktree
  discovery.
- `Config::load()` still returns a Task-1.3 stub error. Wiring the env
  chain through `Config::from_raw` is the next task.

## State/schema effects

None. `Config::from_raw` is read-only; it builds an in-memory
`Config` and never touches the filesystem outside of `expand_leading_tilde`
(which only consults `HOME`/`USERPROFILE`) and `validate_secure_path`
(which calls `std::fs::symlink_metadata` to reject existing symlinks
or non-directories). No file is created, modified, or removed.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- `Config::from_raw` performs **no** filesystem mutation. The only
  filesystem calls are `std::fs::symlink_metadata` (read-only, via
  `validate_secure_path`) and `std::env::var_os` for `HOME` /
  `USERPROFILE`. No file is created, modified, or removed.
- The valid YAML round-trip does not open a socket, call out to
  GitHub, or spawn a subprocess.
- `Config::load()` continues to return a `CaduceusError::Config`
  stub, so no operator can accidentally trigger network or git
  operations through the new code.
- No test mutates the host environment beyond `HOME` for one
  tilde-expansion case, with full restoration in a guard pattern.

## Residual risks

- The Hermes-side env-aware loader (Task 1.3) is the natural
  follow-on: it will call `Config::from_raw` from inside the
  resolution chain and merge `CADUCEUS_DRY_RUN` /
  `CADUCEUS_GITHUB_TOKEN` / `GITHUB_TOKEN` into the validated
  struct. `RawEnv` is already in place for that.
- The worker-command interpolation rejects `$HOME`, `${HOME}`,
  `~`, and `$USER`. The exact list is in
  `FORBIDDEN_INTERPOLATION_TOKENS`. Future harnesses that need
  finer expansion should layer their own allowlist before reaching
  the daemon; the daemon intentionally does not perform shell
  expansion.
- `Config::load()` continues to fail fast. Any future test that
  needs the env-aware chain should write a focused fixture rather
  than relying on the host environment.
