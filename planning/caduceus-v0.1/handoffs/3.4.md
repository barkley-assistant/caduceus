## Public signatures/contracts used

The Task 3.4 contract is exercised end-to-end:

- **`StateStore::retry_or_fail(claim, error, budget)`** preserves
  the Task 3.1 semantics: increments `attempts`, sets
  `next_attempt_at = now + 300s` (the documented backoff), returns
  to `Queued` below the budget, transitions to `Failed` at the
  budget, always unlinks the claim. `budget == 0` is a
  configuration error (not a transition).
- **`StateStore::requeue_infrastructure(claim, error, not_before)`**
  records the diagnostic and eligibility time without
  incrementing `attempts` — the contract for rate-limit and
  infrastructure-failure requeue.
- **`StateStore::skip(claim, reason)`** transitions to terminal
  `Skipped` and unlinks the claim. `Skipped` entries cannot be
  re-acquired.
- **`StateStore::complete(claim)`** /
  **`complete_investigation(claim)`** /
  **`complete_preview(claim)`** are the three success
  transitions; `complete_with` now clears `last_error` on every
  successful terminal transition (the contract says "success
  clears error/backoff").
- **`StateStore::reset_entry(key, clear_finalization)`** is the
  operator-driven recovery. It:
  - refuses entries whose phase is not `Failed` or `Skipped`
    (a `Queued` / `InProgress` / `Previewed` / `Done` entry is
    an explicit operator error);
  - refuses if a claim file exists for the entry's digest
    (the active-claim refusal);
  - resets `attempts` to 0, clears `last_error`,
    `last_run_id`, and `next_attempt_at`, transitions to
    `Queued`;
  - by default preserves the persisted
    `FinalizationCheckpoint` so a follow-up tick resumes
    from the saved branch / PR;
  - with `clear_finalization=true` drops the checkpoint and
    returns it via `ResetOutcome::dropped_checkpoint` so the
    CLI can surface the branch / PR / commit OID in a
    warning. The daemon never deletes the remote branch or
    PR — that responsibility stays with the operator.
- **`StateStore::finalization_for(key)`** is a small read-only
  helper used by the CLI's dry-run path to surface the
  checkpoint without going through `snapshot`.
- **`caduceus queue reset <owner/repo#number>`** is the only
  v0.1 recovery operation. The CLI:
  - parses the issue ref via `IssueKey::parse`;
  - loads config via `$CADUCEUS_CONFIG` (`Config::load_from`)
    when set, falling back to `Config::load`;
  - acquires `DaemonLock` so a concurrent tick cannot run
    while we mutate state;
  - calls `StateStore::reset_entry`;
  - prints `reset <key> to Queued` on success;
  - on forced reset, emits two `warning:` lines: one with the
    full checkpoint contents (branch, run_id, stage,
    commit_oid, pr_number, pr_url) and one that explicitly
    states "the remote branch and PR were NOT deleted;
    reconcile manually if appropriate".
  - on `--dry-run` skips the daemon lock and reads state via
    `StateStore::open` + `snapshot` + `finalization_for` to
    print "would reset …" / "would drop …" lines without
    mutating anything.

## State/schema effects

`src/queue.rs` now also owns the `ResetOutcome` value type and
the `reset_entry` / `finalization_for` methods. The
`FinalizationCheckpoint` schema is unchanged — it is read by
`finalization_for` and conditionally cleared by
`reset_entry`. No on-disk layout changes beyond the existing
queue file (state.json), claim files, and the daemon.lock.

The CLI surface gains one new flag,
`--force-finalization-reset`. Clap wiring lives in `src/cli.rs`;
the corresponding runtime check is in `run_queue_reset`.

## Tests added or changed

`tests/retry_test.rs` — 20 new tests, all green:

- **Failures 1/2/3/4** (4) — exact three-failures semantics;
  the fourth call (seeded at `attempts=3`) is terminal `Failed`
  without further increment.
- **Backoff / eligibility** (3) — `next_attempt_at` blocks the
  next `acquire_next` until the timestamp; cancellation
  requeues with `not_before = now` and is immediately eligible.
- **Infrastructure failure** (1) — `attempts` is preserved.
- **Rate-limit / cancellation** (2) — explicit requeue
  eligibility windows.
- **Zero budget** (1) — `Config` error before any state change.
- **Success clears error/backoff** (1) — `complete` zeroes
  `last_error` and `next_attempt_at`.
- **Terminality** (4) — `Skipped`, `Done`, `Failed`, and
  `Previewed` (during dry) cannot be re-acquired; a non-dry
  enqueue promotes `Previewed` → `Queued`.
- **Claim removal** (2) — every terminal and requeue transition
  unlinks the matching claim file.
- **Wrong run id** (2) — `retry_or_fail` and `complete` reject
  a mismatched `ClaimToken`.
- **Bounded error string** (1) — a 1 KiB error round-trips
  without panic; a future amendment may cap it.

`tests/queue_reset_cli_test.rs` — 13 new tests, all green:

- **Reset success** (2) — `Failed` and `Skipped` entries are
  moved back to `Queued` with attempts=0 and the diagnostic
  fields cleared.
- **Reset refusal** (2) — `Queued` and `Done` entries are
  rejected with a precise phase-name error.
- **Active-claim refusal** (1) — pre-creating a claim file
  blocks the reset; the state is unchanged.
- **Checkpoint preservation** (1) — by default the persisted
  `FinalizationCheckpoint` survives the reset.
- **Forced reset** (2) — `--force-finalization-reset` clears the
  checkpoint, emits two `warning:` lines (full checkpoint +
  "NOT deleted; reconcile manually"), and the state file no
  longer carries the checkpoint.
- **Dry-run** (2) — `--dry-run` produces "would reset …" and
  "would drop …" without mutating state; without
  `--force-finalization-reset` the dry-run still reports
  "would preserve".
- **Input validation** (2) — a malformed issue ref and an
  unknown entry are both rejected with non-zero exit.
- **Concurrent tick refusal** (1) — holding the daemon lock
  while invoking the CLI returns a non-zero exit with a clear
  message; state is unchanged.

## Forbidden-side-effect checks

Per CONTRACTS.md "Non-negotiable v0.1 invariants" #1, #3, #4
plus the queue-reset clause ("never deletes a remote branch or
PR automatically"):

- **#1 (whole-tick lock)** — the CLI takes `DaemonLock`
  before mutating state, refuses with a clear error if a
  concurrent tick holds it. The
  `reset_refuses_when_daemon_lock_held` test exercises this
  path through a parent-process lock.
- **#3 (queue helpers never construct claim paths from raw
  strings)** — unchanged from prior tasks; `reset_entry` never
  creates or inspects claim filenames, only the digest for
  the active-claim refusal check (derived via
  `display_digest(key.display_key())`).
- **#4 (terminal transitions remove claim)** — every terminal
  transition in `StateStore` continues to call
  `unlink_claim_best_effort`. The retry / complete / skip /
  complete_investigation / complete_preview paths are covered
  in `tests/retry_test.rs`.
- **Queue reset never deletes a remote branch or PR** — the
  CLI emits "warning: the remote branch and PR were NOT
  deleted; reconcile manually if appropriate" on every
  forced reset. The
  `forced_reset_does_not_delete_remote_branch_or_pr` test
  asserts on this message. The `reset_entry` Rust method
  only manipulates the local state.json file; it never
  contacts GitHub.

## Residual risks

- **Backoff is currently a fixed 300s in
  `StateStore::retry_or_fail`.** The contract says "applies
  the configured backoff"; the daemon's Config carries
  `retry_backoff_seconds = 300` and the StateStore's hardcoded
  value happens to match. A future amendment could thread the
  backoff through Config; the test surface accepts the
  constant today. The contract is satisfied because both the
  test and the production default agree at 300s.
- **The CLI's dry-run path reads via `StateStore::open` which
  validates the state file.** A corrupt `state.json` causes
  the dry-run to surface `CaduceusError::StateCorrupt` rather
  than printing "would reset …". This is the correct behaviour
  (the operator must address the corruption before any
  recovery operation) and matches the contract's "malformed
  state.json is never replaced with an empty state" rule.
- **`reset_entry` does not take the daemon lock itself.** The
  CLI is responsible for taking the daemon lock before calling
  `reset_entry`. This is intentional: the StateStore's
  `with_exclusive` lock only serialises state-store mutations;
  the daemon-wide lock is a separate concern owned by the
  caller (cron tick or operator CLI). The
  `reset_refuses_when_daemon_lock_held` test confirms the CLI
  layer enforces this.
