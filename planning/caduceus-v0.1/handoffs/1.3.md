## Public signatures/contracts used

- **`Config::load_with_paths(env, hermes, standalone)`** is the
  test-facing entry point. The three `Option<Path>` inputs pin each
  level of the documented chain independently so unit tests can drive
  every precedence case.
- **`Config::load_from(path)`** reads either a Hermes-shaped
  `config.yaml` (with a `caduceus:` section) or a standalone
  Caduceus YAML (whose top-level keys map to `RawConfig` directly).
  The parser detects the shape by looking for a top-level
  `caduceus:` mapping.
- **`Config::apply_dry_run_env(value)`** applies the value of
  `$CADUCEUS_DRY_RUN` to a parsed config. Valid values: `1` /
  `true` / `yes` (true), `0` / `false` / `no` (false); anything
  else is a configuration error.

## Resolution chain (per CONTRACTS.md "Configuration")

1. `$CADUCEUS_CONFIG` â€” an explicit, authoritative file path. A
   missing file is a hard error and never falls through.
2. `$HERMES_HOME/config.yaml` â€” read under `caduceus:`. Hermes files
   with no `caduceus:` section fall through to the standalone
   source if available; otherwise the missing section is an error.
3. `~/.config/caduceus/config.yaml` (or any path passed via
   `standalone`) â€” read directly.

`HERMES_HOME` must be an absolute path; relative paths and empty
strings are rejected with `Config` errors. The loader applies
`CADUCEUS_DRY_RUN` to the parsed config when present.

## State/schema effects

- `Config::load_with_paths` is read-only with respect to the host
  filesystem; it only reads files and reads `CADUCEUS_DRY_RUN` from
  the process environment. The env value is restored by every test
  via the `with_dry_run` helper.
- `parse_raw_from_text` separates Hermes- and standalone-shaped
  files. When a file carries Hermes-style top-level keys
  (`model`, `agent`, `providers`, `tools`, `memory`, `cron`,
  `platforms`, `gateway`, `secrets`, `voice`, `mcp`, `tts`) but no
  `caduceus:` section, the loader returns a clean
  `caduceus-section-missing` error rather than a confusing
  `deny_unknown_fields` failure.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- `Config::load_with_paths` never mutates the filesystem. It only
  reads the candidate files and the `CADUCEUS_DRY_RUN` env var.
- The env-scoping helper in `tests/config_resolution_test.rs`
  saves and restores `CADUCEUS_DRY_RUN` for every test that
  mutates it. The process-wide `ENV_LOCK` plus
  `serial_test::serial` ensures that parallel test threads never
  observe each other's state.
- Hermes-vs-standalone file detection prefers the documented
  contract over `serde`'s parse errors. The contract's
  `deny_unknown_fields` guarantee remains intact for malformed
  standalone configs.
- No network access; no subprocess invocation; no mutation of
  state.json or queue files.

## Residual risks

- `Config::load` itself is still a stub returning "Task 1.3 owns
  the env-aware resolution chain". Callers should use
  `Config::load_from` for now; the cron tick will use
  `load_with_paths` (or an env-aware variant) once Phase 7 wires
  it. `load_with_paths` already accepts the three arguments the
  daemon needs and is the test contract.
- The list of Hermes-shaped top-level keys is hard-coded. Future
  Hermes versions that introduce a new top-level key may start
  failing the strict Hermes-vs-standalone detection; the test
  suite should be updated when the contract evolves.
