## Public signatures/contracts used

- **`poll::IssueSummary { key, title, labels, ticket_type, updated_at }`**
  is the canonical typed shape for one labeled open issue. Fields
  are owned strings; `ticket_type` is `TicketType::Code` or
  `TicketType::Investigation` (re-exported from `queue`).
- **`poll::IssuePollDiagnostic`** enumerates the four non-`Summary`
  outcomes: `Ambiguous`, `PullRequest`, `Unmatched`, `Malformed`.
  Each carries enough context for the operator-facing diagnostic
  writer to surface the reason without re-fetching.
- **`poll::IssuePollOutcome { summaries, diagnostics }`** is the
  per-tick envelope. The poll helpers never mutate queue state —
  admission is the caller's responsibility.
- **`poll::poll_code(client, cfg, repos)`** and
  **`poll::poll_investigation(client, cfg, repos)`** are the public
  entry points. Both paginate per repo, classify rows, and return
  an `IssuePollOutcome`.
- **`poll::merge_outcomes(code, investigation) -> IssuePollOutcome`**
  folds the two streams. A key that appears in both with
  *different* `ticket_type` is recorded as `Ambiguous` and dropped
  from the summary list. A key that appears in both with the
  *same* `ticket_type` is deduplicated.
- **`poll::url_encode_label(label: &str) -> String`** percent-encodes
  ASCII letters/digits/`-._~` verbatim and the rest of the UTF-8
  byte sequence as `%XX`. This matches what `reqwest::Url::set_query`
  produces for the same input.
- **`poll::ISSUES_PER_PAGE = 100`** and the existing
  **`poll::MAX_PAGES_PER_ENDPOINT = 20`** pin the per-call
  pagination envelope.

## Behaviour

- **Per-repo pagination.** For every watched repo the daemon
  issues `GET /repos/{slug}/issues?state=open&labels={url_encoded_label}&per_page=100&sort=updated&direction=desc`,
  follows the `Link: <…>; rel="next"` header, and stops after
  20 pages (typed error beyond that). The label is the configured
  trigger label; the URL is percent-encoded so emoji-laden labels
  (the documented default `🤖 auto-fix`) round-trip through UTF-8.
- **PR exclusion.** Any object that carries a `pull_request` field
  is classified as `IssuePollDiagnostic::PullRequest` and never
  added to the summary list, per the contract.
- **Label re-verification.** Every accepted object is re-checked
  against the configured trigger label by scanning its `labels`
  array (case-insensitive). An object whose `labels` do not
  include the trigger label is classified as
  `IssuePollDiagnostic::Unmatched`. The contract says the query
  alone is not enough; this is the enforcement.
- **Ambiguous merge.** When the same key appears in both the code
  and the investigation poll with different ticket types,
  `merge_outcomes` records a single `Ambiguous` diagnostic and
  drops both halves from the summary list. Same-key / same-type
  duplicates are deduped.
- **Empty / null tolerance.** A row with `title: null`,
  `labels: null`, or `updated_at: null` is still classified
  correctly. Missing `title` becomes an empty string; missing
  `labels` becomes an empty array (which fails the
  re-verification and lands in `Unmatched`); missing `updated_at`
  defaults to `now()` so the queue writer always has a timestamp.
- **Malformed numbers.** A row with `number: 0` or no `number`
  field surfaces as `IssuePollDiagnostic::Malformed { key: None,
  reason }`; a `number: 0` is rejected by the issue-key
  validator; missing `number` is reported as a malformed row.
- **No Events API fields.** The decoder consumes only the
  documented `/issues` fields; any extra JSON keys (e.g. `event`,
  `performed_via_github_app`, `renamed_title`, `transferred`) are
  silently ignored via `serde::Deserialize`.
- **Cache reuse.** All GETs go through the Task 2.1 client and
  its `HttpCache`. A repeat poll with an unchanged issue set
  reuses the cached body via `If-None-Match` / 304.
- **Rate-limit and page cap.** A page whose
  `x-ratelimit-remaining: 0` header indicates exhaustion surfaces
  `CaduceusError::RateLimited` (the same path as Task 2.2).
  Exceeding `MAX_PAGES_PER_ENDPOINT = 20` per repo is a typed
  error, never silent truncation.

## State/schema effects

- No new on-disk files. The Task 2.1 cache gains entries for

## Tests added or changed

`tests/issue_poll_test.rs` — 15 tests, all green:

- **`url_encode_label` round-trip** — ASCII, emoji (UTF-8), and
  `/` (which must be percent-encoded so the URL has one path
  segment per label).
- **Code poll unique summary** — single labeled object is
  classified as `IssueSummary` with `ticket_type = Code`.
- **Investigation poll unique summary** — same for the
  investigation label.
- **PR exclusion** — a `pull_request` field produces a
  `PullRequest` diagnostic; the summary list contains only the
  issue object.
- **Unrelated label** — an object whose `labels` array does not
  include the trigger label is classified as `Unmatched`, not
  enqueued.
- **Ambiguous merge** — the same key in both code and
  investigation polls with both labels on the object becomes
  one `Ambiguous` diagnostic; the summary list is empty.
- **Empty / null tolerance** — `title: null`, `labels: null`,
  `updated_at: null`, and empty `labels: []` are all parsed and
  classified correctly. Missing `number` is `Malformed`.
- **Malformed number** — `number: 0` is rejected.
- **Pagination** — a `Link: <…>; rel="next"` URL with `page=2`
  is followed; both pages' objects appear in the summary list.
- **304 reuse** — the first poll gets 200 + ETag; the second poll
  sends `If-None-Match`; the server returns 304; the cached body
  is reused verbatim.
- **No Events API fields** — extra JSON keys (`event`,
  `performed_via_github_app`, `renamed_title`, `transferred`)
  are silently ignored.
- **Multiple repos** — the configured `watched_repos` list is
  polled in sequence; per-repo mocks exercise the loop.
- **`merge_outcomes`** — empty inputs yield empty outputs;
  same-ticket-type duplicates dedup; different ticket types
  produce an `Ambiguous` record.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 2 task packet:

- The poll is read-only with respect to the queue — admission is
  the caller's responsibility. Verified by every test that
  observes `outcome.summaries` and `outcome.diagnostics` only.
- The HTTP layer still routes through the Task 2.1 client, so
  token redaction, redirect policy, 10 MiB body cap, and the
  persistent cache all apply unchanged.
- The trigger label is re-verified against the object's `labels`
  array — the contract explicitly forbids trusting the query
  alone. The `Unmatched` diagnostic test exercises the negative
  case.
- PR objects are excluded by the documented `pull_request` field
  test, not by URL heuristics.
- Rate-limit and page-cap errors are typed and short-circuit the
  whole poll, so the meta layer (Task 2.4) can persist the
  observation before any other I/O happens.
- No subprocess execution, no filesystem mutation outside the
  Task 2.1 cache, no env mutation.

## Residual risks

- **Out-of-order trigger labels.** The merge walks the code
  summaries first, then the investigation summaries. A key that
  is "ambiguous" (both labels on the object) becomes one
  `Ambiguous` diagnostic with the *investigation* ticket's title
  and labels as the recorded fields. The contract does not pin
  which half wins, so the choice is documented here.
- **UTF-8 emoji in labels.** The `url_encode_label` helper
  percent-encodes the entire UTF-8 byte stream, so multi-byte
  characters take three `%XX` bytes each. This is correct but
  longer than GitHub's documented `application/x-www-form-urlencoded`
  minimisation; the length is bounded by the label's UTF-8 size
  and does not break the 10 MiB body cap.
- **Per-repo rate limit.** A repo with many pages can exhaust
  the per-hour quota mid-poll. The typed `RateLimited` error
  short-circuits the whole tick; subsequent repos are skipped
  and the daemon's next tick resumes at the persisted reset time
  (Task 2.4 wiring).
