## Staleness rule (per the contract)

1. The claim file is unreadable or its JSON does not parse → **quarantine**.
2. `version` mismatches the canonical `CLAIM_FILE_VERSION` → **quarantine**.
3. `started_at` is more than `FUTURE_TIMESTAMP_TOLERANCE_SECS` seconds in the future → **quarantine**.
4. The claim is recent enough that the age rule does not apply (age ≤ `stale_run_hours`) → **leave alone** (regardless of process identity).
5. The claim is older than `stale_run_hours` AND the recorded `pid` is no longer alive OR `process_start_identity` has changed (pid reuse) → **reap**:
   - tear down any attached worktree (`worktree::remove`, with the path-safety/idempotency rules from Task 4.3);
   - for an `InProgress` entry, return it to `Queued` **without** incrementing `attempts`;
   - for any other phase (`Queued`, `Previewed`, `Done`, `Failed`, `Skipped`) leave the phase alone — the claim is residue;
   - unlink the claim file.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No GitHub API calls** — the reaper is purely filesystem.
- **No network I/O.**
- **No claim path is constructed from raw user input** — the digest in the file name is the SHA-256 of the lowercased display key, exactly as the production claim writer does.
- **The `state.lock` flock is taken** for the queue mutation when an `InProgress` entry is reverted. A held `daemon.lock` is the caller's responsibility; the reaper runs at tick start under it.
- **Done/Failed/Skipped are never modified** — the reaper only returns `InProgress → Queued` and treats the other phases as residue (unlink only).
- **Corrupt claim files are never silently deleted** — they are moved into `<state_dir>/claims/corrupt/` with a `<!-- caduceus-reaper ... -->` provenance comment.

## Residual risks

- The worktree teardown path is best-effort: a `worktree::remove` failure is logged as a reaper warning and the reap continues. This means a stale `InProgress` claim with a broken worktree may take two ticks to fully clean up. The contract permits this — "teardown failure retained for retry" — and a `tracing::warn!` is the right level of observability.
- The reaper does not currently walk the `<state_dir>/runs/` directory to GC old run artefacts. The task packet does not pin that surface (Task 4.5 owns the worktree `gc`); the run-artifact retention clause in the packet's second paragraph refers to non-symlink run artefacts older than `run_retention_days`, which the current daemon does not produce — every run artefact is the run's `worker-result.json` (under the worktree path, not under `<state_dir>`), which the worktree teardown already removes. If a future task adds a `<state_dir>/runs/` directory, this is the right place to add the retention sweep.
