## Public signatures/contracts used

- **`StatusReport`** is the JSON-serialisable per-run summary surfaced
  by both the human and `--json` paths. Fields: `version`,
  `state_dir`, `last_tick_started`, `last_tick_finished`,
  `last_outcome`, `last_http_status`, `next_allowed_poll_at`,
  `phases: BTreeMap<String, u64>`, `next_head`, `next_head_earliest_eligibility`,
  `recent_errors: Vec<String>` (capped at 10), `rate_limit`, `live_workers`,
  `diagnostics`, `state_corrupt`. The `version` field carries the
  pinned `STATUS_SCHEMA_VERSION` so consumers can detect schema
  changes.
- **`LiveWorker`** carries `run_id`, `issue`, `pid`, `started_at`,
  `updated_at`, `transcript_path`, and `freshness: "fresh" | "stale"`.
- **`StatusDiagnostic`** — three variants. The CLI surfaces
  `NoState` as a "no state yet" hint, `CorruptState` and
  `CorruptQueue` as preserved-file hints that name the offending
  path and message.
- **`build_report(state_dir)`** is the read-only report builder.
  Short-circuits on missing state directory, routes `MetaStore`
  and `StateStore` errors through distinct diagnostics, and
  computes phase counts, FIFO head, recent errors, and live
  workers without writing anything.
- **`render_human(report, diagnostic)`** — line-oriented,
  README-stable output. Returns the diagnostic hint early when a
  diagnostic is set. The live-worker block surfaces
  `run_id`, `issue`, `pid`, `freshness`, `age`, and
  `transcript_path` so the operator can investigate.
- **`render_json(report)`** — pretty-printed JSON; the caller
  composes the top-level wrapper with the diagnostic string.
- **`report(state_dir, json)`** — the public entry point that
  `Command::Status` calls.
- **`compute_next_head(queue, now)`** — FIFO eligible head and
  earliest future eligibility; the `BTreeMap` iteration order
  guarantees the lexical first eligible wins.
- **`build_report_from_state(state_dir, meta, queue)`** —
  synthetic test seam that bypasses on-disk I/O.
- **`live_worker_from_heartbeat(record, now)`** and
  **`sample_heartbeat(run_id, issue, now)`** — heartbeat test
  seams.
- **`STATUS_SCHEMA_VERSION = "7.3.0"`** — pinned schema version.
- **`Heartbeat`**, **`HEARTBEAT_FILE_VERSION = 1`**,
  **`write_heartbeat_record(record, path)`**, and
  **`read_heartbeat_record(path)`** in `worker_supervisor.rs` —
  the versioned JSON envelope; `read_heartbeat_record` accepts
  both the versioned envelope and the legacy unversioned RFC 3339
  shape for backwards compatibility.

## State/schema effects

- **Heartbeat envelope (`.heartbeat` files):**
  ```json
  {
    "version": 1,
    "run_id": "<id>",
    "pid": <u32>,
    "started_at":  "<RFC3339 UTC>",
    "updated_at":  "<RFC3339 UTC>",
    "issue_key":   {"owner": "...", "repo": "...", "number": <u32>},
    "transcript_path": "<absolute path>"
  }
  ```
  The supervisor rewrites the file atomically (`O_CREAT | O_TRUNC | O_NOFOLLOW` then `sync_all` then `rename`, mode `0600`). Older unversioned RFC 3339 timestamp files continue to be readable via the legacy fallback in `read_heartbeat_record`.
- **Status schema version (`7.3.0`)** — bumped because
  `state_corrupt`, `diagnostics`, and `next_head_earliest_eligibility`
  were not part of the previous report.
- **No new on-disk formats beyond the heartbeat envelope.** The
  reporter composes the existing `state.json`, `state_meta.json`,
  and `runs/*.heartbeat` surfaces.

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 07 gate:

- **#1 (whole-tick lock)** — the reporter is read-only. It
  does not take the daemon lock or the state flock. It is
  always safe to run alongside an active tick.
- **#2 (atomic write with fsync)** — `write_heartbeat_record`
  writes to a temp file, `sync_all`s, then renames into place.
  The rename failure path cleans up the `.tmp`. Mode is `0600`
  on every path.
- **#3 (queue helpers never construct claim paths from raw
  strings)** — `status` does not touch claim files; only
  `runs/*.heartbeat` files are read.
- **#7 (Rust owns heartbeats)** — the supervisor writes the
  structured envelope; the reporter only reads. The reporter
  rejects symlinked heartbeats before parsing them, so a
  redirected heartbeat cannot influence the live worker list.
- **#8 (no credential leakage)** — `status` does not read
  tokens, does not log env, and does not accept config values
  beyond `state_dir` and `worker_command` (the latter is
  ignored by `status`).
- **No PAT in arguments, URLs, or env** — the CLI loads the
  same `Config::load` / `Config::load_from` chain as the
  canonical tick; no GitHub call is made.
- **Preserve corrupt files (Phase 1 rule)** — `CorruptState`
  and `CorruptQueue` diagnostics preserve the underlying file
  by surfacing the error from `MetaStore::open` /
  `StateStore::open` without mutating either file. The
  acceptance tests `corrupt_state_diagnostic_preserves_file`
  and `corrupt_queue_diagnostic_preserves_file` pin this
  behaviour.
- **No network-dependent tests** — every acceptance test uses
  a local `tempdir`, a synthetic `QueueState`, or a
  deterministic `Heartbeat` fixture. No HTTP, no `wiremock`,
  no fixture-loading from disk beyond the state directory
  itself.

## Residual risks

- **Symlink rejection ordering** — the reporter reads
  `fs::symlink_metadata` before `is_file` so a symlink to a
  regular file is rejected on `is_symlink` and never reaches
  the open path. A symlink that points to a deleted target
  also gets rejected (the symlink itself is what matters, not
  the target).
- **Backwards compatibility with legacy heartbeats** — the
  fallback path synthesises an `IssueKey` with empty
  `owner/repo/number` from the legacy RFC 3339 timestamp. The
  reporter surfaces the run id from `file_stem`, but the issue
  field will be empty until the worker rewrites the file with
  the structured envelope. Today's task only requires the
  reporter to *not crash* on the legacy format; issue
  attribution for legacy files is a follow-up.
- **`compute_next_head` correctness** — the FIFO order relies
  on `BTreeMap` iteration, which is canonical across daemon
  versions. The acceptance test pins the order with three
  inserts in mixed insertion order.
- **`status_corrupt` and `state_dir` paths in JSON** — the
  JSON payload embeds the `PathBuf` as a string; consumers
  must tolerate either the JSON-encoded `PathBuf` (string)
  shape or the camelCase `state_dir` shape depending on the
  platform's `serde_json` PathBuf encoding.
