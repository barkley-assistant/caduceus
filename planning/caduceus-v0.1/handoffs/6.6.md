## Public signatures/contracts used

- **`validate_public_text(text, cfg, limit_bytes) -> Result<(), VoiceError>`** is the canonical public-voice check. Substring match runs *first*; length cap runs *after*. The first matching term's lowercase form is captured in `VoiceError::Forbidden { found }` so the operator can update the allowlist.
- **`validate_pr_title` / `validate_pr_body` / `validate_comment`** are the documented channel wrappers. Each uses the documented byte limit (256 / 65 536 / 65 536).
- **`check_voice_or_error(text, cfg, channel) -> CaduceusResult<()>`** is the single chokepoint for outbound public-text mutations. The function maps a `VoiceError` rejection into a `CaduceusError::Other` so the structured logger and the retry-or-fail logic can branch on the message.
- **`VoiceChannel`** is a closed enum (`Comment`, `PrTitle`, `PrBody`, `Custom(usize)`). The `Custom` variant exists for tests and rare helpers that need a custom bound.
- **`first_forbidden_term(text, forbidden) -> Option<String>`** is the canonical lookup helper. It returns the matched term in lowercase, skipping empty entries.
- **`terminal_from_voice(err) -> CaduceusError`** maps a `VoiceError` into the unified error path.
- **`DEFAULT_COMMENT_MAX_BYTES` / `DEFAULT_PR_BODY_MAX_BYTES` / `DEFAULT_PR_TITLE_MAX_BYTES`** are public constants. `github.rs` re-exports them.

## Behaviour

- **Substring matching.** Each configured forbidden term is
  matched against `text.to_lowercase()` with `String::contains`. A
  match in any case (`Foo`, `FOO`, `foo`) is rejected. Empty
  entries are skipped (the config validator rejects them
  outright).
- **Length cap.** `text.len() > limit_bytes` is rejected with
  `VoiceError::TooLong { limit: limit_bytes }`. The cap runs *after*
  the substring check so a long body that also contains a
  forbidden term surfaces `Forbidden` (the more actionable
  reason for the operator).
- **Channel routing.** Every HTTP helper in `github.rs` calls
  `check_voice_or_error` before any (placeholder) HTTP layer is
  touched. A rejected request returns `CaduceusError::Other`; the
  validator runs even on the success path so legitimate text is
  always checked.
- **Logging.** `log_safe_term_match` exists so a future tightening
  of the policy has a single point to update. Today it returns
  the term unchanged because the contract treats every
  configured term as operator-supplied; nothing secret leaks.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 01 gate:

- The validator is pure. No I/O, no subprocess, no global state
  mutation. The mapping from `VoiceError` to `CaduceusError` is
  also pure.
- `log_safe_term_match` is the only function that emits any
  identifier to the structured logger. Today it returns the term
  unchanged; a future tightening of the policy will modify the
  single function body without touching callers.
- The HTTP helpers never reach the placeholder HTTP layer when
  the validator rejects — the test suite confirms this for the
  three mutation channels (comment, PR, investigation).

## Residual risks

- The defaults for the byte limits (256 / 65 536) are hard-coded
  in `finalize.rs`. Operators that need a tighter limit can pass
  an explicit `limit_bytes` to `validate_public_text`; the
  channel helpers remain pinned to the documented defaults.
- The placeholder HTTP helpers in `github.rs` return `Ok` after
  the validator passes. Phase 6 will wire the real HTTP layer
  through `reqwest`; the validator gating is already in place.
