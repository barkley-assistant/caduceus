# Handoff 4.4 — Generate the canonical prompt file

- Work item: Task 4.4 (Phase 05 — Worker context and supervision)
- Outcome: complete
- Date: 2026-07-14

## What shipped

A pure `build_prompt(issue, ticket_type, context_json, branch_name) -> CaduceusResult<String>`
and an atomic `write_prompt(worktree, text) -> CaduceusResult<PathBuf>` in
`src/prompt.rs`. The prompt is delivered to the worker bridge as
`<worktree>/worker-prompt.md` and contains, in order:

1. **Run metadata** — issue identity, ticket type, branch name.
2. **Hard constraints** — no `git commit`, `git push`, `git checkout`,
   `git switch`, `git branch -m`, `git reset --hard`; no editing of
   `.git/`, `worker-prompt.md`, `worker-result.json`, or dry-run
   artefacts.
3. **Branch directive** — the daemon has already created
   `<branch_name>`; do not rename or check out a different branch.
4. **Forbidden paths** — explicit list of paths the worker must not
   modify.
5. **GitHub access reminder** — the worker cannot reach GitHub; the
   daemon will perform the finalization, push, and PR/comment work.
6. **Behavior** — short prose for `Code` ("make the smallest correct
   change, write a specific summary") vs. `Investigation`
   ("do **not** change code; write findings to `worker-result.json`").
7. **Output schema** — exact `worker-result.json` shape with field
   constraints documented inlined (`summary` ≤ 64 KiB Markdown,
   `commit_message` ≤ 256 chars, `pull_request_title` ≤ 256 chars
   one-line, `artifacts` map ≤ 128-char keys).
8. **Issue** — title, owner/repo, number, labels, body fenced under
   ```` ```text ```` with adversarial-Markdown fence sanitisation.
9. **Context** — the verbatim `CADUCEUS_CONTEXT_JSON` document fenced
   under ```` ```json ````, also sanitised.
10. **Footer** — explicit "if truncated, refuse" instruction.

## Public surface

| Symbol | Signature | Purpose |
|---|---|---|
| `MAX_PROMPT_BYTES` | `const usize = 2 MiB` | Hard cap on the encoded prompt. |
| `PROMPT_FILENAME` | `const &str = "worker-prompt.md"` | The single file the worker reads. |
| `build_prompt` | `(&IssueDetail, TicketType, &str context_json, &str branch_name) -> CaduceusResult<String>` | Pure prompt builder. |
| `write_prompt` | `(&Path worktree, &str text) -> CaduceusResult<PathBuf>` | Atomic same-directory temp-file write. |

The file write opens `O_NOFOLLOW`, truncates, writes, `fsync`s, then
renames into place from a temp file in the same directory. Mode 0600.

## Forbidden side-effects

- **No GitHub access from the worker.** The prompt's prohibition is
  explicit and tested (`prompt_does_not_call_gh_or_call_github`).
- **No `.git/` modifications.** The forbidden paths list is asserted
  in the prompt body and is the worker's contract.
- **No network calls.** The worker is told that all finalization
  happens on the daemon side.
- **No hidden credentials.** The prompt content carries only the issue
  body, comments, context JSON, and branch name.
- **Oversized inputs are rejected, not truncated.** If the builder
  exceeds 2 MiB it returns `CaduceusError::Worker { context: "prompt:oversized", … }`
  rather than producing a partial write.

## Cross-cutting effects

- The supervisor in `src/main.rs` and the worker harness in
  `src/worker_supervisor.rs` (Task 5.1) both speak the protocol of
  "deliver the prompt as a file in the worktree". Task 4.4 is the
  half that builds and writes that file; the supervisor uses
  `write_prompt` and `build_prompt` to materialise the contract.
- The `worker-result.json` shape documented in the prompt matches
  `WorkerResult` in `src/worker.rs` (Task 3.x). No new fields were
  added.

## Residual risks

- The Markdown fence sanitiser replaces triple-backtick runs with
  a `~~~~~~\n` line. We do not specifically recognise GitHub's
  indented code-block syntax or attribute blocks; adversarial bodies
  of those forms survive unchanged but are still bound by the
  maximum prompt size.
- The output schema documented in the prompt is the *worker's*
  contract for the JSON it writes. The daemon's
  `parse_worker_result` (Task 6.x) is responsible for enforcing
  field constraints on parse; Task 4.4 only documents them.

Not blocked.
