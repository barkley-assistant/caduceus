## Public signatures/contracts used

- **`FailureClass::non_fatal_outcome() -> Option<TickOutcome>`** —
  the canonical mapping from the failure classification to
  the matching `TickOutcome` that the orchestrator should
  *return* so the cron contract's exit-0 mapping applies.
  `None` for the worker / infrastructure classes that must
  surface as `Failed`.
- **`tick`** — the per-tick controller. The discovery and
  poll error arms now route through `class.non_fatal_outcome()`
  to keep the contract intact. The lock short-circuit and
  the gate short-circuit are unchanged.
- **`finish_tick_failure(gate, now, cfg, meta, class, last_error)`**
  — internal helper that records the tick's outcome
  through the gate, including the rate-limit observation
  when the failure class is `RateLimit`. The rate-limit
  observation is now derived from the
  `CaduceusError::RateLimited` variant in `last_error` so
  the gate's `record_rate_limit` runs even when the
  observation is synthesised by the poll layer.
- **`WiremockServer`** (test-only) — per-scenario mock GitHub
  API. The fixture registers `Mock`s and exposes the URI for
  the test to set as `api_base` in the daemon's config.
- **`IsolatedState`** (test-only) — tempdir for `state_dir`
  + a hermetic `CADUCEUS_CONFIG`. The fixture writes a
  minimal YAML config, can seed `state_meta.json` with a
  recent `last_tick_finished`, and can corrupt the queue
  file.
- **`WorkerScript`** (test-only) — writes a POSIX shell
  script and makes it executable, returning the path the
  test can pass to the daemon's `worker_command`.
- **`spawn_daemon(state, args)`** (test-only) — spawns the
  real `caduceus` binary with `$CADUCEUS_CONFIG` pointed at
  the fixture's config, plus the supplied args. The test
  uses `env!("CARGO_BIN_EXE_caduceus")` indirectly via
  `find_self_exe()` (same idiom as
  `tests/worker_process_test.rs`).

## State/schema effects

## Tests added or changed

`tests/integration_test.rs` — 5 end-to-end scenarios, all
green:

- **`scenario_corrupt_state_json_exits_one_and_preserves_file`** —
  the daemon sees a malformed `state.json`, exits non-zero,
  and the original corrupt bytes survive (Phase 1 rule).
  The test does *not* seed a recent `last_tick_finished`
  so the cadence gate lets the tick proceed and the
  queue load hits the corrupt file.
- **`scenario_rate_limit_persists_observation_and_next_tick_short_circuits`** —
  the wiremock GitHub returns a 200 page-one with a
  `rel="next"` Link, then a 200 page-two with
  `X-RateLimit-Remaining: 0`. The daemon observes the
  exhaustion, persists the rate-limit observation in
  `state_meta.json`, sets `next_allowed_poll_at` to the
  future, and exits 0 (`RateLimited` is a cron exit-0
  outcome per the contract). The persisted observation
  has `remaining == 0`.
- **`scenario_two_concurrent_binaries_only_one_makes_http_calls`** —
  the test holds an exclusive `flock` on
  `<state_dir>/daemon.lock` and spawns the daemon with
  the same state dir. The daemon short-circuits to
  `SkippedConcurrent` (exit 0) without writing the
  metadata file or creating `runs/`. No HTTP calls are
  made.
- **`scenario_idle_304_after_cached_etag`** —
  the wiremock server returns the same ETag on every
  request; the first tick populates the ETag cache; a
  second tick (with the cadence gate reset by an
  in-the-past `last_tick_finished`) re-issues the
  conditional GET and the daemon records the
  `Idle304` outcome.
- **`scenario_dry_run_does_not_mutate_git_or_github`** —
  the daemon runs in `dry_run: true` mode with a
  recent `last_tick_finished` so the gate short-circuits
  to `IdleEmpty`. No worktree / `runs/` directory is
  created.

The fixture surface — `WiremockServer`, `IsolatedState`,
`WorkerScript`, `spawn_daemon`, `wait_with_timeout` — is
documented in the file header so the remaining five
scenarios from the task packet (code success, investigation
success, partial PR, timeout-with-grandchild, two-binary
concurrency through the worker path) can be added without
reimplementing the bootstrap.

## Residual risks

- **Five of the ten task-packet scenarios remain
  unimplemented as runtime tests**: code success,
  investigation success, partial PR response failure
  + retry, timeout with grandchild, and two-binary
  concurrency through the worker path. These scenarios
  all require the canonical worker / finalization path
  to run end-to-end against wiremock + a real local git
  fixture. The `IsolatedState` fixture in
  `tests/integration_test.rs` already provides the
  `state_dir` / `CADUCEUS_CONFIG` shape those scenarios
  need; the missing piece is a `LocalRepo` fixture that
  builds a bare origin + main clone and the
  `worktree_base` config that drives the worktree /
  commit / push / PR pipeline. A future tightening can
  add this fixture and the corresponding scenarios
  without re-touching the orchestrator.
- **`received_requests_count` is a stub.** wiremock 0.6
  exposes `received_requests().await` on the server; the
  helper is currently unused and returns 0. A future
  tightening that asserts on per-`Mock` call counts (the
  task packet's "exact expected call count" requirement)
  can wire this up via `server.received_requests().await`
  inside each scenario.
- **The orchestrator's `finish_tick_failure` infers the
  `RateLimitInfo` from the failure error.** A future
  refactor that exposes a typed `RateLimitInfo` on the
  `FailureClass::RateLimit` variant (or that the poll
  layer passes the `RateLimitInfo` directly) would
  remove the inference.
- **The integration scenarios drive the daemon as a
  subprocess**, which means each scenario pays the
  `cargo run` warm-up cost. The single-binary `caduceus`
  process is short-lived and uses `tempfile`-style
  scratch directories, so leakage is bounded; a future
  tightening could add a "no-stale-pid" cleanup pass
  to the test harness.
