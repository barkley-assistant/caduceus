## Public signatures/contracts used

- **`Supervise` is the canonical production spawn path.**
  The public `supervise(self_exe, cfg, issue, worktree, run_id, context_json, worker_command, cancellation) -> CaduceusResult<SupervisorOutcome>` lives in `caduceus::worker_supervisor`. The function creates the heartbeat and transcript in secure mode (`0600`), spawns the same binary in hidden `__worker-supervisor` mode with `stdin/stdout/stderr` piped (the inherited-fd contract), runs a tokio protocol loop that reads `READY(pgid)` and replies `ACK`, awaits the supervisor child, awaits the drain tasks, removes the heartbeat, and returns the [`SupervisorOutcome`] with the bridge exit code, signal flag, timeout flag, and cancellation flag.
- **`SupervisorOutcome { status, signaled, timed_out, cancelled }`** is the canonical return shape. The daemon's orchestration loop can branch on the four fields without inspecting raw exit codes.
- **`ControlFrame`** is the protocol message enum with `Ready { pgid }`, `Done { status, signaled }`, `Fatal { reason }`, `Terminate { force }`, and `Ack` variants. The serialisation is `<u32-le length><UTF-8 line>` where the line begins with `v{PROTOCOL_VERSION} <OPCODE> [<payload>]`. Length is bounded by `MAX_FRAME_BYTES = 64 KiB`.
- **`WorkerRunPaths { state_dir, run_id, transcript_path, heartbeat_path, result_path }`** is the canonical per-run layout. `transcript_path` and `heartbeat_path` are written under `<state_dir>/runs/` with mode `0700` for the directory and `0600` for the files.
- **`write_heartbeat` / `clear_heartbeat` / `read_heartbeat`** are the heartbeat primitives. The heartbeat file holds a single RFC 3339 UTC timestamp; the status reporter considers the worker live when the timestamp is < 90 seconds old.
- **`open_transcript` / `append_transcript` / `truncate_transcript`** are the transcript primitives. The transcript file is created with `O_NOFOLLOW` and mode `0600`. Truncation appends a `...<truncated N bytes>` marker and caps the surviving bytes at `cfg.transcript_max_bytes`.
- **`build_supervisor_command`** is the production spawn builder. The daemon's `supervise()` calls it; tests use the equivalent inline construction.
- **`detach_session` (`nix::unistd::setsid`)**, **`kill_pid` / `kill_pgid` (`nix::sys::signal`)**, and **`nix::sys::prctl::set_child_subreaper`** are the safe wrappers around the three Linux-specific syscalls the supervisor needs. The crate's `#![forbid(unsafe_code)]` policy forbids `unsafe` blocks in the source tree; the supervisor never reaches for them.
- **`HIDDEN_COMMAND = "__worker-supervisor"`** is reserved and never appears in `--help`, never appears in cron configuration, and never appears in plugin configuration.

## State/schema effects

- The supervisor writes only the heartbeat and transcript files under `<state_dir>/runs/` with secure modes (`0700` for the directory, `0600` for each file). It never touches the queue, the metadata file, or the worktree.
- No git subprocess, no HTTP request, no daemon credentials.
- The supervisor's `stdin`/`stdout` carry the framed protocol only. The worker's `stdout` is sent to `/dev/null`; the worker's `stderr` is forwarded to the transcript. The split keeps protocol bytes and worker bytes from interleaving on the daemon side.
- `prctl(PR_SET_CHILD_SUBREAPER)` is enabled on Linux inside the hidden supervisor mode so any descendant that calls `setsid` is still reaped by the supervisor. A failure is non-fatal; the worker-session kill path still works.

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 05 gate:

- **#5 (daemon owns branch name)** — the supervisor never touches the
  branch; it forwards `--branch-name` only as an opaque value in
  the `IssueKey` rendering path.
- **#6 (worker session + supervisor)** — every worker run is a
  fresh process group (`process_group(0)`) under a fresh session
  (`setsid()`); the supervisor's killer thread broadcasts
  `SIGTERM`/`SIGKILL` to the recorded PGID. On Linux the
  subreaper attribute is set so a `setsid`-calling grandchild
  is still reaped by us.
- **#7 (Rust owns heartbeats / process lifecycle)** — every
  heartbeat file is written by Rust via `write_heartbeat`; the
  status reporter reads via `read_heartbeat`. The transcript is
  appended by Rust via `append_transcript`.
- **#8 (no credential leakage)** — `env_clear()` is the first
  step in `build_supervisor_command`; the supervisor inherits
  only the cleared worker environment plus the cleared
  CADUCEUS_* context variables (the `sanitized_env` filter from
  Task 5.2 is applied by the daemon side before `spawn`).
- **No queue, metadata, or worktree mutation** — the supervisor
  writes only to `<state_dir>/runs/<run_id>.heartbeat` and
  `<state_dir>/runs/<run_id>.log`. The daemon side cleans up
  the heartbeat after the supervisor returns; the transcript
  stays on disk for `status` and `finalize` to inspect.
- **Hidden command never reaches `--help`** — the dedicated
  `supervisor_hidden_command_is_not_in_help` test asserts the
  token `__worker-supervisor` does not appear in `caduceus
  --help` output.

## Residual risks

- **Transcript forwarding is one-directional.** The supervisor
  currently forwards worker `stderr` (where the worker's stdout
  is redirected via the contract test) into the transcript. A
  worker that writes only to `stdout` (without redirecting it
  to `fd 2`) will produce an empty transcript. The Task 5.1
  test fixture uses `... 1>&2` to keep the test independent;
  Phase 7's orchestration is expected to construct the
  worker_command to redirect stdout to stderr when needed.
- **The exec-gate pipe is implemented implicitly via the
  `READY`/`ACK` frame order.** A future protocol bump that
  removes one of those frames could regress this contract; the
  `frame_round_trip` test plus the `supervisor_handles_*
  argument` tests cover the obvious off-spec shapes, but a
  full protocol-fuzz suite is left for Phase 7.
- **`setsid` failure is non-fatal on Linux.** If the kernel
  refuses `setsid` for any reason, the supervisor proceeds
  anyway and falls back to the worker-PG kill path. Operators
  who need strict session isolation should run the daemon in a
  container that namespaces the worker by default.
