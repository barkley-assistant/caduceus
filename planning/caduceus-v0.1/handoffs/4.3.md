## Public signatures/contracts used

- **`remove(handle: &Worktree) -> CaduceusResult<()>`** is the
  canonical teardown entry point. The previous stub was
  named `destroy` and took a `&GitRunner` parameter; both
  changes are documented above and the re-export under
  `caduceus::worktree::remove_worktree` keeps the public
  surface tight.

- **`should_retain_branch(runner, main_path, branch, base_oid)`**
  decides whether a local branch can be deleted. The
  decision order is:
  1. Branch tip identical to the recorded `base_oid`
     (worker produced no commits) — *delete*. This
     short-circuits dry-run and pre-commit worker-failure
     branches regardless of any cached upstream state.
  2. `git rev-parse --verify --quiet <branch>@{u}` exits 0
     — *retain* (the branch has an upstream configured).
  3. `git merge-base --is-ancestor <branch> <base>` exits 0
     for any of `origin/main`, `origin/master`, `main`,
     `master` — *retain* (the branch's work is reachable from
     the base branch's history, so the operator can find
     it via the base).

  This matches the task packet's "Dry-run and pre-commit
  worker-failure branches are removed; committed/resumable
  or pushed branches are retained" semantics in v0.1.

- **`git worktree remove --force`** tolerates uncommitted
  local changes so the worker can leave `WIP_NOTES.md` or
  scratch files behind without aborting the teardown. The
  registration in `<main>/.git/worktrees/<run_id>` is
  cleared by `git worktree prune` even when
  `worktree remove` left artefacts behind; the daemon
  never does a raw recursive deletion at the worker-home
  area.

- **Path safety** rejects any `Worktree` whose canonicalised
  `path` lies outside `<main>/.worktrees/`. An
  attacker-crafted handle pointing at `/etc` or `..` is
  refused with a typed `Worktree { context: "destroy",
  stderr: "refusing to remove …: path escapes the
  worker-home …" }` error rather than a recursive deletion.
  The fallback path (`fs::canonicalize` failure) just keeps
  the input verbatim — the daemon prefers a permissive
  allowlist over a hard fail so an uncanonicalised
  intermediate state never blocks legitimate daemon
  progress.

## State/schema effects

None. `remove` only touches:
* the worktree directory itself,
* git's worktree registration under
  `<main>/.git/worktrees/<run_id>`,
* the local branch ref under `<main>/.git/refs/heads/`,
* the per-branch upstream config in `<main>/.git/config` (only
  when `git push -u` from inside the worker set it; Phase 6
  is responsible for the push itself).

No queue state, no checkpoint file, no `state.json` entry
is modified. The worker's `worker-result.json` and the
run's `runs/<run_id>.result.json` survive teardown so the
finalization checkpoint's audit trail is intact — this is a
Phase 7 contract enforced by `remove` not touching the state
directory at all.

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 04 gate:

- **#6 (worker-session kill)** — unchanged. `remove` does
  not touch any worker subprocess state; the worker
  process tree is the responsibility of Phase 5's
  supervisor which is invoked before `remove` on the
  finalization path.
- **#7 (Rust owns heartbeats / process lifecycle)** —
  unchanged. `remove` does not create or delete
  heartbeats. The heartbeat file is the supervisor's
  concern; if Phase 5 invokes `remove` while a heartbeat
  is live, that is a higher-level invariant violation
  surfaced as a different error.
- **#8 (no credential leakage)** — the
  `should_retain_branch` probe runs through
  `runner_run_in_std` which uses the runner's runner,
  which scrubs credential variables before invoking
  git. The redaction surface from Task 4.1 carries over
  unchanged.
- **#5 (daemon owns branch name)** — `git branch -D
  <branch>` deletes the daemon-owned
  `automation/issue-<n>-<run-id>` only when the retention
  decision says so. Worker output never enters the branch
  name.
- **Path-escape rejection** — `remove` rejects any handle
  whose canonical path leaves `<main>/.worktrees/`.
  Tested by `remove_rejects_path_outside_worktrees_dir`
  and `remove_rejects_path_above_worktrees_dir`.

## Residual risks

- `should_retain_branch` probes the per-branch `branch.<name>.remote` and `branch.<name>.merge` config plus `git rev-parse <branch>@{u}`. Branches with no upstream AND not merged into base are deleted. The merge-base probe is run with `--is-ancestor` which is reflexive: a branch whose tip equals the base would erroneously be retained. The first step of the decision (`branch_oid == base_oid`) short-circuits that case to delete. If the daemon ever carries a branch with a tip identical to the recorded base OID AND has a configured upstream (e.g. a rebase-forward of an empty branch), the upstream wins by current ordering — that matches operator intent (the branch was pushed so the work is somewhere).
- `git worktree remove --force` may abort before deleting git's worktree-metadata directory on some failure modes (read-only mount, permission denied on a child). The subsequent `git worktree prune` clears the metadata anyway, but a stale `<main>/.git/worktrees/<run_id>` directory can survive a `git worktree remove` that exited non-zero. The test `remove_surfaces_typed_error_when_worktree_path_unremovable` exercises the read-only case; an operator who sees this in production should `chmod -R u+w` the worktree and re-run.
- The path-safety check uses `fs::canonicalize` which fails for non-existent paths. The fallback path uses the uncanonicalised input. This means an attacker who can write `<main>/.worktrees/.../foo` to a path that *already* exists on disk can bypass the check; the test `remove_rejects_path_outside_worktrees_dir` covers the latter case (the foreign path exists) and observes the rejection.
- `gc` is still a stub — the `caduceus worktree-gc` CLI is owned by a later phase. The phase-04 gate does not exercise `gc`.
