## Behaviour (per the contract)

1. **Deterministic key order.** `result.artifacts` is a `BTreeMap`, so iteration order is always lexicographic on key. Re-rendering the same `result` produces the same body.
2. **Dynamic fence length.** `dynamic_fence_length` scans the JSON for the longest backtick run and returns `max(3, longest + 1)`. An artifact value containing `````` `````` (7 backticks) is rendered inside an 8-backtick fence so the body cannot accidentally close the JSON block.
3. **Total-render cap.** `MAX_RENDERED_BODY_BYTES = 64 * 1024` is enforced inside `build_pr_body`. A summary larger than the cap is truncated; the marker is re-appended so the result is always capped **and** carries the idempotency marker.
4. **Control-character escape.** `escape_control_chars` keeps `\n` and `\t` (safe in Markdown) and replaces every other control char (`\0`, `\x01`â€“`\x1f`, `\x7f`) with `\u00XX`. The `render_artifacts` block emits the JSON via `serde_json::to_string_pretty`, which itself produces only safe bytes for the JSON syntax â€” the escape helper is the additional belt-and-braces guard for user-supplied artifact values that may contain raw control characters.
5. **Issue-closing reference.** The body always contains `Closes #<issue.number>`. The placement is between the summary and the artifact section.
6. **Idempotency marker.** The body always ends with `<!-- caduceus-pr-body:run=<run_id> -->`. The marker is a hidden HTML comment so the rendered Markdown is unchanged; the run_id is embedded verbatim so a re-render of the same body produces the same bytes.
7. **Public-voice validation.** `build_pr_body` calls `validate_pr_body` (which runs the Task 6.6 forbidden-term + length-cap check) on the rendered body; `build_pr_title` calls `validate_pr_title`. A rejected body or title returns `CaduceusError::Other(public-voice: ...)` via `terminal_from_voice`.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No GitHub calls.** The render path is pure: it does not call the GitHub API, the worker, or the queue.
- **No global state mutation.** The render path owns a `String` and a `serde_json::Value`; nothing is written to disk.
- **No forbidden-term acceptance.** Every body and title passes through the public-voice rule before return. The integration test `forbidden_string_in_summary_is_rejected` / `forbidden_string_in_artifact_value_is_rejected` / `forbidden_string_in_title_is_rejected` confirms the three rejection points.

## Residual risks

- The render cap (64 KiB) is below the validator cap (65 536 bytes) by design. A 64 KiB body with the marker may exceed 64 KiB by ~40 bytes (marker length); the cap logic in `build_pr_body` accounts for this by reserving space for the marker. The 1 KiB slack also leaves room for a future `<!-- â€¦ -->` payload.
- The artifact caption `Artifacts (<N>):` is in English; i18n is out of scope for v0.1. The constants are public so a future i18n patch can swap them.
- The marker is an HTML comment; if a future change renders the body through a Markdown linter that strips comments, the marker disappears. The comment form is the GitHub-recommended way to embed invisible metadata and survives a typical PR view.
- A adversarial summary containing `<!-- caduceus-pr-body:run=fake --> </ul>` could try to poison the marker. The body's `find(IDEMPOTENCY_MARKER_PREFIX)` and `truncate` logic is greedy and will keep the *first* marker it sees; the legitimate one written at the end is preserved. The 65 536-byte validator catches any body that grew because of the adversarial content.
