## Cutover summary

Per the task packet's cutover guidance and `MIGRATION.md` §6:

1. The legacy processor's cron entry is operator-disabled before
   the first `caduceus migrate-state`. The daemon's
   `daemon.lock` is process-local — it does not block the legacy
   processor — so explicit disablement is mandatory.
2. The migration lands via `caduceus migrate-state --from
   <legacy.json>` (dry-run first per `MIGRATION.md` §1) and the
   `state.json` lives at the resolved `state_dir`.
3. Caduceus's cron entry is enabled via
   `hermes caduceus cron-install` (the no-agent, every-2m pulse
   wrapper). Caduceus now polls against one disposable
   repository first; that is the cutover constraint the task
   packet pins.
4. Status, retry, rate-limit, and rollback checks are the
   `caduceus status` output plus the persistence behaviour
   verified by the integration suite's rate-limit and corrupt-
   state scenarios.
5. Repository scope expands only after the operator confirms
   that status / retry / rate-limit / rollback behave correctly
   on the disposable repository.

The task packet forbids running Caduceus alongside the legacy
processor against the same labels. The cron-disable /
migrate-state / cron-install sequence above is the canonical
cutover.

  `NoState` and `CorruptState` diagnostics to exit codes 2 and
  1** as the contract specifies. The dispatch in `src/cli.rs`

  `StatusDiagnostic::NoState` to exit 2 and the corrupt

  and inspect the `diagnostic` field in the meantime.
