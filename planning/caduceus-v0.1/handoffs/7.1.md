## Public signatures/contracts used

- **`run() -> CaduceusResult<u8>`** is the cron / no-argument
  entry point. Loads config from the canonical resolver
  chain, initialises the structured log stream, runs the
  controller under a fresh `CancellationToken`, and maps the
  resulting `TickOutcome` to the documented exit code:
  - `Processed` / `Idle304` / `IdleEmpty` /
    `SkippedConcurrent` / `SkippedCadence` / `RateLimited` /
    `Cancelled` → exit 0
  - `Failed` → exit 1

- **`run_blocking(cfg: Config) -> CaduceusResult<TickOutcome>`**
  is the runtime-owning variant. Used by the CLI host and
  the test suite. Builds a `current_thread` runtime, builds
  a fresh `CancellationToken`, and blocks on
  `run_with_config`.

- **`run_with_config(cfg: Config, cancellation: CancellationToken) -> CaduceusResult<TickOutcome>`**
  is the production async entry point. Builds the production
  `Services` bundle (`SystemClock` clock, `Client` GitHub
  client, `GitRunner` git runner, `ProcessSupervisorAdapter`
  process supervisor) and dispatches to `tick`.

- **`tick(cfg: Config, services: Services, cancellation: CancellationToken) -> CaduceusResult<TickOutcome>`**
  is the testable core. Takes ownership of the
  `Config` and a `Services` bundle so the integration
  tests can swap fakes. Follows the contractually-documented
  pipeline exactly:

  1. Try the whole-tick `DaemonLock`. On contention, return
     `TickOutcome::SkippedConcurrent` without writing.
  2. Open `MetaStore` and `CadenceGate`; enforce the
     rate-limit and cadence gates; persist
     `last_tick_started`. The gate precheck returns
     `SkippedCadence` or `RateLimited` (or `Proceed`) and
     records the corresponding `last_tick_finished` /
     `next_allowed_poll_at`.
  3. Reap stale claims / abandoned worktrees via
     `queue::reap_stale_claims(state_dir, now, stale_run_hours)`.
  4. Build the `Arc<Client>` from the services bundle,
     discover watched repos, poll for the two trigger
     labels, and enqueue summaries. The poll response
     status feeds the `Idle304` vs `IdleEmpty`
     distinction.
  5. Acquire the next eligible entry. If no entry is
     eligible, finish as `Idle304` (every poll response
     reused the cache) or `IdleEmpty` otherwise.
  6. If the entry has a `FinalizationCheckpoint`, jump
     to the matching Phase 6 resume stage (Task 7.5
     composes the resume helper; this task's default
     is `Idle304` so the controller stays contractually
     safe today).
  7. Verify the trigger label, fetch the issue detail,
     build the context JSON, discover the local clone,
     create the worktree + branch, write the prompt.
  8. Spawn the worker through
     `services.process.supervise`. The supervisor's
     `timed_out` / `cancelled` flags route through
     `ActiveRunGuard::finish_cancelled` and return
     `TickOutcome::Cancelled`. Worker exits route through
     `parse_result_file`; an exit-0 / missing / malformed
     result is classified as `FailureClass::Worker` and
     consumes one attempt via `finish_retry`.
  9. On success, run code / investigation / dry-run
     finalization. Teardown always runs through
     `ActiveRunGuard`'s async `finish_*` methods.
  10. Persist `last_tick_finished` and the final
     outcome, including the persisted HTTP status code
     when the failure was a `GitHubApi` typed error.

- **`exit_code_for_tests(outcome: &TickOutcome) -> u8`**
  is the test seam for the cron-tick contract mapping.
  Identical to the private `exit_code_for`.

- **`failure_class_predicates_for_tests(class) -> (bool, bool, bool)`** and **`outcome_for_class_for_tests(class) -> TickOutcome`** are the test seams for the orchestrator's branching logic.

## State/schema effects

- The new `tick` module does not introduce new on-disk
  formats. It composes the existing `state.json`,
  `state_meta.json`, claim files, run-result files, and
  worktrees through the canonical `StateStore` / `MetaStore`
  / `CadenceGate` / `Worktree` / `Supervisor` surfaces.
- The `TickOutcome` enum's variant set changed. Files
  that serialise / deserialise the metadata envelope across
  upgrades will see a backward-incompatible variant
  name; the task packet requires the new variant set
  before the controller ships, so the breaking change
  is accepted at this gate.
- The shared `Arc<Client>` means a regression in
  `Client::with_config` or `Client::with_cache` affects
  every GitHub-mutating path at once. The Phase 6
  integration tests (`pr_test`, `issue_close_test`,
  `failure_investigation_test`) all use the same
  `Arc<Client>` shape and exercise the cache /
  redaction surface.

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 07 gate:

- **#1 (whole-tick lock)** — the controller's first step
  after config load is `DaemonLock::try_acquire`. A
  contended lock returns `TickOutcome::SkippedConcurrent`
  / exit 0 without writing to the queue or metadata.
- **#3 (claim paths through `StateStore`)** — the
  controller only mutates claim state through
  `ActiveRunGuard`'s `finish_*` methods, each of which
  delegates to the canonical `StateStore` operation
  (`complete`, `complete_investigation`,
  `complete_preview`, `retry_or_fail`, `skip`,
  `requeue_infrastructure`). No helper constructs claim
  paths from raw issue strings.
- **#4 (terminal transitions remove claim)** — every
  `finish_*` method on `ActiveRunGuard` is backed by a
  `StateStore` operation that already unlinks the claim
  file. The orchestrator's `tick` body never
  re-acquires a claim without first going through
  `acquire_next`.
- **#5 (daemon owns branch name)** — the controller
  passes `worktree.branch_name` to the worker through
  `CADUCEUS_BRANCH_NAME`; the worker's selection never
  enters the orchestrator's branch decision.
- **#6 (worker session + supervisor)** — the
  `process.supervise` call is the only path that
  spawns a worker; the supervisor's TERM / KILL /
  control-pipe EOF paths are unchanged from Task 5.1.
- **#7 (Rust owns heartbeats)** — the supervisor
  owns heartbeat creation / cleanup; the controller
  never touches heartbeats directly.
- **#8 (no credential leakage)** — the controller never
  resolves tokens, never reads tokens from env, and
  never logs the worker env. The `Client::with_config`
  call applies the documented token redaction.
- **#11 (rate-limit persisted before exit)** — the
  gate's `record_tick_finished` and the controller's
  `finish_tick_failure` both persist rate-limit
  observations before the tick returns.
- **No PAT in arguments, URLs, or env** — the
  controller's only GitHub-bound strings are the
  issue ref (sanitised through `IssueKey::parse`)
  and the worker context JSON (sanitised through
  `sanitized_env`).

## Residual risks

- **Finalization-resume path** — the controller's
  `if claimed.entry.finalization.is_some()` branch
  returns `TickOutcome::Idle304` as a safe default.
  The Phase 6 finalization helpers (`commit_code_result`,
  `push_daemon_branch`, `find_or_create_pull_request`,
  `post_completion_and_close`) are wired through
  `run_code_finalize`; the resume path that
  re-fetches an existing commit and resumes the
  push / PR / comment / close stages will be
  composed in Task 7.5's full-system integration
  suite. Today, a checkpoint-shaped entry still
  releases its claim through `skip` so the reaper
  can clean it up.
- **State-store rate-limit observation** — the
  controller routes rate-limit observations through
  `MetaStore::update` rather than the
  `RateLimitObserver` helper. The `record_rate_limit`
  helper in `tick.rs` is currently unused; the
  controller could be tightened to use the meta
  observer directly in a follow-up. The persisted
  observation is what the next tick's
  `CadenceGate::precheck` reads, so the contract is
  upheld either way.
- **The `record_rate_limit` helper in `tick.rs` is
  dead code today** — kept as a future-tightening
  seam. A future task can drop it without breaking
  the public surface.
- **No automated end-to-end test against a real
  GitHub API** — Task 7.1's tests are deterministic
  and exercise the orchestrator's decision logic. The
  full-system integration suite (Task 7.5) drives
  the live wiremock-based happy paths (code, dry-run,
  investigation, label removal at every stage, rate
  limit at every fetch / finalize stage, etc.).
