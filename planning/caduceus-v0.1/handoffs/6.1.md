## Behaviour (per the contract)

1. **`HEAD` must equal `base_oid`.** A worker-created commit / checkout / merge / rebase / detached HEAD is rejected as a `Worker { context: "commit", stderr: … }` error. The test `commit_rejects_worker_created_commit` covers the positive case (the worker ran `git commit`); `commit_retry_with_existing_checkpoint_skips_work` covers the negative case (after a successful commit, the second call fails with a HEAD-vs-base_oid drift).
2. **`git status --porcelain=v2 -z` is the canonical change-set source.** The parser handles the `1` (changed) and `2` (renamed/copied) header formats plus the untracked and ignored sections.
3. **Control files are excluded.** `worker-result.json` is filtered out of the validated set. The test `commit_skips_control_files` confirms a normal change is committed while a sibling `worker-result.json` is left out.
4. **A code success with no remaining changes is a worker-contract failure.** Both the "worker wrote only a control file" and the "worker wrote nothing" cases return the typed error.
5. **Escaping symlinks are rejected.** The function uses `symlink_metadata` + `read_link` to inspect the file kind; a target that is absolute or starts with `..` is reported.
6. **The daemon's identity is used for the commit.** `git -c user.name=… -c user.email=… commit -m <worker.message>` runs the commit; the `Caduceus Daemon <caduceus@daemon.local>` identity is hard-coded pending the Phase 6 identity-configuration task.
7. **The worker result is atomically copied** to `<state_dir>/runs/<run_id>.result.json` via `write_atomic` after the commit lands.
8. **The commit returns a `CommitOutcome`** with the OID and the daemon branch name; the orchestrator persists the OID in `FinalizationCheckpoint { stage: Committed, commit_oid }` so a retry sees the existing checkpoint and skips the commit step.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No GitHub HTTP call.** The commit step does not touch the API.
- **No push.** `git commit` is local; no `git push` runs in this task.
- **No label change.**
- **No worktree teardown.** The orchestrator tears the worktree down after the commit step lands. The integration test `commit_does_not_touch_parent_main_checkout` confirms the parent clone's `HEAD` is untouched.

## Residual risks

- The daemon identity is hard-coded (`Caduceus Daemon <caduceus@daemon.local>`). The contract says "configured daemon identity" but `Config` does not yet have `git_user_name` / `git_user_email` fields; Phase 6 should add them. A `RawConfig` operator override will land in a later task; today the constants are authoritative.
- The function does **not** re-fetch the issue or check the trigger-label state. The contract requires this *immediately before* the first finalization mutation. That step is the orchestrator's job (Phase 6's `acquire_issue_lock`-equivalent) and lives outside this task's scope. The contract also requires a "user withdrawal" path that transitions the claim to `Skipped`; that, too, is the orchestrator's job.
- The `.git/` guard is structural (a prefix match). The worktree's real `.git/` is a gitlink into the parent clone, so writing into it fails at the file-system layer; the guard is a belt-and-braces check. The integration test pins the contract by re-asserting the prefix rule.
- The retry path is exercised by a direct call: the second `commit_code_and_finalize` returns the HEAD-vs-base_oid error. The orchestrator's "skip on existing checkpoint" logic prevents the second call from happening in the live system; the test asserts the *function's* contract, not the orchestrator's. The full orchestrator test belongs to a Phase 6 task.
