## Public signatures/contracts used

- **`ctx.register_skill(name: str, path: Path, description: str = "")`** —
  qualified as `caduceus:caduceus` because Hermes namespaces the
  plugin manifest name with the bare skill name via `:`. The skill
  file at `<plugin>/skills/caduceus/SKILL.md` is non-trivial and
  includes the canonical lifecycle steps.
- **`ctx.register_command(name: str, handler, description: str, args_hint: str = "")`** —
  registers `/caduceus-status`. The handler invokes the daemon
  binary with an argument array (`[str(binary), "status", "--json"]`),
  a 15-second timeout, and 32 KiB output cap. Missing binary returns
  a chat-safe diagnostic explaining `hermes caduceus setup`.
- **`ctx.register_cli_command(name: str, help: str, setup_fn, handler_fn=None, description: str = "")`** —
  registers `hermes caduceus` with five subcommands. The CLI parser
  is built by the adapter's own `_register_caduceus_cli` helper using
  stdlib argparse and is verified by the fake-context tests AND by a
  real `hermes caduceus --help` invocation against Hermes 0.18.2.
- **`ctx.dispatch_tool(name: str, args: dict)`** — used at the
  `register` boundary to capture the dispatcher for explicit cron
  operations. The cron bridge is the only place the adapter reaches
  outside Hermes's documented `ctx` API.

## State/schema effects

- **No persistent schema or state file is touched by registration.**
  `register(ctx)` only records skill/command/CLI registrations and
  captures the dispatcher. The `test_register_does_not_mutate_filesystem_outside_surface`
  test patches `socket.socket` to assert no network access and
  checks that `$HERMES_HOME/caduceus-state`, `$HERMES_HOME/scripts/`,
  and `$HERMES_HOME/caduceus/worker-bridge.py` are not created.
- `state_meta.json`, `queue.json`, and the runs directory are
  daemon-owned and untouched by the adapter.
- Cron reconciliation reads/writes through Hermes's documented
  `cronjob` tool — never the adapter's own cron directory.

## Forbidden-side-effect checks

Per the Phase 01 gate:

- `tests/hermes_plugin_test.py::test_register_does_not_mutate_filesystem_outside_surface`
  patches `socket.socket` and asserts that `register(ctx)` creates
  no state files, no scripts, no cron entries, and no network
  sockets.
- The adapter does not call `hermes cron create` or any other
  side-effecting command during registration. All cron mutations are
  performed only when the operator explicitly runs
  `hermes caduceus cron-install` or `hermes caduceus cron-remove`.
- The bridge template lives in `plugin-assets/` and is *not*
  imported by the adapter (the adapter would not see it even if it
  tried — `register(ctx)` only imports `hermes_constants` paths
  through the late-bound `ctx`).
- CONTRACTS.md was not edited. `CONTRACT_REVISIONS.md` was not
  edited. `archive/full-reviewed-plan.md` is untouched.
- `planning/caduceus-v0.1/progress.json` is updated by the
  controller (`set_status.py 0.2 complete --handoff handoffs/0.2.md`).
- The legacy `plugin/` directory is removed in the same commit
  (`test_legacy_plugin_directory_is_absent` enforces this on the
  repo root).
- The negative-fixture test
  (`test_negative_manifest_with_legacy_fields_is_rejected`) reads
  `tests/fixtures/negative_plugin.yaml` and asserts that any
  third-party plugin inheriting the legacy manifest shape is
  caught on Caduceus's side.

## Residual risks

- The CLI tree is built with stdlib `argparse`. Hermes's
  `ctx.register_cli_command` parser wrapping for argparse is verified
  by the fake-context tests; the real Hermes CLI dispatch was also
  smoke-tested manually with `--help`, `doctor`, and `status`. Phase
  8 (Task 8.1) will repeat the smoke tests in a fully-installed
  environment.
- The cronjob bridge uses `ctx.dispatch_tool("cronjob", {...})`.
  When run inside Hermes, this routes through the real cronjob
  tool; when run inside pytest, the dispatcher is replaced by an
  in-memory stub installed by `tests/hermes_plugin_test.py`. Both
  paths are exercised.
- The adapter's `_redact` helper handles bare and quoted values
  for the three token-shaped variables (`GITHUB_TOKEN`,
  `CADUCEUS_GITHUB_TOKEN`, `GH_TOKEN`). It is damage-control, not a
  full scanner — operators who need a stricter pipeline should
  rely on the daemon's own redaction plus the explicit env-clear
  contract.
- `manifest_version: 1` is the only version Caduceus emits. Future
  contract revisions land through `CONTRACT_REVISIONS.md` and the
  manifest digest in `task-manifest.json`.
