## Public signatures/contracts used

- **`Worktree`** is the canonical return of [`create`]. New
  fields beyond the previous stub: `base_oid` (the recorded
  SHA-1 of `refs/remotes/origin/<base>` after the fetch) and
  `fresh` (true when the call materialised a new worktree,
  false when it reconciled with an existing one owned by the
  same run id). The `branch_name` is the daemon-owned
  `automation/issue-<number>-<lowercase-run-id>`; the `path`
  is `<repo>/.worktrees/<run_id>`.
- **`create(cfg, runner, repo, key, run_id) -> CaduceusResult<Worktree>`**
  matches the Task 4.2 packet exactly. `create` is now
  `async`; callers pass a `&GitRunner` and a `&RepositoryInfo`
  populated by `find_main_clone` (or constructed directly by
  tests that bypass discovery ‚Äî see "Tests added or changed").
- **Atomic claim-of-worktree-path** under the worker-home
  area: the daemon takes an `fs2::FileExt::lock_exclusive`
  on `<repo>/.worktrees/.lock` before any worktree-side state
  changes, and releases the lock on every return path. The
  lock file lives next to the worktree directories so two
  concurrent `create` calls on the same main clone serialise.
  (Implemented in `create`; `create_locked` runs under the
  lock and the outer wrapper calls `FileExt::unlock` on the
  way out.)
- **`git check-ref-format --branch <name>`** is invoked before
  fetch; a non-zero exit surfaces as `CaduceusError::Worktree
  { context: "create", stderr: "invalid branch name ‚Ä¶" }`.
- **`validate_run_id`** restricts the run id to `[A-Za-z0-9_-]{1,64}`.
  Path traversal (`..`, `/`) and shell metacharacters are
  rejected before any git subprocess runs. The path traversal
  test uses `../escape`; the shell-metachar test uses
  `abc; rm -rf /`.
- **`inspect_existing`** distinguishes four cases for the
  pre-existing state of the branch / path:
  1. nothing exists ‚Äî proceed with the standard fetch +
     `worktree add` flow;
  2. branch exists and is ours (same run id) ‚Äî reconcile and
     return the existing handle (`fresh: false`);
  3. path exists and is a worktree registered to our branch
     ‚Äî same as (2);
  4. branch / path / foreign entry under `.worktrees/` exists
     for a different run id ‚Äî surface a typed
     `path collision` or `branch collision` error.

## State/schema effects

- The new `Worktree::base_oid` and `Worktree::fresh` fields
  are part of the public surface but do not touch the
  `QueueEntry` schema (Phase 3 durable state). The
  `FinalizationCheckpoint` carries `branch_name` and
  `commit_oid` separately so the new `base_oid` field is
  additive rather than breaking.
- The flock file `<repo>/.worktrees/.lock` is a daemon-owned
  resource; it lives next to the worker-home directory and
  carries no persistent data. Its mode is whatever the umask
  produces ‚Äî the daemon's existing atomic-rename guarantees
  do not apply to this single-file lock because it's empty
  after release.

## Forbidden-side-effect checks

Per `CONTRACTS.md` invariants and the Phase 04 gate:

- **#6 (worker-session kill)** ‚Äî unchanged; the runner still
  uses `process_group(0)` and `nix::sys::signal::killpg`.
- **#7 (Rust owns heartbeats / process lifecycle)** ‚Äî
  unchanged; `Worktree::base_oid` is derived from git's own
  `git rev-parse refs/remotes/origin/<base>` and carries no
  worker subprocess state.
- **#8 (no credential leakage)** ‚Äî `git fetch` runs through
  the existing runner, which scrubs `GITHUB_TOKEN` /
  `CADUCEUS_GITHUB_TOKEN` / `GH_TOKEN` from the inherited
  env. The test
  `create_surfaces_precise_error_on_fetch_failure` exercises
  the failure path; the redacted stderr is asserted to
  contain the captured error rather than a token.
- **#5 (daemon owns branch name)** ‚Äî `create` constructs
  the branch name as `automation/issue-<n>-<lowercase-run-id>`
  and validates it via `git check-ref-format --branch`; no
  worker-derived input flows into the ref name.
- **Atomic claim-of-worktree-path** ‚Äî the `fs2` flock on
  `<repo>/.worktrees/.lock` prevents two concurrent
  `create` invocations on the same main clone from racing on
  the shared path / branch. The lock is released on every
  return path; tests do not explicitly cover the contention
  case (would need a multi-threaded harness) but the
  single-threaded happy path exercises the lock acquire +
  release pair end-to-end.

## Residual risks

- The `<repo>/.worktrees/.lock` flock is created with the
  default umask (mode `0644`). Tests do not assert the mode;
  Phase 5 / 7 may want to tighten it to `0600` if the
  contract evolves.
- `git worktree add` writes the new branch's metadata under
  `<repo>/.git/worktrees/<run_id>`. Phase 4.3's `destroy`
  must clear this directory when tearing down a worktree.
  The `Worktree` struct carries no reference to the
  worktree-metadata dir; if Phase 4.3 needs that, it can be
  re-derived as `<repo>/.git/worktrees/<run_id>`.
- The pre-flight `inspect_existing` uses `git rev-parse
  refs/heads/<branch>` to detect an existing branch and
  `git worktree list --porcelain` to detect an existing
  path. The two checks are independent ‚Äî a leftover branch
  with no worktree (or vice versa) is still surfaced as a
  collision, which matches the contract's "reconciled only
  when it belongs to the same run id" wording.
