## Behaviour (per the contract)

1. **Comment idempotency.** The function lists the issue's comments (`GET /repos/{owner}/{repo}/issues/{number}/comments`) and looks for one whose `body` starts with `<!-- automation-run:<run_id>`. If present, no POST is made; the function reports `comment_posted = false` (the comment is already there). The integration test `close_existing_marker_skips_post` confirms the mock's POST handler has `expect(0)`.
2. **Public-voice check.** The summary is validated through `validate_comment` (Task 6.6) before any HTTP request. A rejected text returns `CaduceusError::Other(public-voice: ...)` and never touches the network. The integration test `close_voice_rejection_prevents_http` confirms the mock's GET and POST handlers both have `expect(0)`.
3. **Comment POST.** The body is posted as a comment on the issue. A 201 is required; any other status is a typed `GitHubApi` error.
4. **Close idempotency.** After the comment step (or the no-op skip), the function calls `GET /repos/{owner}/{repo}/issues/{number}` and reads the `state` field. An already-closed issue returns `issue_closed = false`; no PATCH is sent.
5. **Close PATCH.** When the issue is open, the orchestrator routes a PATCH with `{"state": "closed"}`. **The PATCH is *not* in this function** — the v0.1 contract for `Client` exposes only `GET` and `POST`; the orchestrator owns the close PATCH. The function returns `issue_closed = false` for now; the orchestrator's PATCH step (a Phase 6 task) extends the surface.
6. **Withdraw flow.** If the PR exists but the trigger was withdrawn, the contract says the daemon leaves both PR and issue open, persists the checkpoint, transitions to `Skipped`, and emits a diagnostic. **This is the orchestrator's job**; the function always validates the comment body and posts when needed. The orchestrator's `find_or_create_pull_request` (Task 6.3) and `post_completion_and_close` together carry the withdrawal contract; the function here is the comment-and-close hook, not the withdrawal-policy.
7. **Idempotent retry.** A retry that sees the existing comment does not POST again. The function's `comment_posted` flag in the `FinalizeOutput::idempotency_observations` records the no-op so the orchestrator can short-circuit the comment on the next tick.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No `gh` or `git push` here.** The close step talks only to the GitHub API; the runner is not invoked.
- **No PAT in arguments, URLs, or env.** The `Authorization` header carries the bearer token; the runner's credential scrubbing applies.
- **No label change.** The close step does not modify labels.
- **No worktree teardown.** The orchestrator tears the worktree down after the close step lands.

## Residual risks

- The close PATCH is **not** in this function; the `CloseOutcome::issue_closed` is always `false` in v0.1. The orchestrator (a Phase 6 task) must send the PATCH. The integration test pins the comment-idempotency contract, not the close PATCH; the latter is the orchestrator's job.
- The marker detection checks `body.starts_with(&marker_prefix)`. A comment that contains the marker text *in the middle* of the body (e.g. the user quotes it) would not match. The contract's "hidden marker" implies a leading-position match; the integration test confirms this.
- The integration test `close_voice_rejection_prevents_http` covers the canonical forbidden-term case. The validator's other failure modes (over-length, etc.) are tested in `tests/voice_rule_test.rs` from Task 6.6.
- The wiremock 0.6 `query_param` matcher is sensitive to URL encoding; the close tests use the canonical `/repos/owner/repo/issues/1/comments` path with no query string, so the encoding issue does not apply.
