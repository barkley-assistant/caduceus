## Public signatures/contracts used

- **`Client::with_config(cfg: &Config) -> CaduceusResult<Client>`** is the canonical
  production constructor. It parses `cfg.api_base`, opens or creates the
  persistent `HttpCache` rooted at `<cfg.state_dir>/cache/http.json`, and builds
  the inner `reqwest::Client` with `connect_timeout=10s`, the configured
  `cfg.http_timeout_seconds` overall timeout, and `Policy::none()` (the daemon
  walks redirects itself to enforce the same-host rule).
- **`Client::with_cache(cfg: &Config, cache: HttpCache) -> CaduceusResult<Client>`**
  lets tests inject a pre-built cache (e.g. one rooted at a `tempfile::TempDir`).
- **`Client::get(url_path: &str, accept: &str) -> CaduceusResult<Response>`** and
  **`Client::get_url(url: &Url, accept: &str) -> CaduceusResult<Response>`** are
  the typed GET entry points. They split `url_path` on `?`, join the path onto
  the configured `api_base`, send the canonical headers
  (`User-Agent: caduceus/<version>`, `Accept: <accept>`,
  `X-GitHub-Api-Version: 2022-11-28`, plus `Authorization: Bearer <token>` when
  the URL is same-origin and a token is configured), follow up to
  `MAX_REDIRECTS = 3` hops limited to the same scheme/host/port as `api_base`,
  and surface 401/403/404/500 as `CaduceusError::GitHubApi`. The final URL is
  captured on every hop so issue verification can detect a transfer.
- **`HttpCache::open(state_dir: &Path) -> CaduceusResult<HttpCache>`** creates
  `<state_dir>/cache` with mode `0700`, opens `http.json`, recovers from
  malformed JSON by dropping the file, and returns an empty cache.
- **`HttpCache::get(key) -> Option<CacheEntry>`** returns `None` for missing,
  empty, or malformed-ETag entries (so the next request refetches).
- **`HttpCache::put(key, entry) -> CaduceusResult<()>`** rejects invalid ETags
  (drops the entry rather than storing garbage) and persists atomically via
  temp-file + `fsync` + `rename`, with mode `0600` on the destination.
- **`is_valid_etag(value: &str) -> bool`** accepts only the RFC-7232 strong
  (`"abc"`) and weak (`W/"abc"`) quoted forms.
- **`cache_key(url: &str, accept: &str) -> String`** joins the full URL and the
  relevant Accept header with a NUL separator.
- **`Response { status, final_url, body, from_cache }`** is what callers see.
  `from_cache = true` only when a 304 reused a cached body; `final_url` is the
  URL after any redirect chain so transfers are detectable.
- **`HttpClient = Client`** type alias kept so the voice-rule tests
  (Task 6.6) continue to compile.

## Behaviour

- **Headers.** Every request carries `User-Agent: caduceus/<CARGO_PKG_VERSION>`,
  the supplied `Accept`, and `X-GitHub-Api-Version: 2022-11-28`. When the URL is
  same-origin as `api_base` and a token was resolved, the request carries
  `Authorization: Bearer <token>`. Cross-host redirects (rejected before any
  further HTTP) do **not** carry the token.
- **Timeouts.** `connect_timeout = 10s` and the per-request timeout =
  `cfg.http_timeout_seconds` (default 60). Both are wired through
  `reqwest::Client::builder`.
- **Redirects.** Disabled in reqwest (`Policy::none()`); the daemon walks them
  manually, at most `MAX_REDIRECTS = 3` hops, only when the Location header
  points to the same scheme/host/port as `api_base`. A redirect loop or a
  cross-origin target surfaces a `CaduceusError::Other` with the source and
  destination URLs. The `Authorization` header is **not** forwarded to a
  cross-origin target.
- **Conditional GET.** On a 200/201 response, the body's bytes are stored in the
  cache under the URL+Accept key, alongside the server-supplied ETag (if it
  passes `is_valid_etag`). The next request to the same URL sets
  `If-None-Match: <etag>`; on 304 the cached body and final URL are returned
  with `from_cache = true`. A 304 with no cached body is surfaced as a typed
  `CaduceusError::GitHubApi`.
- **Body cap.** Responses are streamed through `bytes_stream` with a 10 MiB
  cap; an over-cap body returns a typed "exceeds … bytes" error before the
  next chunk is allocated.
- **Status mapping.** 401/403/404/500/429 and other non-2xx/304 statuses map
  to `CaduceusError::GitHubApi { status, message }` with the message scrubbed
  through `crate::error::scrub` (token-shaped assignments are replaced with
  `<redacted>`).
- **Cache corruption recovery.** Malformed JSON in `http.json` drops the file
  on read; the cache returns an empty state. An invalid ETag is not stored;
  the next request refetches unconditionally.
- **Atomic writes.** `HttpCache::put` writes to
  `<dir>/.http.json.<ulid>.tmp`, calls `sync_all`, then `rename`s onto the
  final path. The directory is created with mode `0700` and the cache file
  with mode `0600` on Unix.

## State/schema effects

- A new directory `<state_dir>/cache/` (mode `0700`) is created on first use.
- A new file `<state_dir>/cache/http.json` (mode `0600`) holds a JSON object
  `{ "by_key": { "<url>\0<accept>": { "etag": "...", "status": 200, "body":
  "base64-or-utf8", "final_url": "..." } } }`. Empty when the daemon has not
  yet made a request.
- `state_meta.json` is **not** modified by this task; rate-limit observation
  (Task 2.4) and the meta schema (Task 7.x) own that file.
- No subprocess execution, no worker invocation, no network egress outside the
  loop the test suite drives with `wiremock`.

## Tests added or changed

`tests/github_client_test.rs` — 15 tests, all green:

- **Required headers** — every outbound GET carries `User-Agent`,
  `Accept: application/vnd.github+json`, `X-GitHub-Api-Version: 2022-11-28`,
  and (when a token is configured) `Authorization: Bearer …`.
- **Missing token** — when `github_token` is `None`, no `Authorization` header
  is set.
- **Request timeout** — a mock server that sleeps for 60s is rejected well
  inside the client's 5s per-request timeout.
- **Same-host redirect** — a 302 to a same-host path is followed; the
  `final_url` reflects the post-redirect URL.
- **Cross-host redirect** — a 302 to a different mock server is refused, the
  token never reaches the second server (verified by an `expect(0)` mount on
  the second server).
- **Redirect loop** — `/loop → /loop` is bounded by `MAX_REDIRECTS = 3` and
  surfaces a "too many redirects" error.
- **Conditional GET** — first client gets a 200 with ETag `"abc"`; second
  client opens the same on-disk cache, sends `If-None-Match: "abc"`, receives
  304, and the cached body is returned verbatim with `from_cache = true`.
- **Cache corruption** — a hand-written malformed JSON file is dropped on
  `HttpCache::open`, the next call returns an empty cache.
- **Invalid ETag** — `not-a-quoted-tag` is not stored; the next request
  refetches unconditionally.
- **Oversized body** — a `MAX_BODY_BYTES + 1` byte body is rejected before
  full allocation; the error carries an "exceeds … bytes" message.
- **Status mapping** — 401, 403, 404, and 500 each round-trip through
  `CaduceusError::GitHubApi { status, message }`.
- **Token redaction** — an error message containing `GITHUB_TOKEN=…` never
  reaches `Display` or `Debug`.
- **Pure-helper unit tests** — `is_valid_etag` accepts strong and weak
  quoted forms and rejects everything else; `cache_key` is sensitive to both
  URL and Accept.
- **Final URL** — survives a 301 redirect, allowing downstream verification
  to detect a transfer.

## Forbidden-side-effect checks

Per CONTRACTS.md and the Phase 2 task packet:

- The HTTP client never forwards the resolved GitHub token to a cross-origin
  target — verified by `cross_host_redirect_is_rejected_and_token_is_not_forwarded`.
- The cache file is written via temp-file + `fsync` + `rename`; on Unix the
  directory is `0700` and the file is `0600`. Atomic replacement is exercised
  on every successful 200.
- Cache corruption drops the affected file and refetches; never silently
  truncates a real cache or replaces it with an empty state.
- The streaming body cap is enforced before a body that would exceed
  `MAX_BODY_BYTES` is fully allocated; the test sends a single body just over
  the cap.
- Token values never appear in `Display` or `Debug` of any error variant
  produced by the HTTP layer — verified by `token_value_never_appears_in_errors`
  and by the existing 28 `error_test` cases.
- All tests are deterministic; no real network egress is performed (wiremock
  runs in-process).
- No subprocess execution, no file mutation outside `<state_dir>/cache/`, no
  env mutation. The only filesystem effect in tests is the cache directory
  under `tempfile::tempdir()`.

## Residual risks

- The 10 MiB streaming cap is enforced by accumulating chunks into a
  `Vec<u8>` and bailing when `len() + chunk.len() > MAX_BODY_BYTES`. The
  in-flight buffer can briefly hold up to `MAX_BODY_BYTES + max_chunk - 1`
  bytes; that is well below the daemon's normal address-space and is
  documented here so a later reviewer doesn't mistake it for a true
  zero-copy stream.
- The cross-host redirect check compares scheme + host + port only. A
  redirect to a different path on the same host is followed; a redirect to
  `api.github.com:443` while the configured `api_base` is
  `api.github.com:443` is treated as same-origin. Operators that route
  GitHub through a corporate TLS-terminating proxy should set `api_base`
  to the proxy URL so the comparison is exact.
- `Client::new(base_url)` remains as an inert constructor for the voice-rule
  tests (Task 6.6) so that file keeps compiling without rippling changes
  outside Phase 2. The constructor builds an `HttpCache` rooted at a fresh
  `tempdir` location; it never sees real HTTP traffic in the voice-rule test
  path.
- The cache file uses serde's default JSON encoding. Bodies are stored as
  `Vec<u8>` which serialises as a JSON array of integers; for a typical
  GitHub JSON issue list this is acceptable. A future task that adds
  binary endpoints (e.g. release assets) should re-encode the body as
  base64 to keep the file UTF-8 clean.
