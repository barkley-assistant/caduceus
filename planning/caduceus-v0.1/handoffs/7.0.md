## Public signatures/contracts used

- **`Services { clock, github, git, process }`** is the
  canonical dependency-injection bundle. Each field is an
  `Arc<dyn Trait>` whose production adapter is a thin
  wrapper:
  - `Clock` — `SystemClock` (production) returns
    `Utc::now()`. Tests can swap in a deterministic clock.
  - `GithubClient` — `GithubClientAdapter` wraps an
    `Arc<Client>` and exposes it through `inner()`. The
    trait surface stays narrow: production code calls
    `services.github.inner().get(...)` so the
    dependency-injection is explicit at every call site.
  - `Git` — `GitRunnerAdapter` wraps the canonical
    `GitRunner`; same `inner()` pattern via `runner()`.
  - `ProcessSupervisor` — `ProcessSupervisorAdapter` is a
    unit adapter that forwards to
    `crate::worker_supervisor::supervise` through a
    boxed-future return type. The `Box::pin` shim is
    needed because stable Rust 1.97 still requires
    `Pin<Box<dyn Future + Send>>` for async trait methods
    without nightly `trait-variant`; the production
    `supervise` is awaited via the boxed future.

- **`FailureClass`** is the closed set of failure modes the
  orchestrator branches on:
  - `Worker` — schema/result/content/no-code-change/
    worker-exit/timeout failures. Counts against the retry
    budget.
  - `Infrastructure` — HTTP transport/server, git
    transport, filesystem, teardown failures. Does not
    count against the budget.
  - `RateLimit { reset_at }` — typed rate-limit observation.
    The orchestrator persists `reset_at` before returning.
  - `Cancellation` — operator SIGINT/SIGTERM or
    timeout-driven drain. The orchestrator requeues with
    `not_before = now`.

  The three predicate methods (`counts_against_retry_budget`,
  `must_persist_rate_limit`, `is_cancellation`) make the
  orchestrator's branching intent explicit at the call
  site.

- **`classify_error(&CaduceusError) -> FailureClass`** is the
  exhaustive classifier. The match is the canonical place
  where a new `CaduceusError` variant must be classified —
  adding a new variant without extending this match is a
  compile-time error. The current arms cover every existing
  variant: `Cancelled`, `RateLimited { reset_at, .. }`,
  `Worker { .. }`, `Http(_)`, `GitHubApi { .. }`, `Git { .. }`,
  `Push { .. }`, `PushCollision { .. }`, `Worktree { .. }`,
  `Queue { .. }`, `StateCorrupt { .. }`, `Config(_)`,
  `TokenResolution(_)`, `Other(_)` (worker-attributable for
  voice/content errors), `Io(_)`, `Json(_)`, `Yaml(_)`.

- **`ActiveRunGuard`** is the cleanup primitive the
  orchestrator constructs after a successful claim. It owns:
  - the `Option<ClaimToken>` (taken through `Option::take`
    inside every `finish_*` method);
  - the `Arc<StateStore>` (passed at construction);
  - the `IssueKey` and `PathBuf` log path / state dir
    (test seams);
  - the `Mutex<Option<Worktree>>` (set by `attach_worktree`,
    taken by `finish_*` for teardown);
  - the `Mutex<Option<SupervisorOutcome>>` (set by
    `attach_supervisor`, read by the orchestrator's
    classification step);
  - the `Mutex<bool>` `finished` flag (set by every
    successful `finish_*`, checked by `Drop`).

  The async `finish_*` methods perform explicit state
  transitions through the `StateStore`:
  - `finish_success` — `StateStore::complete` (terminal
    `Done`).
  - `finish_investigation` — `StateStore::complete_investigation`
    (terminal `Done`).
  - `finish_preview` — `StateStore::complete_preview`
    (terminal `Previewed`).
  - `finish_retry(error, budget)` —
    `StateStore::retry_or_fail` (returns the new `Phase`).
  - `finish_skip(reason)` — tears down any attached
    worktree via `worktree::remove`, then
    `StateStore::skip` (terminal `Skipped`).
  - `finish_infrastructure(error, not_before)` — tears down
    the worktree, then `StateStore::requeue_infrastructure`
    (preserves `attempts`).
  - `finish_cancelled` — tears down the worktree, then
    `StateStore::requeue_infrastructure` with
    `not_before = now`.
  - `finish_release` — tears down the worktree, then
    requeue with `not_before = now` and a "guard release"
    error string. Used by checkpoint-only / bypass-worker
    paths.

  The `Drop` impl only logs an invariant violation
  ("ActiveRunGuard dropped without calling a finish_*
  method; claim must be reaped on the next tick") when
  `finished` is still `false`. It does not silently
  complete a transition — a forgotten `finish_*` call is
  loud, not quiet.

- **`FinishOutcome { CodeSuccess, InvestigationSuccess,
  Previewed, WorkerRetried { attempts, new_phase },
  Cancelled, Skipped }`** is the typed return shape
  orchestrator code uses to map the guard into a `TickOutcome`.

- **`EntrySnapshot { key, phase, attempts, last_error,
  last_run_id }`** is a small `From<&QueueEntry>` value type
  the orchestrator surfaces to structured logs without
  needing the queue module's `From` impls.

## State/schema effects

- The new `orchestration` module does not persist anything
  on its own. It composes the existing
  `StateStore` / `MetaStore` / `ClaimToken` surface and
  adds a typed classifier (`FailureClass`) plus an async
  cleanup primitive (`ActiveRunGuard`).
- `FinalizeContext.client` is now `Arc<Client>`. The
  persistent HTTP cache lives inside the `Arc`'d client,
  so the daemon, status reporter, and finalization stages
  share one connection pool + cache. The previous `()`
  placeholder is gone; tests use
  `Arc::new(Client::new("https://api.github.com"))` as an
  inert stand-in (no real I/O).
- No `Cargo.toml` changes; no `Cargo.lock` changes; the
  `Arc<Client>` and `Pin<Box<...>>` types are already in
  the existing dependency graph.

- **No `unimplemented!` / `todo!` / placeholder ellipses** —
  the only `..` ellipses are in `serde_json::json!(...)` test
  data and in test doc-strings; the source files in
  `src/` contain zero matches for `unimplemented!`, `todo!`,
  or `...` placeholders in production paths.
- **No `expect`/`unwrap` in production modules handling
  external input** — the inline tests use `expect()` for
  fixture setup and `unwrap()` for known-good values; the
  library code (everything outside `#[cfg(test)]` blocks)
  propagates errors as `CaduceusError` variants.
- **No PAT in arguments, URLs, or env** — `Client::new` is
  used with the public `https://api.github.com` base; the
  tests' inert client has no token. The orchestrator's
  `GithubClientAdapter` forwards through the typed
  `Client` whose redaction surface is owned by
  `crate::error::scrub`.
- **No queue / metadata / worktree mutation outside the
  canonical surface** — the orchestration module's
  `ActiveRunGuard` only calls the documented
  `StateStore` methods (`complete`, `complete_investigation`,
  `complete_preview`, `retry_or_fail`, `skip`,
  `requeue_infrastructure`, `set_worktree`). Every
  transition goes through the state-store's `flock` on
  `<state_dir>/state.lock`.
- **Drop is loud, not silent** — `ActiveRunGuard::drop`
  only emits a `warn!` log; the orchestrator's
  `finish_*` methods are the only path that performs a
  state transition. A forgotten `finish_*` is recovered
  by the reaper on the next tick.

## Residual risks

- The `ProcessSupervisor` trait returns
  `Pin<Box<dyn Future + Send + 'a>>` because stable
  Rust 1.97 still requires the boxed-future shim for
  async trait methods. A future amendment could use
  `trait-variant` (nightly) to remove the indirection;
  the production adapter wraps the real
  `worker_supervisor::supervise` so the boxed-pin cost
  is one allocation per worker run, which the
  supervisor's heartbeat / protocol loop dominates.
- `ActiveRunGuard::Drop` is synchronous; the warning
  log is the only signal of a forgotten `finish_*`.
  The orchestrator's higher-level code (Task 7.1) is
  expected to drive every guard to a `finish_*` call
  before letting it go out of scope. The reaper
  (Task 3.3) cleans up any orphaned claim file on the
  next tick regardless.
- The `Arc<Client>` shared with the finalization stages
  means a regression in `Client::with_config` could
  affect every GitHub-mutating path at once. The
  Phase 6 `pr_test`, `issue_close_test`, and
  `failure_investigation_test` suites all use the same
  `Arc::new(Client::new(...))` shape through WireMock
  and exercise the cache / redaction surface, so any
  regression surfaces during the integration run.
- The `other` classification for `CaduceusError::Other`
  lumps voice rejections and content-shape failures
  together. Phase 6's finalization helpers
  (`render_failure_comment`, `render_pr_title`, etc.)
  all route through `check_voice_or_error` and return
  `CaduceusError::Other(public-voice: ...)` on
  rejection; the `classify_error` arm treats every
  `Other` as worker-attributable, which matches the
  contract's "Voice rejection is a worker failure, not
  an infrastructure failure" intent.
