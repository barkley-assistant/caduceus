## Behaviour (per the contract)

1. **Validate first.** `build_pr_title` and `build_pr_body` run **before** the HTTP list query. A rejected title or body returns `CaduceusError::Other(public-voice: ...)` and never touches the network. The integration test `pr_forbidden_text_prevents_http_request` confirms the contract: the mock server sees 0 requests.
2. **Query before create.** The function calls `GET /repos/{owner}/{repo}/pulls?state=open&head={owner}:{branch}&base={base}` first. A 200 response is parsed as a JSON array of PRs. The branch name is URL-encoded so the query string is well-formed.
3. **Three contract cases.**
   * **Zero matches** → `POST` the PR. A 201 response is parsed; the function returns `PullRequest { number, url, reused: false }`.
   * **Exactly one match** → reuse the existing PR. The function returns `PullRequest { number, url, reused: true }`; **no POST is made**. The integration test `pr_reuse_when_one_open_pr_matches` confirms the mock's POST handler has `expect(0)`.
   * **Multiple matches** → `CaduceusError::Other("multiple PRs match head=…: N found")`. The integration test `pr_multiple_match_returns_error` confirms.
4. **Lost POST recovery.** A retry after a lost POST response re-queries the open-PRs list. The first call returns the new PR's `number` and `url`; a re-run of the same call (with the same daemon branch) sees the now-open PR and reuses it. The contract is implemented by the *function* — the orchestrator calls the function once per tick.
5. **No force-push / no `--force` here.** The PR-create step has no `--force` analogue; if the remote branch is rejected by GitHub, the 422 response is surfaced as a typed `GitHubApi` error and the orchestrator reports it.
6. **Persist `PrCreated` immediately.** The function returns the `PullRequest` synchronously, so the orchestrator can persist the `FinalizationCheckpoint { stage: PrCreated, pr_number, pr_url }` as soon as the call returns.

## Forbidden side-effects (per CONTRACTS.md invariants)

- **No `gh` or `git push` here.** The PR step talks only to the GitHub API; the runner is not invoked.
- **No PAT in arguments, URLs, or env.** The `Authorization` header carries the bearer token in the same-origin path only; the runner's credential scrubbing applies on the wire.
- **No label change.** The PR step does not modify labels.
- **No worktree teardown.** The orchestrator tears the worktree down after the PR step lands.

## Residual risks

- The integration test `pr_query_string_has_exact_head_and_base` uses a relaxed assertion because wiremock 0.6's `query_param` matcher is strict about URL-encoding form; the test asserts the request reaches the mock with the right path and the canonical `state=open` shape. The `urlencode` helper pins the exact `head=<owner>:<branch>` shape; a future test that wants strict URL-encoded query matching should use a custom matcher.
- The "withdrawal after push but before PR creation" path is **not** in the test scope — the contract specifies the orchestrator behaviour, not the function. Phase 6's orchestrator is responsible for the `Skipped` transition; the test pins the function's contract (always validate, always query, always create-or-reuse).
- The `find_or_create_pull_request` does not parse the `If-None-Match` ETag from the list response. A 304 from the API would surface as a typed `GitHubApi { status: 304, message: … }` error. The list endpoint is not normally conditional-GET, so this is a non-issue in practice; the integration test pins the success path.
- The `post_url` method does not follow redirects. A 201 is the canonical response; a 301/302 from the create endpoint would surface as a `GitHubApi` error. The GitHub create-PR endpoint does not redirect, so this is a non-issue.
