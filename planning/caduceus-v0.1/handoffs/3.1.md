## Public signatures/contracts used

The implementation matches the Task 3.1 packet and CONTRACTS.md
verbatim. Highlights:

- **`StateStore::open(state_dir: &Path) -> CaduceusResult<Self>`**
  creates `<state_dir>` and `<state_dir>/claims/`, validates the
  existing `state.json` (a missing file is the empty envelope; a
  malformed file is preserved and surfaced as
  `CaduceusError::StateCorrupt`).
- **`StateStore::snapshot()`** holds a *shared* `flock` on
  `<state_dir>/state.lock` and never writes. Repeated calls preserve
  `state.json` mtime, even when the file does not exist (no
  implicit create).
- **`StateStore::enqueue(key, ticket_type, dry_run)`** returns
  `EnqueueOutcome::{Inserted, AlreadyPresent, Promoted}`. A dry-run
  call against an existing `Previewed` entry is `AlreadyPresent`
  (no promotion); a non-dry call promotes it to `Queued`.
- **`StateStore::acquire_next(run_id, pid, now)`** picks the
  eligible `Queued` entry with the smallest `(queued_at, display_key)`
  where eligibility means `next_attempt_at <= now` (or `None`). The
  entry is rewritten `InProgress` *before* the claim file is
  created; the claim file is written with `O_CREAT | O_EXCL`
  (`OpenOptions::create_new(true)`) so two concurrent acquires
  cannot both win. Returns `None` when nothing is eligible.
- **`ClaimToken`** owns its `claims_dir` + `digest` + `run_id`.
  The `for_test` constructor lets acceptance tests exercise the
  "wrong token" rejection path without going through
  `acquire_next`.
- **`StateStore::set_worktree(claim, path)`** and
  **`save_finalization(claim, checkpoint)`** verify the token
  matches the current `last_run_id` of the in-progress entry
  before persisting.
- **`StateStore::complete(claim)`**,
  **`complete_investigation(claim)`**, **`retry_or_fail(claim,
  error, budget)`**, **`requeue_infrastructure(claim, error,
  not_before)`**, and **`skip(claim, reason)`** are the terminal
  transitions pinned by CONTRACTS.md invariant #4. Each unlinks
  the claim file at the end; a unlink failure is logged and
  surfaced via `tracing::warn!` without rolling back the durable
  phase change (the reaper repairs orphan claims idempotently).

The atomic write strategy â€” `atomic_write(path, body)` writes to
`path.with_extension("json.tmp")`, `fsync`s, then `rename`s â€” is
the same pattern used by `meta::save_atomic` so the state file
gets the same crash-safety guarantee the metadata envelope has had
since Phase 1. After the rename, the state directory is
`fsync`'d on Linux (best-effort no-op elsewhere) so the directory
entry is flushed before the lock is released.

## State/schema effects

`src/queue.rs` now owns:

- `QUEUE_FILE_VERSION`, `STATE_FILENAME`, `CLAIMS_DIRNAME`,
  `STATE_LOCK_FILENAME`, `CLAIM_FILE_VERSION` constants so the
  on-disk file names have a single source of truth.
- The `ClaimFileBody` schema:
  `{ version, key, run_id, pid, process_start_identity, started_at, worktree_path }`.
  `process_start_identity` is composed from the boot-id at
  `/proc/sys/kernel/random/boot_id` and the starttime field of
  `/proc/<pid>/stat`; on failure to read either component it
  falls back to `"<unknown-boot>":0` rather than panicking â€” the
  reaper is the authority that re-validates identity before
  trusting a claim, and the production daemon will run as root
  on the same host as the daemon.
- `display_digest(&str) -> String` â€” SHA-256 hex of the lowercase
  display key. Used both to derive the claim file basename and to
  verify a `ClaimToken`'s digest against the candidate entry.

## Forbidden-side-effect checks

Per CONTRACTS.md "Non-negotiable v0.1 invariants" #1, #2, #3,
and #4:

- **#1 (whole-tick lock) is unaffected by this task.** The
  `state.lock` flock introduced here is *distinct* from
  `<state_dir>/daemon.lock` (CONTRACTS.md invariant #1). The
  daemon-wide lock is added by a later task; this task only adds
  the per-store mutation lock.
- **#2 (atomic write with fsync)** â€” every mutating
  `StateStore` operation calls `atomic_write` (write-temp +
  `sync_all` + rename) followed by `sync_dir` on Linux. A
  malformed `state.json` is preserved verbatim; opening one
  returns `CaduceusError::StateCorrupt { path, message }`
  without touching the file.
- **#3 (queue helpers never construct claim paths from raw
  strings)** â€” claim filenames are derived from
  `display_digest(entry.key.display_key())` inside
  `StateStore::acquire_next`. The only call site that constructs
  a `ClaimToken` from a raw digest is the test-only
  `ClaimToken::for_test`, which is gated by `#[doc(hidden)]` and
  is not used by production code.
- **#4 (terminal transitions remove claim)** â€” `complete`,
  `complete_investigation`, `retry_or_fail`,
  `requeue_infrastructure`, and `skip` all call
  `unlink_claim_best_effort`. A failed unlink is logged at WARN
  and surfaced through `tracing`; the durable phase change is
  not rolled back, matching the Task 3.1 packet ("a
  claim-unlink failure is reported without rolling back the
  durable phase and is repaired idempotently by the reaper").

## Residual risks

- **`sync_dir` is a best-effort Linux no-op on platforms where
  opening a directory fails.** Linux supports opening and
  fsyncing a directory; macOS allows `open()` on a directory but
  the behaviour of `fsync` on a directory fd is filesystem
  dependent. The implementation logs and continues â€” the rename
  itself is atomic on every supported Unix platform, so the
  *contents* of `state.json` survive a crash either way. Only the
  *directory entry* durability is best-effort.
- **`process_start_identity` falls back to `"<unknown-boot>":0`
  on platforms without `/proc`.** The reaper in Phase 3.x is
  expected to re-validate identity before trusting a claim; the
  current implementation records what it has and never panics.
- **`set_worktree` and `save_finalization` take a separate
  `verify_claim_run_id` snapshot read before the mutating
  lock.** This costs one extra `flock` round-trip per write;
  acceptable here because both operations happen at most a few
  times per worker run. A future optimisation could fold the
  verification into the exclusive-locked path.
- **The 8-thread / 16-entry concurrency test asserts that every
  winning thread gets a distinct entry.** Because `BTreeMap`
  iteration order is stable, two threads that both pick the
  "first eligible" entry would serialise at the `flock`, so the
  invariant holds empirically; the test does not prove it
  formally. The `O_CREAT | O_EXCL` claim-file create is the
  definitive no-lost-update guarantee.
