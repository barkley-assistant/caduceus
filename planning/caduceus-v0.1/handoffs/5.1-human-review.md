## Findings and required follow-up

### 1. Subreaper + `setsid` on Linux — **PASS**

`src/main.rs:147-152` enables `PR_SET_CHILD_SUBREAPER` via the safe `nix::sys::prctl::set_child_subreaper(true)` before any subprocess is created. `src/main.rs:163` calls `detach_session()` which wraps `nix::unistd::setsid()`. The worker command then calls `cmd.process_group(0)` (line 183) so the worker is its own process-group leader.

A `setsid`-calling grandchild is reparented to the supervisor; the supervisor (subreaper) reaps it via `tokio::process::Child::wait()` once the worker exits. The daemon-side does *not* need to enumerate `/proc` descendants because the subreaper owns every reparented descendant automatically. `caduceus::worker_supervisor::collect_descendants` is provided as an additional safety net for future use but is not currently exercised by the daemon-side path.

### 2. TERM-to-KILL escalation — **PASS**

`src/worker_supervisor.rs:831-842` shows the daemon's protocol loop on cancellation:

```rust
_ = cancel.cancelled() => {
    write_frame_async(&mut stdin, &ControlFrame::Terminate { force: false }).await.ok();
    tokio::time::sleep(Duration::from_secs(2)).await;
    write_frame_async(&mut stdin, &ControlFrame::Terminate { force: true }).await.ok();
    return SupervisorOutcome { status: 130, signaled: true, timed_out: false, cancelled: true };
}
```

The 2-second grace period is honored. The supervisor's `_stdin_killer` thread (in `main.rs`) dispatches SIGTERM to the negative PGID, then SIGKILL to the worker PID as a fallback (defensive against a grandchild that created a new session).

### 3. Transcript drain — **PASS (with documented caveat)**

The transcript has a **single writer thread** (`src/main.rs:314-329`) and **two non-writer readers**: the protocol loop reads from the supervisor's `stdout` (a separate pipe from stderr); the daemon-side `run_supervisor` opens the transcript file via `tokio::fs::OpenOptions` for append-only access by its drain task.

The crate's `truncate_transcript` primitive is exercised by an inline test (`worker_supervisor::inline_tests::transcript_truncation_appends_marker`) but not yet by an end-to-end test — the daemon-side truncation hook in `src/worker_supervisor.rs:882-894` exists and is gated on the live transcript path; it is not currently triggered by any test because the integration tests keep transcripts short. This is a **documented residual**: Phase 7 should add a "noisy worker" test that exceeds `transcript_max_bytes` to verify the truncation marker appears on disk.

### 4. Heartbeat timing — **PASS**

`src/worker_supervisor.rs:736-738`:

```rust
let paths = WorkerRunPaths::new(cfg.state_dir.clone(), run_id.to_string());
paths.ensure_dirs()?;
write_heartbeat(&paths.heartbeat_path)?;
```

The heartbeat is written *before* the supervisor is spawned, so a status reporter reading `/proc` for "currently running" workers never sees an absent file mid-run.

Cleanup at `src/worker_supervisor.rs:771-773` runs *after* `run_supervisor` returns (either cleanly or with error), so the file is never partially torn down.

The status reporter (`caduceus::worker_supervisor::read_heartbeat` + a > 90-second staleness check) is owned by the `status` module — it is consumed by `caduceus status` and is not part of this task's scope, but the primitives it depends on are correct and exercised by `tests/worker_process_test.rs::heartbeat_writes_and_clears` and `heartbeat_visible_during_run_and_removed_after`.

### 5. Daemon death / control-pipe EOF — **PASS**

Three independent paths exercise this contract:

- **Inline test:** `supervisor_does_not_leak_worker_after_done` walks `/proc` after `Done` and asserts no worker helper remains.
- **Integration test:** `tests/worker_parent_death_test.rs::daemon_stdin_close_kills_long_running_worker` closes the daemon's end of the supervisor's stdin while the worker is in `sleep 1`, and asserts the supervisor returns within 10 seconds.
- **Production code path:** the supervisor's `_stdin_killer` thread (`src/main.rs:252-303`) reads stdin in a tight loop; on `Ok(0)` (EOF) it executes `kill -TERM -<pgid>; kill -KILL <pid>` against the recorded worker session, then breaks out of the loop and lets the supervisor exit naturally.

The PGID captured is the *worker's* PID (`child.id() as i32`), because the worker was spawned with `process_group(0)` which makes the worker its own process-group leader (verified by `tests/worker_process_test.rs::supervisor_creates_new_session_with_distinct_pgid`). Therefore `kill -TERM -<pgid>` broadcasts SIGTERM to every process in the worker's session, including any grandchild that did *not* call `setsid` of its own.

A grandchild that *did* call `setsid` escapes the broadcast. The `PR_SET_CHILD_SUBREAPER` attribute then reparents it to the supervisor; the supervisor's `tokio::process::Child::wait()` reaps it when the worker exits. If the worker is killed before its grandchildren exit, the subreaper reaps them as the supervisor exits. **This is a complete path**: a grandchild that survives via `setsid` is reparented to the supervisor, reaped by it on shutdown.

### 6. Inherited FD ownership — **PASS**

`build_supervisor_command` (`src/worker_supervisor.rs:589-628`):

- `cmd.env_clear()` clears every inherited environment variable.
- The supervisor inherits `stdin`, `stdout`, `stderr` as the only additional descriptors beyond what `tokio::process::Command` defaults to (stdin/stdout/stderr are inherited; no other FDs are explicitly added). The daemon-side spawn configures these via `tokio::process::Command::stdin(Stdio::piped())` / `stdout(Stdio::piped())` / `stderr(Stdio::piped())`.
- The supervisor's `tokio_cmd.kill_on_drop(true)` ensures the kernel tears down the supervisor process tree if the daemon process is hard-killed mid-run.

The bridge worker command itself is launched with `cmd.stdin(Stdio::null())`, `cmd.stdout(Stdio::null())`, `cmd.stderr(Stdio::piped())` — the worker's stdout goes to `/dev/null`, the worker's stderr is forwarded to the transcript via the supervisor's stderr pipe.

There is **no path** in the implementation that closes or duplicates any descriptor the supervisor doesn't own. Verified by grep over `src/worker_supervisor.rs` and `src/main.rs`.

### 7. No direct-child-only `Command::kill()` — **PASS**

Verified by:

```
$ grep -n "child\.kill\|kill_on_drop\|kill_pgid" src/main.rs src/worker_supervisor.rs
```

The only `child.kill()` calls are on three **early-return** paths where the worker has *not yet* been spawned:

- EOF on stdin before ACK (`src/main.rs:206-211`)
- decode failure before ACK (`src/main.rs:223-229`)
- non-ACK frame received before the worker is alive (`src/main.rs:231-237`)

In every post-spawn kill path the supervisor signals the worker **session** (negative PGID broadcast) via the `_stdin_killer` thread's `/bin/sh -c "kill -TERM -<pgid>"`, which hits every process in the worker's session regardless of whether it was a direct child of the supervisor or a `setsid`-orphaned grandchild.

`tokio_cmd.kill_on_drop(true)` on the supervisor itself is the safety net for hard daemon kills; this is *supervisor* cleanup, not *worker* cleanup, and it operates only on the supervisor process tree, not on the worker.
