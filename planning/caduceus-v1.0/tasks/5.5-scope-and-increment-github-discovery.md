# Task 5.5: Scope and increment GitHub discovery

## Outcome and required behavior

Scope and increment GitHub discovery. Deliver the behavior assigned here and in
`CONTRACTS.md` as one
reviewable GitHub issue or SDD work unit.

**`api_base` allowlist.** This task owns the positive allowlist
that validates `api_base` at configuration load time. Per
`CONTRACTS.md` GH-001, `api_base` MUST be either the literal
`https://api.github.com` or an `https://` URL matching the GHES
host pattern documented in `docs/configuration.md` §"`api_base`".
The validator MUST reject anything else with a configuration error
and MUST NOT rely on string matching against
`comment_forbidden_strings` or any other forbidden-list filter to
detect a non-GitHub endpoint.

## Execution boundaries

- Owned surfaces: repository scope, polling, ETags, and the
  `api_base` allowlist validator.
- Dependencies: 3.7, 4.4.
- Do not implement deferred v1.x work or modify the v0.1 planning tree.
- Preserve unrelated changes and the established public documentation voice.

## Primary tests

- scope, pagination, ETag, retry, and redaction tests.
- `api_base` allowlist: positive (`https://api.github.com`,
  `https://ghes.example.com/api/v3`) and negative
  (`http://...`, `https://bitbucket.example.com`, custom
  path-prefixed proxies, malformed URLs).

## Acceptance checks

- **5.5-AC-01** — Discover only explicitly scoped repositories.
- **5.5-AC-02** — Poll incrementally within bounds.
- **5.5-AC-03** — Never expose PATs to workers, transcripts, Git, or public
  text.
- **5.5-AC-04** — Use noninteractive ephemeral PAT transport with no secret in
  argv, URLs, Git configuration, durable files, logs, or evidence.
- **5.5-AC-05** — Validate `api_base` against the positive GitHub.com +
  GHES allowlist at `Config::load`; reject every other form with a
  configuration error; never use forbidden-string matching as a proxy for
  endpoint validation.
