# Task 0.1: Publish the v1.0 task catalog

## Outcome and required behavior

Publish the v1.0 catalog with a public, evidence-backed readiness audit. The
audit identifies implementation gaps and routes them; it does not claim that
Phase 00 fixes production behavior.

## Execution boundaries

- Own the catalog plus six public Markdown attachments under
  `handoffs/readiness/`: capability inventory, reachability map, operator
  journeys, fault injection, requirement evidence, and gap register.
- Dependencies: none.
- Do not implement deferred v1.x work or modify the v0.1 planning tree.
- Preserve unrelated changes and the established public documentation voice.

## Primary tests

- Readiness artifact schema, cross-link, ownership, contradiction, command
  reproduction, validator, and selector tests.

## Acceptance checks

- **0.1-AC-01** — Inventory every public capability found in README, docs, CLI,
  plugin, configuration, bridge, install, migration, recovery, and release
  surfaces as `working-production`, `integrated-not-proven`, `stub`,
  `fake-only`, `planned`, or `contradicted`.
- **0.1-AC-02** — Map each public command through its adapter and production
  function to a test, detecting stubs, dead paths, and helper-only coverage.
- **0.1-AC-03** — Cover install, setup, cron, doctor, status, manual and
  scheduled run, issue-to-PR, restart, merge/reject, update, migrate, recover,
  remove, and uninstall in the operator journey matrix.
- **0.1-AC-04** — Cover Hermes tool errors and malformed/timeout/side-effect
  outcomes, configuration, GitHub, Git, worker, SQLite, OCI, gateway, and
  permissions in the fault-injection matrix.
- **0.1-AC-05** — Map every contract requirement to its exact acceptance and
  current evidence status.
- **0.1-AC-06** — Give every gap exactly one existing task and acceptance owner,
  or one explicit approved deferral; reject unowned or contradicted promises.
- **0.1-AC-07** — Cross-link all six attachments and provide independently
  reproducible commands for every factual classification.
- **0.1-AC-08** — Keep every phase issue-ready and every manifest requirement,
  progress entry, link, dependency, and acceptance ID synchronized.
- **0.1-AC-09** — Inventory every shipped plugin, hook, script, asset, skill,
  generated text, comment, docstring, error, manifest field, and command path;
  flag stubs, pending-work notes, fake paths, and development-only behavior.

## Handoff expectations

Write `handoffs/0.1.md` and the six `handoffs/readiness/` attachments. Map every
acceptance ID to a reproducible procedure, result, and attachment. Record all
source links, exact commands, gap owners or approved deferrals, contradictions,
and limitations without presenting planned behavior as shipped.
