## Public contracts used

- `CONTRACTS.md` RUN-005 — `caduceus status` returns 0 for valid rendered state,
  2 for missing state, 1 for corrupt state or queue data.
- `CONTRACTS.md` DEBT-STATUS — status exit-code defect owned by Task 2.7.

## State/schema effects

None. The `StatusReport` struct, `StatusDiagnostic` enum, and `STATUS_SCHEMA_VERSION`
are unchanged. The `report()` return type changed from `CaduceusResult<String>` to
`CaduceusResult<(String, Option<StatusDiagnostic>)>`, but the test helpers
(`build_report`, `render_human`, `render_json`, `build_report_from_state`) are
unaffected — only the top-level `report()` entry point changed.

## Tests added or changed

- 4 new shell-level tests in `tests/status_test.rs`:
  - `status_exit_0_for_valid_state`
  - `status_exit_2_for_missing_state`
  - `status_exit_1_for_corrupt_state`
  - `status_exit_1_for_corrupt_queue`
- 1 existing test (`idle_output_matches_readme_fixture`) updated to destructure
  the new `(String, Option<StatusDiagnostic>)` return type.

## Forbidden side-effect checks

- No state files are written or modified by the status command.
- No daemon lock is acquired by the status command (read-only).
- The corrupt file is preserved on disk (existing test `corrupt_state_diagnostic_preserves_file`
  and `corrupt_queue_diagnostic_preserves_file` assert this).

## Residual risks

- The `std::process::exit()` calls inside the CLI handler bypass normal Rust
  cleanup. This is acceptable because the CLI is the top-level process and the
  exit codes are part of the contract (RUN-005). No resources need cleanup
  after rendering the status report.
