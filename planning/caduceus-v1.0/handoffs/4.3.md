## Implementation deviations from issue body

The GitHub issue body drafted for #21 named `src/runtime/finalize.rs`, `src/runtime/needs_attention.rs`, and `src/runtime/queue.rs` as new modules. The implementation placed the lifecycle glue in `src/daemon/tick.rs`, the queue state methods on `StateStore` in `src/state/queue.rs`, and the polling function inline in `src/daemon/tick.rs` — all into the existing v0.1 daemon code paths rather than a new `src/runtime/` namespace. The v1.0 plan (`planning/caduceus-v1.0/tasks/4.3-add-human-review-lifecycle.md`) does NOT specify file names — only the owned surface ("AwaitingReview, Done, and NeedsAttention") and acceptance criteria. The implementation satisfies every packet requirement and every AC. The file-layout choice is consistent with the v0.1 daemon's existing structure.

## Residual risks

- The polling function `poll_awaiting_review_entries` runs on every tick. If a single PR is consistently polling as `StillOpen` across many ticks, the log volume from the `info!` call could be noisy. Mitigation: the function only logs on state changes (Merged, ClosedWithoutMerge) and on errors, not on the `StillOpen` / `NotFound` no-op branches.
- The `NeedsAttention` queue state is new. Any external monitoring or operator UI enumerating queue states will need updating to render it. Out of scope for 4.3.
- Local pytest from the repo root fails on a pre-existing `tests.fake_ctx` import path issue; CI pytest passes in 25s. Tracked separately.
- The single `unlink_claim_best_effort` call from `ClaimGuard::finish_awaiting_review` is best-effort (`let _ =`). If the unlink fails, the reaper handles the orphan on a later tick.
