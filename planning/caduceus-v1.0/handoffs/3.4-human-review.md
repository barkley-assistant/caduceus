### AC-01 — Operate under the daemon lock ✅

The `hold_daemon_lock` parameter controls lock acquisition. When `true`, the
function acquires `DaemonLock::try_acquire` before any mutation. The Rust
RAII guard ensures the lock is released when the function returns. Tests
pass `false` to avoid needing a lock file.

### AC-02 — Archive corruption with a useful manifest ✅

The corrupt database is renamed to `<state_dir>/state.db.corrupt-<unix-timestamp>`
before the fresh database is created. The archive preserves the original
corrupt file for forensic analysis. The `RecoveryReport` surfaces the archive
path to the caller.

### AC-03 — Install only validated repaired data atomically ✅

When a backup is provided, it is first validated with `store::open(bk)` which
opens the SQLite connection and checks the schema version. Only after
validation succeeds is the backup installed via `install::atomic_write` (the
write-temp → sync → rename pattern from Task 3.1). If `store::open` fails,
the function returns an error without modifying the active database.

### AC-04 — Preserve valid state after failure ✅

The corrupt database is renamed to the archive BEFORE the fresh database is
created. If the fresh database creation fails, the archive is already on disk
(not lost). The operator can restore from the archive manually. The backup
is also preserved on disk — recovery never deletes the source backup.
