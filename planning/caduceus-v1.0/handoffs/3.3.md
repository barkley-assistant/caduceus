## Public contracts used

- `CONTRACTS.md` STATE-002 — `caduceus migrate-state --to sqlite` acquires lock,
  validates, imports in one tx, activates SQLite, leaves JSON backup.
- `CONTRACTS.md` STATE-003 — recovery commands install atomically.

## State/schema effects

The `state.db` SQLite file is created and populated with queue entries.
The JSON `state.json` is left untouched as a validated backup.

## Forbidden side-effect checks

- No existing JSON files are modified.
- No daemon lock acquired when `LockPolicy::Skip` (test-only).
- No GitHub API calls.
- No git mutations.

## Residual risks

- The migration reads the entire JSON state into memory before writing
  to SQLite. This is acceptable for the expected scale (< 10K entries).
- `INSERT OR REPLACE` is used so re-running migration is idempotent.
- The CLI flag is `--to-sqlite` (hyphenated) matching clap's convention,
  not `--to sqlite` (space-separated) as the v0.1 docs describe. The
  actual command is `caduceus migrate-state --to-sqlite`.
