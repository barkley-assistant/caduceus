## Public contracts used

- `CONTRACTS.md` STATE-003 — recovery commands operate under daemon lock,
  archive corrupt input, validate repaired data, install atomically.

## State/schema effects

The `state.db.corrupt-<ts>` archive file may be created in the state
directory during recovery. The original corrupt database is always
preserved as an archive before a fresh database is created.

## Forbidden side-effect checks

- No existing state files are modified (the corrupt file is renamed to archive).
- No GitHub API calls.
- No git mutations.

## Residual risks

- `PRAGMA integrity_check` is used to verify database health. This is a
  read-only operation that checks internal consistency. It does NOT detect
  all forms of corruption (e.g., a valid database with wrong data).
- The backup is validated with `store::open()` before installation. If the
  backup itself is corrupt, the recovery fails with a clear error.
- When no backup is provided, a fresh empty database is created. This loses
  all queue entries and metadata — the operator is responsible for providing
  a backup.
