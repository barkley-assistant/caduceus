- Residual risks:
  - setup_config uses shellexpand to expand ~/.config/caduceus/config.yaml;
    shellexpand reads $HOME which must be set in the test environment. The
    tests use load_with_context (no env access) for deterministic paths and
    never depend on shellexpand for directory creation.
  - The readiness diagnostics check bridge file existence at
    `$HERMES_HOME/caduceus/worker-bridge.py`. In v0, harness === bridge.
    Task 6.x separates them; the readiness field schema (7.4.0) is forward-
    compatible because consumers that don't know about the new field ignore it.
  - Cross-filesystem atomicity: setup_config writes the temp file to the same
    directory as the target, so rename(2) is always on the same filesystem.
    The TmpGuard Drop impl ensures cleanup on panic/error.
  - The default_worker_command hermes_home fallback resolves to
    `$HERMES_HOME/caduceus/worker-bridge.py`. The user-owned bridge is seeded
    by `hermes caduceus setup` (Python adapter). If that step was skipped,
    the bridge file won't exist and the daemon will surface a file-not-found
    error at tick time.
  - The CLI `caduceus setup` variant reads $HERMES_HOME from the process
    environment directly rather than through the config chain, because
    setup is used to create the config that the config chain loads.
    This is intentional — setup and load use the same $HERMES_HOME source.
