# Caduceus v0.1 MSRV — Historical Deviation (DEBT-MSRV)

> Sole owner: Task 0.3 (`0.3-AC-01`, `0.3-AC-02`, `0.3-AC-03`).
> Closes `DEBT-MSRV` from `CONTRACTS.md` "V0.1 carryover debt".
> The v0.1 planning tree is byte-for-byte unchanged.

## What the v0.1 packet says (the historical surface)

- `planning/caduceus-v0.1/tasks/9.2-execute-the-release-gate-and-cutover-checklist.md`
  line 19 names "Rust 1.75" as the toolchain for the v0.1 release
  gate and cutover checklist.
- `planning/caduceus-v0.1/archive/full-reviewed-plan.md` (the
  pre-`CR-002` reviewed plan) calls out "Rust 2021, MSRV 1.75" in
  the Toolchain section, and the same archive's `0.1-create-the-rust-crate-and-module-graph.md`
  packet names "Rust 2021 edition (≥ 1.75)" as the locked toolchain.

The release-cutover acceptance line is the load-bearing one: the
v0.1 release captain's checklist assumes the suite can be run on
Rust 1.75, which was the toolchain pinned at v0.1 plan creation
(2026-05). The v0.1 archive's `archive/full-reviewed-plan.md` and
the in-tree `CONTRACT_REVISIONS.md` entry `CR-002` are the
authoritative record of the deviation.

## What the current toolchain says (the live surface)

- `Cargo.toml` declares `rust-version = "1.97"` under `[package]`.
- `planning/caduceus-v1.0/CONTRACTS.md` `CONFIG-002`:
  "Rust 2021 and MSRV 1.97 remain required. `Cargo.lock` is
  committed and CI uses `--locked`."
- `planning/caduceus-v0.1/CONTRACT_REVISIONS.md` `CR-002`
  (2026-07-13): "Rust 1.97 is the current stable toolchain as
  of 2026-07-13. The previously pinned MSRV 1.75 is two years
  old and predates several language and standard-library
  improvements used in modern async, error-handling, and
  serialization code."

The bumped toolchain is the live one; the v0.1 release captain
shipped v0.1 against it, with the deviation explained in `CR-002`
rather than in a v0.1 archive edit.

## Why this is a Phase 00 task, not a silent fix

The v0.1 archive is sealed by `v01_tree_sha256` in
`task-manifest.json`. `validate_plan.py` hashes sorted relative
paths and file bytes on every run and refuses to proceed on
drift. The seal cannot be relaxed to repair a "stale text"
discrepancy, so the only honest fixes are:

1. **Add this handoff** in the v1.0 planning tree (here). The
   v1.0 tree is the planning authority for v1.0+; the deviation
   is recorded where future readers will look for it.
2. **Cite the deviation in the v0.1 archive's existing entry.**
   The v0.1 `CONTRACT_REVISIONS.md` already names the bump; this
   handoff points future readers at the v0.1 entry that already
   explains the rationale and the affected task packet. The
   packet text itself is not edited (the archive is sealed).
3. **Document the routing in `CONTRACTS.md`.** `CONFIG-002` is
   the v1.0 contract surface that pins 1.97; `V0.1 carryover
   debt` lists `DEBT-MSRV` with Task 0.3 as the sole owner. No
   v0.1 archive edit is required because the v0.1 archive
   already explains the deviation in its own revision log.

The deviation is closed for v1.0 because the v1.0 contract pins
1.97 and the v1.0 plan tree records the history here. The v0.1
archive remains a historical artifact whose deviation is
explained by its own revision log, not by an edit to its
frozen plan packets.

## Scope of the deviation

The deviation is **MSRV only**, not behavior. The v0.1 contract
text the packet referenced ("pass the release suite on Rust
1.75") was *replaced* by `CR-002` with "pass the release suite
on Rust 1.97" in the same sentence. The other CR-002 changes
(dependency MSRV disclosure, locked-toolchain discipline) are
unchanged in the live daemon.

There is no other v0.1 deviation routed through Task 0.3:

- The `--from <legacy.json>` vs `--to sqlite` migration-flag
  split is a separate historical concern owned by Task 3.3; it
  is closed in `CONTRACT_REVISIONS.md` §"v0.1 ↔ v1.0 migration-
  command split" and recorded in this handoff's sister file
  `planning/caduceus-v1.0/handoffs/readiness/06-gap-register.md`
  row G-11.
- The `Config::load` stub is a Phase 02 implementation gap
  owned by Task 2.1, not a historical deviation.
- The status exit-code defect is a Phase 02 correction owned
  by Task 2.7 (`DEBT-STATUS`), not a historical deviation.
