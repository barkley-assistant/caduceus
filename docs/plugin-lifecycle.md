# Plugin Lifecycle (Hermes v0.18.2)

This doc is the lifecycle contract between Caduceus and
Hermes Agent v0.18.2. If you are a Hermes user, the
README gives you the install commands; this doc
explains what each one does and why. If you are writing
a Hermes plugin of your own, this doc is a worked
example of how to play nicely with the v0.18.2
directory-plugin loader.

## The Canonical Layout

Hermes v0.18.2 expects a "directory plugin" — a
directory with a `plugin.yaml` manifest and an
`__init__.py` adapter. The directory root is the
installable plugin root; `hermes plugins install
<owner>/<repo>` clones the entire repository root into
`~/.hermes/plugins/<name>/`. **You cannot install only
a subdirectory**; Hermes would move the subdirectory
without the rest of the workspace the plugin needs.

The canonical Caduceus layout:

```
plugin.yaml
__init__.py
Cargo.toml
Cargo.lock
src/
plugin-assets/worker-bridge.py
plugin-assets/caduceus-pulse.sh
skills/caduceus/SKILL.md
bin/                         # generated by setup; ignored by git
```

`bin/` is generated by `hermes caduceus setup` and
git-ignored. The daemon binary is built and installed
there by setup; subsequent `hermes plugins update` runs
do not rebuild it (you must rerun setup).

## What `plugin.yaml` Contains

```yaml
manifest_version: 1
name: caduceus
version: 0.1.0
description: GitHub issue orchestrator with worker supervision
author: barkley-assistant
kind: standalone
requires_env: []
provides_tools: []
provides_hooks: []
```

That's it. Caduceus does **not** put any of the
following in the manifest, even though some pre-0.18
plugins did:

- defaults
- file lists
- binary declarations
- lifecycle shell commands (install/upgrade/uninstall
  hooks)
- profiles
- cron declarations

Hermes v0.18.2 silently ignores unknown manifest
fields, so a plugin could ship a manifest with
`commands/*.md` references and Hermes would appear to
install fine — but the references would be dead
weight. Caduceus's contract test
(`tests/docs_contract_test.rs`) explicitly rejects a
hypothetical manifest with these legacy fields so we
don't drift back into the pre-0.18 shape.

## What `__init__.py` Does

`__init__.py` is a small, standard-library-only Hermes
adapter. Its `register(ctx)` explicitly registers:

- `ctx.register_skill("caduceus",
  <root>/skills/caduceus/SKILL.md, ...)`, resolvable
  as `caduceus:caduceus`. **Plugin skills are opt-in
  and are not automatic trigger rules.** This means
  a user has to explicitly invoke `caduceus:caduceus`
  (or `apply_skill caduceus:caduceus`) — the skill
  does not fire on conversational triggers.
- `/caduceus-status` through `ctx.register_command`.
  This invokes `<root>/bin/caduceus status --json`
  with an argument array, a short timeout, and
  bounded output. When setup has not built the
  binary, the command returns a chat-safe diagnostic
  instead of crashing.
- `hermes caduceus ...` through
  `ctx.register_cli_command`, with `setup`, `doctor`,
  `status`, `cron-install`, and `cron-remove`
  subcommands.

**Plugin import/registration never compiles code,
mutates config, creates cron jobs, or performs
network access.** The registration is metadata-only;
the side effects happen when the operator runs the
corresponding commands.

## The Install Path

The Hermes lifecycle for Caduceus is:

```text
hermes plugins install barkley-assistant/caduceus --enable
        ↓
hermes caduceus setup
        ↓
hermes caduceus cron-install
        ↓
(cron fires every 2 minutes)
        ↓
hermes caduceus status     # at any time
```

`setup` is the explicit build/install step. Hermes does
not run manifest build/hook steps; Caduceus ships an
explicit idempotent `setup` that:

1. Verifies Rust/Cargo/Git/Python prerequisites.
2. Runs `cargo build --release --locked
   --manifest-path <root>/Cargo.toml`.
3. Atomically installs the resulting binary as
   `<root>/bin/caduceus`.
4. Creates the configured state directories with
   secure modes (0700 for directories, 0600 for
   files).
5. Seeds the user-owned bridge at
   `$HERMES_HOME/caduceus/worker-bridge.py` (default
   `~/.hermes/caduceus/worker-bridge.py`) — only when
   absent.

If the shipped bridge template has changed since you
installed, setup writes a sibling `.new` candidate and
prints a notice; it never overwrites your edits.

## The Cron Path

Hermes cron does not import `cron/*.yaml`. Instead,
`hermes caduceus cron-install` atomically writes a
generated Bash wrapper at
`$HERMES_HOME/scripts/caduceus-pulse.sh`; the wrapper
contains the absolute installed binary path and uses
`exec <binary> run`.

It then creates or reconciles exactly one named
`caduceus` no-agent job equivalent to:

```text
hermes cron create "every 2m" --name caduceus \
    --script caduceus-pulse.sh --no-agent
```

Reconciliation uses Hermes's registered `cronjob`
tool through `ctx.dispatch_tool`, not direct edits
to `cron/jobs.json`. Zero matches creates, one match
updates/reuses, multiple matches fail with their IDs.
`cron-remove` removes the recorded/matching job and
wrapper idempotently.

**The gateway (or a configured managed cron provider)
must be running for scheduled jobs to fire.** This is
a Hermes contract; Caduceus does not override it.

## The Update Path

```text
hermes plugins update caduceus    # refresh source only
hermes caduceus setup             # rebuild + reinstall binary
hermes caduceus status            # verify
```

`hermes plugins update` updates the plugin source. It
does **not** rebuild the binary. The operator runs
`hermes caduceus setup` to rebuild. This is intentional:
Hermes never runs manifest build steps, and the rebuild
is operator-controlled because the rebuild can take a
minute on a cold cache.

## The Removal Path

```text
hermes caduceus cron-remove       # removes cron job + wrapper
hermes plugins remove caduceus    # tears down the plugin
```

Hermes has no plugin uninstall hook. Removal preserves:

- `$HERMES_HOME/caduceus/` (the user-owned bridge).
- The daemon's state directory
  (`$HERMES_HOME/caduceus-state/` by default).
- Your `~/.hermes/config.yaml` caduceus section.
- The watched repositories on disk.

The watched repositories are unrelated to Caduceus;
remove them with their own tooling if you want them
gone.

## What Caduceus Does Not Put in the Manifest

Caduceus explicitly omits:

- **`commands/*.md` references** — Hermes v0.18.2 does
  not load them. Including them would be misleading.
- **`bin/caduceus`** — the binary is generated by
  `setup`. Listing it in the manifest would mean
  Hermes has stale references to a file that does not
  exist yet.
- **`lifecycle:` hooks** — Hermes does not invoke
  them. Caduceus's lifecycle is the explicit
  `setup` / `cron-install` / `cron-remove`
  subcommands.
- **defaults, profiles, env vars other than
  `requires_env`** — `requires_env` is the
  v0.18.2-supported way to declare an env-var
  requirement. Caduceus does not need any.

The contract test enforces this. If a future change
adds a deprecated manifest field, the test fails and
you notice before the change ships.
